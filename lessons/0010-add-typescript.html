Lesson 0010 · Node.js 后端之路

给项目加上 TypeScript（TS 正式上场）

为什么学这个？ 到目前为止 LinkPulse 全是 .js：你写过 req.body.url、写过 controller / service 分层、用 zod 校验过输入——但编辑器一直「半瞎」，拼错字段名、把字符串当数字用，都要等运行起来报错才发现（还记得 0009 那个忘记声明就用的变量 bug 吗？）。本章我们让 TypeScript（TS，给 JS 加上「类型」的超集语言）正式上场：装好工具链、写好 tsconfig.json、把现有文件从 .js 改成 .ts，并给 Express 的 req/res 标注类型。从这一章起，LinkPulse 的所有代码都用 TypeScript 写——这正是现代 Node 后端的主流做法，也是你未来面试和上线收费 SaaS 时的标配。

1TypeScript 是什么？为什么后端值得用

定义。 TypeScript 是微软开发的、JavaScript 的一个超集（superset）：任何合法的 JS 都是合法的 TS，但 TS 额外允许你给变量、参数、返回值标注类型（type annotation）。浏览器和 Node 都不直接认识 TS——它必须先被「去掉类型」变回普通 JS 才能运行。所以 TS 本质是一个开发期的工具：在你写代码时帮你查错，在交付前消失。1

类比。 把 JS 想成「口头约定」：你说「这个函数收一个 URL」，但没人拦着别人传个数字进去。TS 是「写进合同的字段」：参数写明 url: string，谁要传数字，编辑器当场红线警告，编译时直接拒绝。合同不改变程序运行时的行为，但能在签字（部署）之前把扯皮挡掉。

作为一个前端工程师，你大概率已经在 React 项目里见过 TS。这里只点明它在后端语境下凭什么值得你认真投入：

  • 类型安全（type safety）。 后端处理的是外部不可信数据：HTTP 请求体、数据库行、第三方 API 返回。一个字段名写错（req.body.ur）、一个该是数字的字段当字符串拼接，在 JS 里要等到线上用户触发那条路径才崩。TS 在你敲代码的瞬间就告诉你。
  • 编辑器自动补全（autocomplete）。 当 req 有了类型，敲 req. 编辑器会列出 body / params / query / headers……你不用再翻文档猜字段名。这对后端尤其值钱——Express、数据库客户端的 API 面非常大。
  • 自文档（self-documenting）。 一个写着 function createLink(input: CreateLinkInput): Promise<Link> 的函数，光看签名就知道吃什么、吐什么。半年后的你（和接手的同事）不必读实现就能用对。
  • 重构友好（refactor-safe）。 SaaS 是要长期维护的。当你把 Link 接口里的 shortCode 改名为 slug，TS 会把项目里每一处用到旧名字的地方标红——漏改一个都过不了编译。在纯 JS 里，这种改名只能靠全局搜索 + 祈祷。
🌍 实战：现代 Node 后端的事实标准就是 TypeScript。NestJS（企业级 Node 框架）默认 TS；Prisma、Drizzle 等主流 ORM 把「类型安全」当成头号卖点；几乎所有正经招聘的「Node 后端」岗位都默认你写 TS。你做 LinkPulse 这种要上线收费的项目，用 TS 不是「加分项」，而是基本盘——它直接决定了你迭代订阅、计费这些核心逻辑时，敢不敢大胆改代码。

2安装工具链：typescript、tsx、@types/*

在 study-node 根目录（就是有 package.json 那一层）执行：

npm i -D typescript tsx @types/node @types/express

-D 是 --save-dev 的简写，把它们装进 devDependencies（开发期依赖）。这点你在前端已熟：这些工具只在写代码 / 构建时需要，部署运行后的纯 JS 不依赖它们。逐个说清每个包是干嘛的：

包作用
typescriptTS 编译器本体，提供 tsc 命令——把 .ts 翻译成 .js，并做类型检查。
tsx开发期的「直接运行器」：让 Node 不用先编译就能跑 .ts 文件（下一节细讲）。2
@types/nodeNode.js 内置 API（process、fs、Buffer……）的类型声明。Node 本身是用 JS 写的，没带类型，靠这个包补上。
@types/expressExpress 的类型声明，让 req/res 这些有类型、能自动补全。

新概念：@types/* 是什么？ 很多 npm 库（包括 Express）本身是用纯 JS 写的，不自带类型信息。社区维护了一个叫 DefinitelyTyped 的巨型仓库，专门为这些 JS 库补写「类型声明文件」（.d.ts，只描述「长什么样」、不含实现），发布成 @types/库名 这种包。装上它，TS 就「认识」这个 JS 库了。3

怎么判断要不要装 @types？ 先直接 import 用。如果库自带类型（像 zod、Prisma，包里就有 .d.ts），就什么都不用装；如果 TS 报 Could not find a declaration file for module 'xxx'，再去装对应的 @types/xxx。Express 属于后者，所以这里要装 @types/express。
🌍 实战：@types/* 生态是 TS 能落地的关键。 正因为 DefinitelyTyped 几乎覆盖了所有老牌 JS 库，你才能在一个 TS 项目里自由混用「祖传 JS 库」而不失去类型。看一个库 TS 支持好不好，老手会先看：① 是否自带类型（最佳）；② 没有的话 @types/ 有没有、新不新。这是选型时的真实考量。

3tsconfig.json：告诉编译器怎么干活

新概念。 tsconfig.json 是 TS 项目的「配置中枢」——只要某个目录里有它，TS 就把这个目录认作一个项目，并按里面的规则来检查和编译。1 你可以手敲，也可以用 npx tsc --init 生成一份带超多注释的默认版。这里我们手写一份贴合 LinkPulse 的精简配置。在根目录新建 tsconfig.json：

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "sourceMap": true
  },
  "include": ["src/**/*.ts"]
}

逐项用大白话解释最关键的几个（其余几个看下方提示）：

选项通俗解释
target编译产物用哪个版本的 JS 语法。我们写 ES2022：现代 Node（18+）原生支持，能直接用 async/await、可选链 ?. 等新语法，不必降级成一堆老旧代码。4
module编译产物用哪种模块系统。你的项目 package.json 里写了 "type":"module"（ESM），所以选 NodeNext——它让 TS 完全按 Node 的现代模块规则来处理 import/export。5
moduleResolutionTS 用什么策略去「找到」你 import 的文件/包。配 module:NodeNext 时就该配 NodeNext，两者成套出现。5
strict本章重点。 一个总开关，打开一整套最严格的类型检查。强烈建议 true（下一节专门讲它帮你抓的 bug）。6
rootDir源码的根目录。我们规定所有 .ts 放在 src/ 里。它决定编译时「从哪个前缀开始算相对路径」。7
outDir编译出来的 .js 放到哪。我们放 dist/。配合 rootDir：src/app.ts → dist/app.js，目录结构原样镜像。7
另外三个一句话过： esModuleInterop: true 让 import express from 'express' 这种「默认导入」写法对 CommonJS 老库也好用（不写它有时得用别扭的 import * as）；skipLibCheck: true 跳过对第三方 .d.ts 的类型检查，显著加快编译、避开别人库里的类型噪声；sourceMap: true 生成「源码地图」，让运行编译后的 JS 报错时，行号能映射回你的 .ts 源码，方便调试。

include 在 compilerOptions 之外，告诉 TS把哪些文件纳入这个项目——这里是 src/ 下所有 .ts。（注意：rootDir 不负责挑文件，它只管编译时的路径前缀；挑文件是 include 的活儿。7）

⌨️ 顺手做：建好目录骨架

  1. 在根目录新建文件夹 src/，等下我们把代码搬进去。
  2. 把上面的 tsconfig.json 写进根目录。
  3. 在 .gitignore 里加一行 dist（编译产物不进 git，和 node_modules 同理）。

4两种「运行 TS」的方式：开发用 tsx，生产用 tsc

Node 不直接认识 .ts（虽然新版 Node 有实验性的「类型剥离」，但默认仍不开启、且不做类型检查2）。所以我们要在「写代码时怎么跑」和「上线时怎么跑」之间做选择。先把两种心智模型摆清楚：

开发期 · tsx（直接跑）

# 跑一次
npx tsx src/app.ts

# 改文件自动重启
npx tsx watch src/app.ts

tsx 底层用 esbuild（一个用 Go 写的超快转译器）把 .ts 即时转成 JS 再交给 Node 跑，几乎无感、无需先编译。但它为了快，不做类型检查——类型错了照样跑。2

生产期 · tsc（先编译）

# 把 src/ 编译到 dist/
npx tsc

# 用纯 Node 跑产物
node dist/app.js

tsc 会完整做类型检查，类型不过就拒绝出产物。上线跑的是 dist/ 里的纯 JS，运行环境里压根不需要 TS。这就是「TS 是开发期工具」的含义。

为什么要分两套？ 一句话：开发图快、上线图稳。开发时你每改一行都想立刻看到效果，tsx watch 用 esbuild 做到「保存即重启」，体验和你前端的热更新很像；上线时你要的是一份不依赖任何 TS 工具、且经过类型检查保证过的 JS。

关键提醒：tsx 不查类型 ≠ 你可以不查类型。 Node 官方文档明确建议：用 tsx 这类运行器开发时，另外用 tsc 做一次类型检查再交付。2 实操上常加一个 "typecheck": "tsc --noEmit" 脚本（--noEmit = 只检查、不生成文件），CI/提交前跑一下。编辑器（VS Code）里那些红线，本质也是在帮你实时做这件事。

现在把这些固化进 package.json 的 scripts。打开根目录 package.json，把 scripts 改成（注意路径已指向 src/，入口文件名按你的实际情况，下一节我们就建 src/app.ts）：

"scripts": {
  "dev": "tsx watch src/app.ts",
  "build": "tsc",
  "start": "node dist/app.js",
  "typecheck": "tsc --noEmit"
}

四个脚本的分工：npm run dev 开发（自动重启）；npm run typecheck 查类型；npm run build 编译出 dist/；npm start 跑生产产物。从今往后，开发 LinkPulse 你只需 npm run dev。

🌍 实战：tsx watch 是当下最舒服的 Node+TS 开发姿势。 它取代了老牌组合 ts-node + nodemon。区别要知道：ts-node 默认会做类型检查，所以启动慢、改一下要等；tsx 走 esbuild、不查类型，所以飞快，把「类型检查」这件事交给编辑器和 tsc 专门去做。2 职责分离，体验更顺——这也是新项目越来越多选 tsx 的原因。

5TS 基础回顾（只讲本章用得到的）

下面这几样是给 Express 和 LinkPulse 数据结构标类型的最小够用集合。循序渐进，每个都给定义 + 例子。

① 类型注解（type annotation）

在变量/参数/返回值后面加 : 类型。基础类型有 string / number / boolean / string[]（字符串数组）等。注解写在被注解物的后面（和你熟的 let x: number 一致，不是 C 那种 int x）。8

const shortCode: string = 'a1B2c'
const clicks: number = 0
const tags: string[] = ['marketing', 'launch']

// 函数：参数和返回值都标注
function buildShortUrl(code: string): string {
  return `https://lnk.pulse/${code}`
}

很多时候 TS 能自动推断（infer）类型——const clicks = 0 它就知道是 number，不必手写。原则：能推断就别画蛇添足，函数参数和公共 API 的返回值则建议显式写清楚（当文档用）。8

② interface vs type：给「对象的形状」起名

这俩都能描述「一个对象长什么样」。这正是我们要给 LinkPulse 数据建模时最常用的工具。

interface

interface Link {
  id: string
  url: string
  shortCode: string
  clicks: number
}

type（类型别名）

type Link = {
  id: string
  url: string
  shortCode: string
  clicks: number
}

怎么选？ 描述对象形状，两者几乎等价。官方 Handbook 的建议是：默认用 interface，当你需要表达「联合类型」「给原始类型起别名」等 interface 做不到的事时，才用 type。8 一个实用差异：interface 可被「再次打开」追加字段（多次声明同名 interface 会合并），type 一旦定义不可更改——给 Express 扩展类型时会用到前者（后续章节遇到再说）。

③ 可选属性（optional property）

在字段名后加 ?，表示「这个字段可以没有」。LinkPulse 里很多字段就是可选的——比如短链可以不设过期时间：

interface CreateLinkInput {
  url: string          // 必填
  customCode?: string  // 可选：用户想自定义短码就传，不传我们自动生成
  expiresAt?: string   // 可选：不传 = 永不过期
}

关键：可选字段的真实类型是「你写的类型 或 undefined」。所以用之前必须先判断有没有，否则在 strict 模式下 TS 会拦你：8

function resolveCode(input: CreateLinkInput): string {
  // input.customCode 的类型是 string | undefined
  if (input.customCode !== undefined) {
    return input.customCode      // 这里 TS 确信它是 string
  }
  return generateRandomCode()
}

④ 联合类型（union type）

用 | 表示「是这几种之一」。对 SaaS 太常用了——比如订阅套餐只能是固定几个值：

type Plan = 'free' | 'pro' | 'business'   // 这叫「字面量联合类型」，比 string 更精确

interface User {
  id: string
  plan: Plan        // 只能是这三个字符串之一，写错 'pr' 会报错
}

用联合类型的值时，TS 只允许你做「所有成员都支持的操作」；要区别对待就得收窄（narrowing）——用 typeof、=== 比较等把范围缩小，TS 才放行对应分支的专属操作。8 上面 ③ 里的 if (input.customCode !== undefined) 就是一次收窄。

⑤ 泛型初步（generics）

类比。 泛型就是「类型的参数」——像函数能收参数一样，类型也能收一个「类型占位符」（习惯用 T），用的时候再填进去。你其实已经见过它：string[] 也可以写成 Array<string>，这里 Array 就是个泛型，string 是填进去的类型。

本章只需认得一种形态：Express 的 Response、还有异步函数的返回值 Promise<T>。比如「一个最终给出 Link 的异步操作」写作 Promise<Link>：

// service 层：返回「一个最终是 Link 的 Promise」
async function createLink(input: CreateLinkInput): Promise<Link> {
  // ...存库、生成短码...
  return { id, url: input.url, shortCode, clicks: 0 }
}

现在记住「Promise<Link> = 异步拿到的 Link」就够了，深入用法后面接触数据库时再展开。

⑥ any vs unknown：处理「不知道是什么」的数据

后端天天要面对「类型未知的外部数据」（HTTP body、第三方返回）。TS 给了两个工具，差别巨大：

anyunknown
含义「别管它，关掉类型检查」「确实未知，但用前必须先检查」
能否直接用能，TS 完全不拦——等于退回纯 JS不能，必须先收窄成具体类型才能操作
安全性低：错误会漏到运行时高：逼你先验证
function handle(raw: any) {
  raw.toUpperCase()   // TS 不报错，但 raw 万一是数字 → 运行时崩
}

function handleSafe(raw: unknown) {
  raw.toUpperCase()   // ❌ 编译报错：raw 类型未知，禁止直接调方法
  if (typeof raw === 'string') {
    raw.toUpperCase() // ✅ 收窄成 string 后才允许
  }
}

结论：能不用 any 就别用——它等于亲手关掉 TS 的全部保护。8 处理不可信输入时优先 unknown，再用 zod 之类的工具校验成可信类型。

🌍 实战：any 是「类型逃生舱」，但漏的水会淹到生产环境。 团队里常约定开启 noImplicitAny（strict 已含），禁止「悄悄变成 any」；代码评审看到显式 any 一般要求写注释说明理由。还记得 0009 里 zod 干的活吗——它正是把 req.body（本质是 unknown 级别的不可信数据）校验并收窄成一个有类型的安全对象。TS 的 unknown 和 zod 是天生一对：unknown 负责「逼你检查」，zod 负责「替你检查」。

6动手迁移：把 LinkPulse 改成 TypeScript

理论齐了，开始把现有项目搬到 TS。我们的目标结构（代码全进 src/）：

study-node/ ├─ tsconfig.json ← 新增 ├─ package.json ← scripts 已改 ├─ dist/ ← tsc 编译产物（git 忽略） └─ src/ ├─ app.ts ← 原 express-server.js ├─ config.ts ← 原 config.js（0009） ├─ types.ts ← 新增：LinkPulse 的数据类型 ├─ schemas/ │ └─ link.schema.ts ← 原 zod schema（0009） ├─ routes/ │ └─ link.routes.ts ← 原 routes（0008） ├─ controllers/ │ └─ link.controller.ts └─ services/ └─ link.service.ts

第 1 步：定义 LinkPulse 的核心类型。 新建 src/types.ts，把前面学的接口/可选/联合都用上：

// src/types.ts —— LinkPulse 的领域数据类型

export type Plan = 'free' | 'pro' | 'business'

// 一条短链在系统里的完整形状
export interface Link {
  id: string
  url: string
  shortCode: string
  clicks: number
  createdAt: string
  expiresAt?: string   // 可选：不传 = 永不过期
}

// 创建短链时，外部允许传入的字段（注意：没有 id/clicks，那是系统生成的）
export interface CreateLinkInput {
  url: string
  customCode?: string
}

第 2 步：给 Express 类型化。 把 express-server.js 搬到 src/app.ts，关键变化是从 express 同时导入 Request、Response 两个类型，用来标注 req/res。9（这正是 0006/0008 里那些路由处理函数现在该有的样子）：

// src/app.ts
import express, { Request, Response } from 'express'

const app = express()
app.use(express.json())

app.get('/', (req: Request, res: Response) => {
  res.send('LinkPulse API up')
})

// req/res 有了类型后，敲 req. 会自动补全 body/params/query/headers
app.get('/greet/:name', (req: Request, res: Response) => {
  res.json({ hello: req.params.name })   // params.name 自动补全且为 string
})

// 404 兜底中间件（和 0006 一样，只是现在有了类型）
app.use((req: Request, res: Response) => {
  res.status(404).send('Not Found')
})

app.listen(3000, () => {
  console.log('Server running at http://localhost:3000')
})
导入语法解读： import express, { Request, Response } from 'express' 一行做了两件事——express（默认导入，是那个能调用的函数）+ { Request, Response }（具名导入的两个类型）。这些类型来自你装的 @types/express。能这么干净地用「默认导入」，靠的就是 tsconfig.json 里那句 esModuleInterop: true。

第 3 步：给分层（controller/service）和 zod 标类型。 把 0008 的 controller、0009 的 service / zod schema 也搬进 src/ 并改成 .ts。这里给出 controller 的类型化版本，让你看到「类型如何贯穿一整条请求链路」：

// src/controllers/link.controller.ts
import { Request, Response } from 'express'
import { createLinkSchema } from '../schemas/link.schema.js'  // 见下方注意
import * as linkService from '../services/link.service.js'

export async function createLink(req: Request, res: Response) {
  // zod 把不可信的 req.body 校验 + 收窄成有类型的对象
  const parsed = createLinkSchema.safeParse(req.body)
  if (!parsed.success) {
    return res.status(400).json({ error: parsed.error.flatten() })
  }
  // parsed.data 现在的类型 ≈ CreateLinkInput，全程有补全
  const link = await linkService.createLink(parsed.data)
  res.status(201).json(link)   // link 是 Link 类型
}
⚠️ 一个会绊住所有人的坑：相对导入要写 .js 后缀。 因为我们用了 ESM + module: NodeNext，TS 要求你在相对导入里写出扩展名，而且写的是 .js 不是 .ts——比如导入 link.service.ts 要写 '../services/link.service.js'。这看着违反直觉，原因是：你 import 的是「编译之后那个 .js」，TS 只是按运行时的实际文件名来对齐。5 记住规则即可：相对路径一律写 .js。（导入 'express' 这种第三方包则不用加后缀。）

第 4 步：跑起来，并故意撞一次类型错误。 启动开发服务器：

npm run dev        # = tsx watch src/app.ts，改文件自动重启

现在体会 TS 的「当场抓 bug」。在 app.ts 里故意写错一个字段名：

app.get('/greet/:name', (req: Request, res: Response) => {
  res.json({ hello: req.params.nam })   // ❌ 把 name 拼成了 nam
})

VS Code 立刻在 nam 下画红线，提示类似：Property 'nam' does not exist on type ...。这就是 0009 那个「未声明就用变量」级别的 bug——只不过现在它在你按下保存键的瞬间就被逮住，而不是等用户访问 /greet/... 时才在日志里看到一个 undefined。 跑一次 npm run typecheck 也会报同样的错并拒绝通过。改回 name，红线消失。

请求链路上类型如何一路传递（这就是 TS 在后端的全部价值）：
HTTP 请求 → req: Request（@types/express 给的类型）
  → zod 校验 req.body → CreateLinkInput（不可信 → 可信，且有类型）
  → service.createLink(input) → Promise<Link>
  → res.json(link) → 出错的字段名/类型全程红线兜底
🌍 实战：开 strict 才是 TS 真正发力的地方。 很多人把 .js 改名成 .ts 就以为「上了 TS」，却把 strict 关着——那 TS 几乎等于没用，undefined 照样满天飞。strict 里最值钱的是 strictNullChecks：它强制你区分「有值」和「可能为 null/undefined」，逼你在用 input.customCode、数组 .find() 的结果之前先判空——这恰好封死了后端最高频的一类线上崩溃：Cannot read properties of undefined。6 LinkPulse 从第一天就开 strict，省下的是未来无数个深夜排查 undefined 的小时。
