Lesson 0020 · Node.js 后端之路

后台任务与队列：用 BullMQ 异步处理

为什么学这个？ 现在 LinkPulse 的跳转路由是这样的：用户点开短链 → 你同步写一条 clicks 记录、再把 click_count 加 1 → 然后才 302 跳转。问题是：用户在等你写完两次数据库才被放走。访问量一大，跳转就变慢，数据库还可能成为瓶颈。本章我们引入「任务队列」：把「记点击」这种用户不需要等结果的活儿，从请求主线里搬出去，丢进队列，让独立的 worker 慢慢消化。跳转因此能先放人走，再后台算账。这是所有高并发后端的核心套路，也是发邮件、生成报表、Webhook 重试的通用解法。

1为什么需要后台任务？把慢活儿赶出请求主线

先回到一个最朴素的观察：一个 HTTP 请求的响应时间 = 你在 handler 里做的所有事情的耗时之和。在前端你早就知道「主线程别做重活，否则卡 UI」；后端是同一个道理的镜像——请求处理函数别做重活，否则卡响应。

哪些活儿属于「不该让用户等」的？判断标准只有一句话：用户需要这个操作的结果才能继续吗？

  • 需要等：查询短链目标地址（不查到就没法跳转）、校验登录密码（不验完不能放行）。这些必须同步做。
  • 不需要等：记录这次点击的统计、给用户发欢迎邮件、生成月度报表、重试一个失败的 Webhook。用户根本不关心这些什么时候完成,只要最终会完成就行。
❌ 同步写 click（现状）
查目标 URL (2ms) + 写 clicks (8ms) + UPDATE click_count (6ms) ≈ 16ms 才 302
✅ 异步入队
查目标 URL (2ms) + 入队 (~1ms) ≈ 3ms 就 302，统计交给后台

对 LinkPulse 这种「核心价值就是快速跳转」的 SaaS，跳转延迟直接决定体验。把统计写入异步化，跳转能快 3～5 倍，而且就算统计系统暂时挂了，跳转照样工作——这叫关注点解耦（decoupling）:核心链路不被附属链路拖累。

🌍 实战：「先 302 跳转，点击事件异步入队，worker 再写 clicks + 累加 click_count」是几乎所有短链/统计产品的标准做法。Bitly、t.co 这类系统在跳转主线里绝不会同步写分析库——它们把原始点击事件丢进队列/日志流，下游慢慢聚合。你换来的是：跳转飞快、数据库写入可以批量/限流、统计管线挂了也不影响主业务。
什么不能异步？ 凡是「失败必须立刻告诉用户」「结果要返回给前端」的操作不能异步。比如「创建短链」要把生成的 short_code 返回给前端，这步必须同步。异步只适合「fire-and-forget（发了就不管）」的副作用。

2队列模型：生产者 → 队列 → 消费者

「队列（queue）」这个词你在前端见过——比如事件循环里的任务队列。后端的任务队列是同一种数据结构的应用：一个先进先出（FIFO）的待办列表，外加三个角色：

Producer 生产者把「要做的活儿」打包成一个 job 丢进队列。在 LinkPulse 里就是跳转路由。
→
Queue 队列job 排队待办的地方。存在 Redis 里（持久、可被多进程共享）。
→
Worker 消费者一个或多个进程不断从队列取 job 并执行。写 clicks 的逻辑在这里。

类比： 想象一家快餐店。点餐员（生产者）把订单（job）贴到取餐台的轨道（队列）上就转身招呼下一位客人——他不站着等汉堡做好。后厨（worker）按顺序取订单做汉堡。点餐和做汉堡解耦了:点餐很快，做汉堡慢也不堵着前台。高峰期还能多加几个后厨(多开 worker)提升吞吐。

这个模型带来几个关键好处，正好对应「为什么用队列而不是直接 setTimeout / 不阻塞地直接调用」：

  • 持久化：job 存在 Redis,进程崩了重启后 job 还在,不会丢。直接 fireAndForget() 的话进程一挂活儿就没了。
  • 削峰填谷：瞬时涌入 1 万次点击,队列先囤着,worker 按自己的速度（并发限制）慢慢消费,数据库不会被打爆。
  • 重试:job 执行失败可以自动重试(下一节),而裸调用失败就失败了。
  • 水平扩展:worker 可以是独立进程、独立机器,和 Web 进程分开部署、独立扩容。
为什么用 Redis 存队列？ Redis 在内存中、速度极快,且天生支持原子操作和发布订阅——非常适合做队列后端。你在 Lesson 0014 已经用 ioredis 做缓存了,同一套 Redis 就能直接拿来当队列后端,无需额外中间件。这也是我们选 BullMQ 的原因。

3BullMQ 入门：Queue、add、Worker

BullMQ 是 Node.js 生态最主流的队列库,基于 Redis 构建,原生 TypeScript 支持,提供了重试、并发、定时、优先级等生产级能力1。先安装:

npm i bullmq

① 共享一个专用的 Redis 连接

BullMQ 的 Queue 和 Worker 都接收一个 connection 选项,可以是连接配置,也可以是一个已存在的 ioredis 实例——正好复用我们现有的连接习惯2。但有一个硬性要求:传给 Worker 的连接必须设置 maxRetriesPerRequest: null,否则 BullMQ 会直接抛异常2。

为什么是 null? Worker 要用阻塞命令长时间「守着」队列等新 job。ioredis 默认会在请求重试几次后报错,这会打断 worker 的阻塞等待。设成 null 表示「无限重试」,让 worker 即使 Redis 短暂抖动也能扛过去继续工作2。注意:这条连接和你给跳转缓存用的那条连接要分开,不要把缓存连接的配置直接共享给 worker。
// src/queue/connection.ts
import IORedis from 'ioredis';

// 专门给 BullMQ 用的连接配置（注意 maxRetriesPerRequest: null）
export const bullConnection = new IORedis(process.env.REDIS_URL!, {
  maxRetriesPerRequest: null,
});

② 定义队列（生产者侧）

一个 Queue 实例代表一条命名队列。我们给「点击事件」建一条 clicks 队列。给 job 的数据定义一个 TS 类型,这样生产者和消费者两边类型对齐:

// src/queue/click.queue.ts
import { Queue } from 'bullmq';
import { bullConnection } from './connection.js';

// 一次点击事件携带的数据（即一个 job 的 payload）
export interface ClickJobData {
  linkId: number;
  ipAddress: string | null;
  userAgent: string | null;
  referrer: string | null;
  clickedAt: string; // ISO 字符串，入队时就定好时间
}

// 队列名 'clicks'，泛型 <数据类型, 返回值类型>
export const clickQueue = new Queue<ClickJobData>('clicks', {
  connection: bullConnection,
  defaultJobOptions: {
    attempts: 3,                              // 失败最多重试 3 次
    backoff: { type: 'exponential', delay: 1000 }, // 退避：1s、2s、4s
    removeOnComplete: 1000,  // 成功的 job 最多留 1000 条（防 Redis 撑爆）
    removeOnFail: 5000,      // 失败的多留一些方便排查
  },
});

③ 添加 job（add）

用 queue.add(name, data, opts) 把一个 job 推入队列。第一个参数是 job 的名字(一条队列里可以有多种 job 名),第二个是 payload1:

await clickQueue.add('record-click', {
  linkId: 42,
  ipAddress: '1.2.3.4',
  userAgent: 'Mozilla/5.0 ...',
  referrer: 'https://x.com',
  clickedAt: new Date().toISOString(),
});

④ 写 Worker（消费者侧）

一个 Worker 实例接收队列名、一个处理函数（processor）和选项。处理函数是 async (job) => {...},每来一个 job 就被调用一次,函数返回值会被记为该 job 的结果;抛错则视为失败,触发重试3。

// src/queue/click.worker.ts
import { Worker, type Job } from 'bullmq';
import { bullConnection } from './connection.js';
import type { ClickJobData } from './click.queue.js';
import { prisma } from '../db.js';
import { logger } from '../logger.js';

const worker = new Worker<ClickJobData>(
  'clicks',                          // 必须和队列名一致
  async (job: Job<ClickJobData>) => {
    const { linkId, ipAddress, userAgent, referrer, clickedAt } = job.data;

    // 一个事务里同时：插一条 clicks + click_count 自增
    await prisma.$transaction([
      prisma.click.create({
        data: { linkId, ipAddress, userAgent, referrer,
                clickedAt: new Date(clickedAt) },
      }),
      prisma.link.update({
        where: { id: linkId },
        data: { clickCount: { increment: 1 } },
      }),
    ]);
  },
  {
    connection: bullConnection,
    concurrency: 10,   // 同时最多处理 10 个 job（见下一节）
  },
);

worker.on('completed', (job) => {
  logger.debug({ jobId: job.id }, 'click recorded');
});
worker.on('failed', (job, err) => {
  logger.error({ jobId: job?.id, err }, 'click job failed');
});
🌍 实战：Worker 通常作为独立进程部署,和 Web 服务器分开。原因:① 写库的 CPU/IO 不抢占处理 HTTP 请求的资源;② 可以独立扩容(点击暴涨就多开几个 worker 进程,Web 进程不动);③ 一边崩溃不拖垮另一边。在 package.json 里通常会有两条启动命令,例如 "start": "node dist/server.js" 和 "worker": "node dist/queue/click.worker.js",分别用不同的容器/进程跑。多个 worker 进程连同一个 Redis 队列,BullMQ 保证一个 job 只被一个 worker 取走,不会重复处理。

4重试、失败处理与并发

队列真正值钱的地方在于「失败不是终点」。在 defaultJobOptions(或单个 add 的第三参)里我们配了:

  • attempts: 3:job 抛错后最多再试 2 次(共 3 次)。3 次都失败才进「failed」状态。
  • backoff: { type: 'exponential', delay: 1000 }:退避(backoff)=失败后不立刻重试,而是等一会儿。指数退避意味着等待时间翻倍:1s → 2s → 4s。这能给「临时性故障」(数据库瞬断、网络抖动)恢复的时间,避免疯狂重试雪上加霜。
🌍 实战 · job 必须幂等(idempotent): 因为 job 可能被重试,你的处理逻辑要保证「执行一次和执行多次效果相同」,否则重试会把 click_count 多加几次、邮件发好几封。常见做法:① 给数据加唯一约束 + 用「存在即跳过」的写法;② 入队时指定一个稳定的自定义 jobId——BullMQ 规定:如果用已存在的 id 添加 job,该 job 会被直接忽略、不入队,天然去重4。注意 jobId 不能含冒号 :,也不能是纯数字字符串4:
// 用 “事件指纹” 当 jobId，重复事件不会被入队两次
await clickQueue.add('record-click', data, {
  jobId: `click-${linkId}-${requestId}`, // 用 - 不用 :，且非纯数字
});

并发(concurrency): 默认一个 worker 一次只跑一个 job。concurrency: 10 让单个 worker 进程同时处理最多 10 个 job——因为我们的活儿是「等数据库 IO」(非 CPU 密集),并发等待能大幅提升吞吐。但别盲目调大:并发数 × worker 进程数 = 同时打向数据库的连接数,要和你的 PG 连接池上限对齐,否则把数据库连接占满反而更糟。

失败的 job 去哪了? attempts 用尽后,job 进入 failed 集合(我们用 removeOnFail: 5000 保留最近 5000 条)。生产中你会接一个监控面板(如 Bull Board)查看失败原因,或对特定失败做死信(dead-letter)处理。先知道「失败不会无声消失」就够了。

5定时 / 重复任务（类 cron）

有些活儿不是「事件触发」,而是「按时间触发」:每小时聚合一次统计、每天凌晨清理过期数据、每周给用户发周报。BullMQ 用重复任务(repeatable jobs)实现,支持两种节奏5:

  • cron 表达式:pattern: '0 * * * *' = 每小时第 0 分执行(分 时 日 月 周)5。
  • 固定间隔:every: 60000 = 每 60 秒一次5。

给 LinkPulse 加一个「每小时把热门短链汇总并刷新缓存」的定时任务示例。先建一条队列并注册重复 job:

// src/queue/stats.queue.ts
import { Queue } from 'bullmq';
import { bullConnection } from './connection.js';

export const statsQueue = new Queue('stats', { connection: bullConnection });

// 启动时注册一次：每小时整点跑一次聚合
export async function scheduleHourlyAggregate(): Promise<void> {
  await statsQueue.add(
    'hourly-aggregate',
    {},                                     // 这个 job 不需要 payload
    {
      repeat: { pattern: '0 * * * *' },     // cron：每小时第 0 分
      jobId: 'hourly-aggregate',            // 固定 id，避免重复注册堆积
    },
  );
}

再写它的 worker(可以和点击 worker 放在同一个 worker 进程里,也可以分开):

// src/queue/stats.worker.ts
import { Worker } from 'bullmq';
import { bullConnection } from './connection.js';
import { prisma } from '../db.js';
import { logger } from '../logger.js';

new Worker(
  'stats',
  async () => {
    // 示例：算出点击量 Top 10 的短链，可写入缓存供 dashboard 用
    const top = await prisma.link.findMany({
      orderBy: { clickCount: 'desc' },
      take: 10,
      select: { shortCode: true, clickCount: true },
    });
    logger.info({ top }, 'hourly aggregate done');
  },
  { connection: bullConnection },
);
🌍 实战 · 队列还能做这些定时/异步活: ① 发邮件——新用户注册后把欢迎邮件入队,注册接口立刻返回,邮件 worker 慢慢发;② Webhook 重试——给客户回调失败时,靠 attempts+backoff 自动重投,这正是队列最经典的场景;③ 定时聚合/报表——每天凌晨低峰期跑重活,生成报表;④ 清理——定期删过期数据。LinkPulse 后面的「计费」「邮件通知」章节都会复用这套队列基础设施。
定时任务的进程归属: 重复 job 的「触发」由 Redis 里的调度记录驱动,只要有 worker 监听对应队列,到点就会冒出一个普通 job 被消费。注册(scheduleHourlyAggregate)只需在启动时调用一次;用固定 jobId 可避免每次重启都堆积出新的重复调度。

6改造 LinkPulse 跳转：从同步写库到异步入队

现在动真格。看改造前后的对比——核心变化是把「写库 + 自增」从跳转 handler 里整段搬走,只留一行「入队」:

❌ 改造前（同步写库，慢）

// redirect.controller.ts
router.get('/:code', async (req, res) => {
  const link = await getLinkByCode(req.params.code);
  if (!link) return res.sendStatus(404);

  // 用户被卡在这两步写库上 ↓
  await prisma.click.create({
    data: { linkId: link.id, /* ... */ },
  });
  await prisma.link.update({
    where: { id: link.id },
    data: { clickCount: { increment: 1 } },
  });

  res.redirect(302, link.targetUrl);
});

✅ 改造后（异步入队，快）

// redirect.controller.ts
import { clickQueue } from '../queue/click.queue.js';

router.get('/:code', async (req, res) => {
  const link = await getLinkByCode(req.params.code);
  if (!link) return res.sendStatus(404);

  // 先放人走（这步极快）↓
  res.redirect(302, link.targetUrl);

  // 跳转后再入队，绝不阻塞用户
  void clickQueue.add('record-click', {
    linkId: link.id,
    ipAddress: req.ip ?? null,
    userAgent: req.get('user-agent') ?? null,
    referrer: req.get('referer') ?? null,
    clickedAt: new Date().toISOString(),
  }).catch((err) => logger.error({ err }, 'enqueue click failed'));
});

几个值得讲透的细节:

  • 先 res.redirect 再 add:响应一发出,用户的浏览器就开始跳转了,后面入队耗时用户完全感知不到。
  • void + .catch:我们故意不 await 入队(await 会把入队耗时算回响应时间)。但 fire-and-forget 的 Promise 一旦 reject 会变成 unhandled rejection,所以挂一个 .catch 记日志。void 是明确告诉 TS/读者「我知道我没等它」。
  • 时间戳在入队时定:clickedAt 用入队那一刻的时间,而不是 worker 写库的时间——否则队列积压时统计时间会偏。
  • worker 做的事(写 clicks + click_count 自增)放进事务:见第 3 节,保证两步要么都成、要么都败,失败整体重试,配合幂等不会重复计数。
