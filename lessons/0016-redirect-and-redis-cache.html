Lesson 0016 · Node.js 后端之路

短链跳转 + 用 Redis 缓存热点短链
（LinkPulse 核心闭环达成 🎉）

这一课，LinkPulse 终于「活」了。 前面你已经能创建短链并存进数据库（0015）。但短链的灵魂是——别人点一下 yourdomain.com/ab12cd，就跳转到原始长网址。这一课实现这个跳转；再用 Redis 给热门短链加缓存，让跳转快到飞起。学完，LinkPulse 的「创建 → 跳转 → 缓存」核心闭环就打通了。

1重定向（Redirect）：短链的本质

短链跳转，靠的是 HTTP 的重定向（redirect）机制：服务器不直接返回内容，而是回一个状态码 3xx + 一个 Location 响应头，告诉浏览器「你要的东西在另一个地址，自己再去那儿」。浏览器收到后会自动再发一次请求到新地址。

Express 把这件事封装成了一个方法 res.redirect()。默认情况下，它发的是 302 状态码1：

res.redirect('https://example.com')        // 默认 302（临时重定向）
res.redirect(301, 'https://example.com')   // 显式指定 301（永久重定向）

301 还是 302？这对短链很关键

  • 301 Permanent（永久）：告诉浏览器/搜索引擎「这个地址永久搬到新址了」。浏览器会缓存这个跳转，下次直接去新址、不再请求你的服务器。
  • 302 Found（临时）：「暂时去那边」。浏览器每次都会先来问你的服务器，再跳转1。
🌍 实战：短链几乎都用 302，不用 301。 道理在于——LinkPulse 要统计每一次点击。如果用 301，浏览器缓存了跳转，用户第二次点同一个短链时根本不会经过你的服务器，你就统计不到这次点击了！用 302，每次点击都会先打到你的服务器，你才能记一次点击、再放行。「想统计点击」就用 302——这是所有短链服务（Bitly 等）的共识。

📌 顺带一提：上面说的 301/302 是给 GET 用的。如果要重定向一个 POST 请求并保持方法不变，对应的是 307（临时）和 308（永久）。短链跳转是 GET，用 302 即可。

2实现跳转路由

逻辑很直白：拿到短码 → 查数据库找到原始 URL → 记一次点击 → 302 跳过去。用上你学过的路由参数（0005）和 Prisma（0014/0015）：

// src/routes/redirect.ts
import { Router } from 'express'
import { prisma } from '../db'

const router = Router()

// GET /:code  —— 例如 /ab12cd
router.get('/:code', async (req, res, next) => {
  try {
    const { code } = req.params
    const link = await prisma.link.findUnique({ where: { shortCode: code } })

    if (!link) {
      return res.status(404).send('短链不存在')
    }

    // 记一次点击（异步，别阻塞跳转——这里先简单 await，后续可优化成「先跳转后异步记录」）
    await prisma.click.create({ data: { linkId: link.id } })

    // 302 临时重定向到原始长链接
    res.redirect(302, link.targetUrl)
  } catch (err) {
    next(err)   // 交给 0007 学的统一错误处理中间件
  }
})

export default router
⚠️ 实战坑：/:code 是「贪吃」路由，必须放在最后挂载。 /:code 会匹配几乎所有路径（/abc、/api、/health 都被它吃掉）。所以在 app.ts 里，要把它放在 所有 /api/... 业务路由之后再挂载，否则它会把你的 API 请求也劫持去当短码查。这正是 0006「中间件/路由顺序很重要」的又一次实战体现：
app.use('/api/links', linksRouter) → 先
app.use('/', redirectRouter) → 后（兜底）

3为什么要缓存？Redis 是什么

想象 LinkPulse 上有个爆款短链，一分钟被点几万次。按上面的代码，每一次点击都要查一次 PostgreSQL（findUnique）。数据库虽然不慢，但它要读磁盘、走网络、处理事务——当 QPS 极高时，这会成为瓶颈，而且跳转应该越快越好（用户在等着跳走）。

解法是缓存（cache）：把「短码 → 长 URL」这种读多写少、又要极快的数据，放进一个内存里的高速存储，避免每次都查数据库。这个存储，最常用的就是 Redis。

  • Redis 是一个内存键值（key-value）数据库：数据存在内存里（不是磁盘），所以读写是微秒级，比查 PostgreSQL 快几个数量级。
  • 最基础的两个操作：SET key value（存）、GET key（取）——就像一个超大、超快、所有进程共享的 Map。
  • 还能给 key 设过期时间（TTL），到点自动删除——天生适合做缓存。
🌍 实战：Redis 远不止做缓存。 它在真实后端里还常用来做：会话存储（session）、限流（记一个 IP 每分钟请求数）、排行榜/计数器、消息队列 / 发布订阅。LinkPulse 后面的「实时点击数」「接口限流」也会用到它。先从最经典的「缓存」用起。

4用 Docker 起 Redis + 连接

和 0011 起 PostgreSQL 一样，用 Docker 一条命令就能跑一个 Redis：

docker run --name linkpulse-redis -p 6379:6379 -d redis

Node 端用 ioredis 这个流行的客户端连接它2：

// src/redis.ts
import Redis from 'ioredis'

// 默认就连 127.0.0.1:6379；生产从 env 读连接串
export const redis = new Redis(process.env.REDIS_URL ?? 'redis://localhost:6379')

📌 和数据库一样，REDIS_URL 该放进 .env（0009 学的配置管理），不要硬编码。

5cache-aside：最常用的缓存模式

知道了 Redis，关键是怎么用对。最经典的缓存模式叫 cache-aside（旁路缓存，也叫 lazy loading）：应用自己负责读写缓存和数据库，缓存只是「旁路」3。流程是：

请求短码→ 查 Redis→ 命中？直接用 ✅
Redis 未命中→ 查数据库→ 写回 Redis(带 TTL)→ 返回
  1. 先查缓存：缓存里有（cache hit）→ 直接用，不碰数据库。
  2. 缓存没有（cache miss）→ 查数据库，拿到后写回缓存（并设个 TTL），再返回。
  3. 下次同样的请求就会命中缓存了。

这就是「缓存在旁边、数据库是真相源」：第一次慢（要查库 + 回填），之后都快。3

6给跳转加上缓存

把第 2 节的跳转路由改造成 cache-aside（用 TypeScript）：

router.get('/:code', async (req, res, next) => {
  try {
    const { code } = req.params
    const cacheKey = `link:${code}`

    // ① 先查 Redis
    let targetUrl = await redis.get(cacheKey)

    // ② 未命中 → 查数据库 → 回填缓存
    if (!targetUrl) {
      const link = await prisma.link.findUnique({ where: { shortCode: code } })
      if (!link) return res.status(404).send('短链不存在')

      targetUrl = link.targetUrl
      // 写回缓存，设 1 小时过期（EX = 秒）
      await redis.set(cacheKey, targetUrl, 'EX', 3600)
    }

    // ③ 记点击（命中缓存也要记！点击统计不能漏）
    //    注意：这里仍需 link.id，可在缓存里存 JSON，或为简单先按需查；
    //    生产中常把「记点击」做成异步任务，避免拖慢跳转。
    res.redirect(302, targetUrl)
  } catch (err) {
    next(err)
  }
})
⚠️ 实战：缓存一致性——更新/删除短链时要让缓存失效。 如果用户改了某条短链的目标 URL，而 Redis 里还存着旧值，用户就会被跳到旧地址。解决：在「更新/删除短链」的逻辑里，顺手 redis.del(\`link:\${code}\`) 删掉缓存。缓存最难的不是写，是「什么时候让它失效」——这是后端经典难题（"there are only two hard things in CS: cache invalidation and naming things"）。TTL 是一道兜底防线（最坏 1 小时后自动同步），主动 del 是即时防线。
