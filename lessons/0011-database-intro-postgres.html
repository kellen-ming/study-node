Lesson 0011 · Node.js 后端之路

数据库入门：为什么选 PostgreSQL
用 Docker 把它跑起来

为什么学这个？ 到 Lesson 0006 为止，你的 POST /api/links 返回的是一个假短码——它没存到任何地方，服务一重启就全没了。 一个能上线收费的 LinkPulse，必须把短链、点击记录、用户、订阅这些数据持久地保存下来，并且在很多人同时访问时还能正确、可靠。这正是数据库（database）的工作。 这一课我们先想清楚为什么要数据库、为什么 LinkPulse 选 PostgreSQL，再用 Docker 一条命令把一个真正的 Postgres 跑起来、连上它。下一课（0012）我们就在里面用 SQL 建表、写数据。

1为什么需要数据库：内存里的「假短码」有什么问题

先回忆你现在的代码。Lesson 0006 的 POST /api/links 大致是这样（用伪代码表示）： 

// 现状：短码只活在内存里
app.post('/api/links', (req, res) => {
  const url = req.body.url;
  const code = Math.random().toString(36).slice(2, 8); // 假短码
  res.json({ code, url });   // ← 返回完就忘了，哪儿都没存
});

就算你把它存进一个模块级变量（比如 const links = []），它仍然只活在 Node 进程的内存里。这会带来三个致命问题，正好对应数据库要解决的三件事：

问题一：持久化（Persistence）——重启就丢

持久化指的是：数据被写到磁盘这种「断电也不消失」的介质上，进程重启、机器重启都还在。内存（RAM）是易失（volatile）的——进程一退出，里面的东西就归零。

内存方案用户创建 100 条短链
→
你部署新版本
（重启进程）
→
100 条全没了 ✗

类比：内存像白板（下班擦掉），数据库像账本（写上去就一直在）。SaaS 卖的就是「你的数据一直安全地在那」——丢数据等于丢客户。

问题二：并发安全（Concurrency）——多人同时写会乱

你的后端会被很多请求同时打到。假设两个用户在同一毫秒都想生成短码 abc123，纯内存的数组没有任何机制阻止「两条一样的短码」同时写进去——短链就撞车了，A 用户点开会跳到 B 用户的目标网址。

数据库专门处理这种并发（concurrency）：它用事务（transaction）和约束（constraint，如「这一列必须唯一」）来保证「即使一万个请求同时来，结果也是一致、正确的」。PostgreSQL 以可靠性著称1，并发正确性正是它的核心价值之一。这部分原理我们后面专门讲，这里你先建立印象：并发正确，是自己用数组手搓搞不定、必须交给数据库的事。

问题三：强大查询（Querying）——内存里筛数据要自己写循环

LinkPulse 上线后你会想问：「上周哪条短链点击最多？」「这个用户名下所有短链按创建时间倒序的前 20 条是什么？」用内存数组，你得自己写一堆 filter / sort / slice，数据一多就慢、就难维护。数据库用 SQL 一句话就能表达这种查询，而且内部有索引（index）帮你查得飞快（下一课会用到）。

🌍 实战：「先用内存数组/全局变量顶着，以后再换数据库」是新手最常见、也最容易反噬的决定。一旦业务跑起来，数据迁移比一开始就用数据库痛苦十倍。正确做法是：项目第一天就接上数据库，哪怕表里只有一行。我们这门课就是这么做的——这一章把库跑起来，下一章马上往里写真数据。

2关系型 vs 文档型：SQL 还是 NoSQL

「数据库」是个大家族。你最常听到两类：关系型数据库（Relational / SQL）和文档型数据库（Document / 一种 NoSQL）。先用前端能秒懂的方式说清区别。

关系型（SQL）· 如 PostgreSQL、MySQL

数据存在表（table）里：像一张 Excel——固定的列（column）、一行行记录（row）。表之间用关系连起来（哪条点击属于哪条短链）。

  • 强 schema：每列类型预先定好，写错类型直接报错（数据更干净）
  • 天生擅长关联多张表查询（JOIN）
  • 事务 / 约束强，数据一致性有保证
  • 用标准语言 SQL 查询

文档型（NoSQL）· 如 MongoDB

数据存成一个个 JSON 文档，扔进集合（collection）里。每个文档结构可以不一样，像前端直接存 JS 对象。

  • 弱 / 无 schema：字段随时加，灵活但容易脏
  • 读「一整个对象」很顺手，跨集合关联偏弱
  • 水平扩展、海量非结构化数据是其长处
  • 各家自有查询 API，不是标准 SQL

一个常见误解：「SQL 老、NoSQL 新、所以 NoSQL 更好」。不对。它们解决的是不同问题：

  • 数据之间关系清晰、需要强一致（电商订单、账务、SaaS 的用户/订阅）→ 关系型更稳。
  • 结构松散、变化快、要存海量日志/画像、能接受最终一致 → 文档型 / 其他 NoSQL 有优势。
澄清两个词：「SQL」严格说是查询语言，「关系型数据库」是数据库种类；日常口语里大家用「SQL 数据库」指代关系型，「NoSQL」泛指非关系型的一大类（文档型只是其中一种，还有键值、列式、图等）。本课不展开 NoSQL 细节，知道「有这么一类、各有适用场景」即可。

3为什么 LinkPulse 选 PostgreSQL

先认识它：PostgreSQL（常简称 Postgres）是一个开源的对象-关系型数据库系统（object-relational database system），有 35 年以上的活跃开发历史，以可靠性、功能完整、性能著称1。它高度遵循 SQL 标准——到 PostgreSQL 18 已满足 SQL:2023 核心标准 177 项强制特性中的至少 170 项1，所以你学到的 SQL 知识是通用、可迁移的。

对 LinkPulse 这种短链 + 分析 + 订阅的 SaaS，选 Postgres 的理由很具体：

  • 关系天然清晰：「一个用户拥有多条短链，一条短链产生多次点击」——这是教科书般的一对多关系，关系型表 + 外键表达得干净利落（第 4 节讲外键）。
  • 可靠、一致：订阅状态、计费、配额这些数据错一条都不行，Postgres 的事务和约束给你兜底。
  • 查询强：访问分析要做各种统计聚合（按天、按地区、按来源），SQL + 索引能高效搞定；Postgres 还原生支持 JSONB，需要时也能存半结构化数据——关系型为主、文档能力随时备用。
  • 生态强、托管多：Node 有成熟驱动（pg，下一阶段会用），云上有大量托管 Postgres（见下方实战），从本地到生产路径顺滑。
🌍 实战：2020 年后大量新创 SaaS 的默认起手式就是 「Postgres 打底，需要时再加专用组件」：用它存核心业务数据，缓存交给 Redis、全文搜索交给专门引擎或 Postgres 自带的全文检索。先用一个可靠的关系型库把业务跑通，远比一上来堆五种数据库明智——LinkPulse 全程只用 Postgres 也完全够用到付费上线。

4基础术语：数据库 / 表 / 行 / 列 / 主键 / 外键

在敲命令前，把几个核心词一次讲清。官方定义其实非常简洁2：

数据库集群 cluster —— 一个 Postgres 服务实例管理的所有数据库的集合 └─ 数据库 database —— 一组表的集合（如 linkpulse） 　　└─ 表 table —— 一个有名字的「行的集合」（如 links） 　　　　├─ 列 column —— 表的「字段」，每列有固定的数据类型 　　　　└─ 行 row —— 一条记录，拥有该表同一组命名的列

官方原话：「关系型数据库管理系统（RDBMS）就是管理存在关系（relation，其实就是表的数学叫法）里的数据的系统；每张表是一个命名的行的集合，同一张表里每一行都有同一组命名的列，每列有特定的数据类型；表被分组进数据库，由单个服务实例管理的一组数据库构成集群。」2

前端类比一句话带过：

  • 表 ≈ 一个 TypeScript 的对象数组的「形状定义」；列 ≈ 接口里的字段 + 类型；行 ≈ 数组里的一个对象。

还有两个新概念必须讲透，因为它们是关系型数据库的灵魂：

主键 Primary Key（PK）：每行的「身份证」

主键是表里一列（或几列），它的值唯一且不为空，用来唯一标识每一行。就像每个人有唯一身份证号。LinkPulse 的 links 表会有一个 id 主键；短码 code 则会加唯一约束，保证不撞车。

外键 Foreign Key（FK）：连接两张表的「引用」

外键是一列，它的值必须指向另一张表某行的主键，以此表达「谁属于谁」。这是「关系型」里「关系」二字的来源。 

users 表                       links 表
┌────┬─────────┐              ┌────┬───────┬───────────┐
│ id │ email   │              │ id │ code  │ user_id ← FK
├────┼─────────┤              ├────┼───────┼───────────┤
│ 1  │ a@x.com │   ←──┐       │ 10 │ abc12 │    1      │
│ 2  │ b@x.com │      └───────│ 11 │ xyz98 │    1      │  这两条短链都属于用户 1
└────┴─────────┘              └────┴───────┴───────────┘
            外键 user_id 必须等于某个真实存在的 users.id

外键的妙处在于数据库帮你守规矩：你无法插入一条 user_id=999 但 users 里根本没有 999 的短链——这叫引用完整性（referential integrity）。在内存数组里，这种「孤儿数据」全靠你自己小心；在 Postgres 里，它从源头上被禁止。这些表和外键我们下一课（0012）正式动手建，本课先把概念吃透、把库跑起来。

5用 Docker 一条命令把 Postgres 跑起来

传统上「装数据库」意味着在本机安装一堆软件、改系统配置，卸载还卸不干净。我们换个现代做法：用 Docker。

Docker 是什么（轻量版）：它能把一个软件（比如 Postgres）连同它需要的运行环境一起打包成一个镜像（image）；你用这个镜像启动一个容器（container）——容器是一个隔离的小盒子，里面的 Postgres 跑得好好的，却不碰你本机的系统。类比：镜像像「软件的安装光盘 + 操作系统」，容器像「用这张盘开起来的一台一次性小电脑」。本课你只需把它当成「起数据库的遥控器」，深入留到后面的运维阶段。

前置：先安装 Docker Desktop（Mac / Windows）并把它打开（菜单栏出现鲸鱼图标即就绪）。验证：终端运行 docker --version 能打印版本号即可。本课不教安装 Docker 本身，遇到问题告诉我。

确认 Docker 在跑后，一条命令起一个 Postgres：

docker run --name linkpulse-db \
  -e POSTGRES_PASSWORD=devpassword \
  -e POSTGRES_USER=linkpulse \
  -e POSTGRES_DB=linkpulse \
  -p 5432:5432 \
  -v linkpulse_pgdata:/var/lib/postgresql/data \
  -d postgres:16

官方镜像页给出的最小示例其实只要密码这一项3：docker run --name some-postgres -e POSTGRES_PASSWORD=mysecretpassword -d postgres。我们在它基础上把每个参数讲清：

参数作用
docker run根据镜像启动一个新容器。
--name linkpulse-db给容器起个名字，方便之后 docker stop / start linkpulse-db 引用，不用记一长串 ID。
-e POSTGRES_PASSWORD=…唯一必填的环境变量：设超级用户密码3。-e = environment（往容器里塞环境变量）。本地随便设，别用真密码。
-e POSTGRES_USER=linkpulse可选：创建这个用户（拥有超级权限）。不写则默认用户名 postgres3。
-e POSTGRES_DB=linkpulse可选：初始化时建一个同名数据库。不写则默认用 POSTGRES_USER 的名字建库3。
-p 5432:5432端口映射：把本机的 5432 转发到容器内的 5432（Postgres 默认端口）。格式是 -p 本机端口:容器端口。有了它，你本机的程序（你的 Node 服务、psql）才连得进去。
-v linkpulse_pgdata:/var/lib/postgresql/data数据卷（volume）：把容器内存数据的目录映射到一个由 Docker 管理的命名卷。关键：删了容器，数据还在这个卷里——下次重建照样能读到。不加它，容器一删数据就真没了。3
-ddetached，后台运行，终端立刻还给你（不会被日志占住）。想看日志：docker logs -f linkpulse-db。
postgres:16用哪个镜像 : 标签（版本）。务必钉版本（这里 16），别用 latest——否则哪天镜像升级，你团队各人版本就不一致了。
反斜杠 \ 是在 Mac/Linux 的 zsh/bash 里把一条长命令拆成多行的续行符（让命令更好读）。复制时连 \ 一起复制即可；若在 Windows PowerShell，请把它写成一行（去掉所有 \ 和换行）。

常用配套命令（先混个眼熟）：

docker ps                 # 看正在跑的容器（应能看到 linkpulse-db、0.0.0.0:5432）
docker logs linkpulse-db  # 看 Postgres 启动日志，出现 "database system is ready" 就绪
docker stop linkpulse-db  # 停（数据不丢，在卷里）
docker start linkpulse-db # 再启动（沿用同一份数据）
docker rm -f linkpulse-db # 删容器（卷仍在；想连卷一起删要另外 docker volume rm）
🌍 实战：「用 Docker 起数据库」是当下本地开发的事实标配。 好处是：① 不污染本机（不用全局装 Postgres，删容器干干净净）；② 一键重建、版本统一（团队都跑 postgres:16，杜绝「在我机器上是好的」）；③ 一个项目想要一套独立的库，再 docker run 一个换个端口即可。等学到运维阶段，这套 docker run 通常会写进 docker-compose.yml3，一行 docker compose up 起全套——本课先体会单条命令。

6连接串 DATABASE_URL：怎么「找到并登录」这个库

库跑起来了，程序怎么连进去？需要五样信息：用什么协议、用谁的身份、密码是啥、连哪台机器的哪个端口、进哪个数据库。把它们拼成一个标准 URL，就是连接串（connection string / connection URI）。PostgreSQL 官方定义了这个 URI 格式4： 

postgresql://[user[:password]@]host[:port][/dbname][?参数]

对照我们上面起的容器，连接串就是（postgres:// 和 postgresql:// 两种写法等价，官方都接受4）：

postgres://linkpulse:devpassword@localhost:5432/linkpulse
scheme 协议：固定 postgres://，告诉客户端「这是个 Postgres 连接」。
user 用户名：linkpulse（我们用 POSTGRES_USER 设的）。
password 密码：devpassword。
host 主机：localhost（容器通过 -p 映射到了本机）。
port 端口：5432（Postgres 默认；省略则用默认端口4）。
dbname 数据库名：linkpulse。

下一阶段我们会用 Node 的官方 Postgres 驱动 pg 连库，它的标准用法正是把这串喂给 connectionString5，而约定俗成的环境变量名就叫 DATABASE_URL（Heroku 等平台带火的惯例5）： 

// 预告：下一阶段会这样连（本课不用写，先看一眼形状）
import { Pool } from 'pg';
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

所以我们现在就把这串放进项目根目录的 .env 文件——你在 TypeScript 项目（0010 起）里已经习惯用环境变量管理配置了：

# study-node/.env
DATABASE_URL=postgres://linkpulse:devpassword@localhost:5432/linkpulse
🌍 实战：连接串里有密码，所以 .env 绝不能进 Git。 务必在 .gitignore 里加上 .env，并提交一份不含真值的 .env.example（值留空或写假值）给队友照着填。把密码、连接串这类密钥（secrets）写进环境变量、而不是硬编码进源码，是后端最基本的安全红线——这次泄露的可能只是本地库，等接了生产托管库，泄露就是事故。
🌍 实战：本地用 Docker，生产用托管 Postgres。 真要上线 LinkPulse，没人自己拿裸机扛数据库——大家用托管服务：Supabase、Neon（都对小项目友好、有免费档）、或 AWS RDS。它们替你搞定备份、高可用、打补丁，并直接给你一条 DATABASE_URL（通常还要求 ?sslmode=require）。美妙之处：因为你的代码只认 process.env.DATABASE_URL，从本地 Docker 切到云上托管，改的只是 .env 里那一行，代码一字不动。

7连上去看一眼：psql 或 GUI

怎么确认库真的能用？连进去看一眼。两条路任选：

路线 A：psql（命令行，最快验证）

psql 是 Postgres 官方自带的交互式命令行客户端。容器里就装着它，所以不用在本机额外装任何东西，直接进容器用3： 

docker exec -it linkpulse-db psql -U linkpulse -d linkpulse

docker exec=在正在跑的容器里执行命令；-it=交互式（你能输入）；后面是「用 linkpulse 用户连 linkpulse 库」。看到提示符变成 linkpulse=# 就成功了。试几条：

linkpulse=# SELECT version();   -- 打印 PostgreSQL 版本，证明连通
linkpulse=# \l                  -- 列出所有数据库（应看到 linkpulse）
linkpulse=# \dt                 -- 列出当前库的表（现在是空的，正常！）
linkpulse=# \q                  -- 退出
以 \ 开头的是 psql 的元命令（meta-command，psql 自己的快捷指令，不是 SQL）；以分号结尾的才是发给数据库执行的 SQL。现在 \dt 显示「没有表」完全正确——建表是下一课的事。

路线 B：GUI 客户端（看得更直观）

如果你更习惯图形界面，装一个数据库 GUI：TablePlus（轻巧好看）或 DBeaver（免费、功能全）。新建连接时，把上面 .env 里那几项填进去即可：

Host: localhost Port: 5432 User: linkpulse Password: devpassword Database: linkpulse

连上后你能在侧边栏点开数据库、看到（暂时为空的）表列表，以后建了表能直接点开看数据、改数据。前端工程师通常更喜欢这条路——所见即所得。

🌍 实战：日常开发里大多数人两样都用：写脚本、做 CI、快速 SELECT 验证用 psql（能进容器、能塞进自动化）；浏览数据、调试一条复杂记录、给非技术同事演示用 GUI（直观）。先有一个能连上的客户端，比纠结用哪个更重要。

⌨️ 动手：把 LinkPulse 的数据库跑起来并连上

  1. 装好并打开 Docker Desktop；终端跑 docker --version 确认可用。
  2. 用第 5 节那条命令起容器（建议照抄，含 --name linkpulse-db、-p 5432:5432、-v 卷、钉死 postgres:16）。
  3. 跑 docker ps，确认 linkpulse-db 在跑、且 PORTS 列出现 0.0.0.0:5432->5432；再 docker logs linkpulse-db 看到 database system is ready to accept connections。
  4. 在项目根目录（study-node）新建 .env，写入：
    DATABASE_URL=postgres://linkpulse:devpassword@localhost:5432/linkpulse
  5. 在 .gitignore 里加一行 .env；再建一个 .env.example，内容把密码换成占位（如 ...:你的密码@...）。
  6. 用 psql（docker exec -it linkpulse-db psql -U linkpulse -d linkpulse）或 TablePlus/DBeaver 连上。
