产品化收官：Stripe 订阅计费 + 上线清单

为什么学这个？ 到现在 LinkPulse 已经能注册登录、创建/跳转/缓存短链、看实时数据、跑测试、容器化部署。但它还差最后一块——怎么把它变成能真正收钱的生意。本章给 LinkPulse 接入 Stripe 订阅计费：用户能从「免费版」一键升级成 pro，每月自动扣费；免费版限制最多 N 条短链。学完这一章，LinkPulse 就是一个真实可上线运营、能持续收入的 SaaS。最后我们用一份「上线清单」做收尾，并回顾你走过的整条路（0001 → 0026）。

1SaaS 计费模型 & Stripe 的四个核心对象

先把心智模型立起来。SaaS（软件即服务）最常见的收费方式是订阅制（subscription）：用户不是一次性买断，而是按月/按年自动续费。这背后有一堆活儿：收信用卡、判断卡是否有效、每月自动扣款、扣款失败要重试、用户退款、发票、对账……自己做这些等于自建一家小银行，还要过 PCI 合规（处理信用卡数据的安全标准）。

Stripe 就是把这些全包了的支付基础设施。你只跟 Stripe 的 API 打交道，信用卡号永远不经过你的服务器（合规风险归 Stripe），你只负责「谁付了钱、给他开通什么权益」。

Stripe 的计费世界由四个对象搭成，理解它们的关系是本章的地基1：

对象是什么类比在 LinkPulse 里
Product
(产品)		你卖的东西本身，跟价格无关菜单上的「Pro 套餐」这道菜"LinkPulse Pro"
Price
(价格)		挂在 Product 上的具体定价：金额+币种+计费周期（recurring）这道菜「¥39 / 月」这个标价牌$9/月 的 recurring price
Customer
(客户)		Stripe 里代表「一个付款人」的对象，存着支付方式餐厅给你建的会员档案对应一个 LinkPulse user
Subscription
(订阅)		把 Customer 和 Price 绑在一起的「正在生效的合同」，Stripe 据此每月自动扣费你签的「每月自动续」的会员卡active = 用户是 pro

关系链一句话：一个 Customer 按某个 Price（属于某个 Product）建立一条 Subscription。Subscription 有状态（active / past_due / canceled 等），我们的任务就是：让 LinkPulse 的 users.plan 字段始终跟 Stripe 里这条 Subscription 的状态保持一致。

Product / Price 在哪建？ 不用写代码——在 Stripe Dashboard 里点几下就能建（测试模式下）。建好后拿到一个 price_xxx 形式的 ID，写进环境变量给后端用。代码层面我们几乎只跟 Customer / Subscription / Checkout Session 打交道。

2最省心的集成：Stripe Checkout（托管支付页）

收信用卡有两种做法：① 自己用 Stripe Elements 在前端拼一个支付表单（灵活但要写很多前端、要处理 3D 验证等）；② Stripe Checkout——Stripe 托管的一整页支付界面，你把用户重定向过去，付完再跳回来。

🌍 实战：对绝大多数 SaaS，Checkout + Webhook 是官方推荐、也是最稳的集成方式。Checkout 页面由 Stripe 维护，自带卡号校验、3D Secure 验证、Apple Pay、各国本地支付方式、防欺诈、移动端适配——这些你一行都不用写，而且永远跟着监管更新。先用 Checkout 上线收钱，等真有定制需求再考虑 Elements。别一开始就自建支付 UI，那是过早优化。

流程很简单：你在后端创建一个 Checkout Session（一次结账会话），Stripe 返回一个 URL，你让用户跳过去；用户付完款，Stripe 把他重定向回你的 success_url。

关键参数2：mode: 'subscription'（订阅模式，区别于一次性付款的 'payment'）、line_items（买什么——填我们的 recurring price ID）、success_url / cancel_url（付成功/取消后跳回哪）。

先装库，并准备 Stripe 客户端单例（贴合我们已有的 ESM + 配置风格）：

npm i stripe
// src/lib/stripe.ts —— 全局唯一的 Stripe 客户端
import Stripe from 'stripe';

if (!process.env.STRIPE_SECRET_KEY) {
  throw new Error('缺少 STRIPE_SECRET_KEY 环境变量');
}

// apiVersion 锁版本：Stripe 会演进 API，固定版本能避免某天行为突变。
// 订阅相关的 flexible billing mode 需要 2025-06-30.basil 或更新。[1]
export const stripe = new Stripe(process.env.STRIPE_SECRET_KEY, {
  apiVersion: '2025-06-30.basil',
});

现在写「创建 Checkout Session」的接口。注意它需要登录态——我们沿用之前章节的认证中间件，从 req.user 拿到当前用户：

// src/modules/billing/billing.service.ts
import { stripe } from '../../lib/stripe.js';
import { prisma } from '../../lib/prisma.js';

const PRICE_PRO = process.env.STRIPE_PRICE_PRO!; // price_xxx，Dashboard 里建好

export async function createCheckoutSession(userId: string): Promise<string> {
  const user = await prisma.user.findUniqueOrThrow({ where: { id: userId } });

  // 一个 LinkPulse user 应对应一个固定的 Stripe Customer。
  // 第一次升级时创建并把 id 存回我们的库（建议给 users 表加 stripe_customer_id 列）。
  let customerId = user.stripeCustomerId;
  if (!customerId) {
    const customer = await stripe.customers.create({
      email: user.email,
      metadata: { linkpulseUserId: user.id }, // 反向关联，排查问题时很有用
    });
    customerId = customer.id;
    await prisma.user.update({
      where: { id: user.id },
      data: { stripeCustomerId: customerId },
    });
  }

  const session = await stripe.checkout.sessions.create({
    mode: 'subscription',
    customer: customerId,
    line_items: [{ price: PRICE_PRO, quantity: 1 }],
    success_url: `${process.env.APP_URL}/billing/success?session_id={CHECKOUT_SESSION_ID}`,
    cancel_url: `${process.env.APP_URL}/billing/cancel`,
    // client_reference_id：把我们自己的 userId 写进去，webhook 里能立刻认出是谁。[2]
    client_reference_id: user.id,
  });

  return session.url!; // 让前端 302 跳到这个 URL
}
// src/modules/billing/billing.controller.ts
import type { Request, Response } from 'express';
import { createCheckoutSession } from './billing.service.js';

export async function postCheckout(req: Request, res: Response) {
  const url = await createCheckoutSession(req.user!.id); // 认证中间件已注入 req.user
  res.json({ url }); // 前端拿到 url 后 window.location = url
}
为什么把 userId 同时塞进 client_reference_id 和 Customer 的 metadata？ 因为 webhook 回来时，Stripe 给的是它自己的 customer / session 对象，里面没有你的 userId。靠这两个字段你才能把「Stripe 世界的事件」映射回「LinkPulse 的哪个用户」。这是新手最常踩的断点。

3Webhook：让付款结果真正写进数据库

这里有个反直觉但极其重要的点：用户被重定向回 success_url，不等于钱已到账、权益该开通。用户可能关掉页面、网络可能断、有人可能伪造这个跳转。所以绝不能在 success 页面里给用户升 pro。

真正可信的信号来自 Webhook——Stripe 在事件真实发生时，主动向你预先登记的一个 URL 发 POST 请求。「钱真的扣到了」这种事，由 Stripe 服务器异步告诉你，这才是权益开通的唯一可信来源。3

前端复习：你熟悉的「我发请求、服务器回响应」是客户端拉（pull）。Webhook 反过来——服务器主动推（push），Stripe 当客户端、你当服务器。本质就是一个普通的 POST 路由，只不过调用方是 Stripe。

订阅场景下我们关心几个事件4：

事件含义LinkPulse 该做什么
checkout.session.completed用户在 Checkout 页付款成功、订阅已建立把该用户 plan 设为 pro
customer.subscription.updated订阅状态变了（续费成功、进入 past_due 等）按订阅 status 同步 plan
customer.subscription.deleted订阅被取消/到期终止把 plan 降回 free

整条计费数据流串起来是这样：

1用户点「升级 Pro」→ 前端调 POST /billing/checkout
↓ 后端创建 Checkout Session，返回 url
2浏览器跳到 Stripe 托管页，用户输卡付款
↓ 付款成功
3Stripe 重定向用户回 success_url（仅 UI 提示，不开权益）
↓ 同时，Stripe 服务器异步 POST
4Webhook 收到 checkout.session.completed → 验签 → 更新 users.plan = 'pro'

3.5Webhook 签名校验：安全的生死线

🌍 实战：你的 webhook URL 是公开的，任何人都能往它发 POST。如果不验签，攻击者只要伪造一个 checkout.session.completed 请求，就能白嫖开通 pro。所以验签不是可选项，是必须项。Stripe 用你 endpoint 的密钥（whsec_...）对请求体做了 HMAC 签名放在 Stripe-Signature 头里，你用 SDK 重新算一遍、比对一致才信。

验签靠 stripe.webhooks.constructEvent(payload, sig, secret)5。但有一个所有人都会踩的坑：它需要未经解析的原始请求体（raw body）。

🌍 实战（高频翻车点）：我们之前全局用了 express.json()，它会把请求体解析成 JS 对象并丢掉原始字节。可验签必须用原始字节——哪怕重新 JSON.stringify 也会因为空格/键顺序不同而签名对不上。解决：webhook 路由必须在 express.json() 之前注册，并对它单独用 express.raw()，让这一条路由拿到 Buffer。5

看 app 里中间件的注册顺序（顺序就是一切）：

// src/app.ts（节选，注意顺序）
import express from 'express';
import { webhookRouter } from './modules/billing/webhook.router.js';
import { apiRouter } from './routes.js';

const app = express();

// ① Webhook 必须在 express.json() 之前，且用 raw 拿原始 Buffer
app.use('/webhooks/stripe', express.raw({ type: 'application/json' }), webhookRouter);

// ② 之后才挂全局 JSON 解析，给其余所有业务路由用
app.use(express.json());
app.use('/api', apiRouter);
// src/modules/billing/webhook.router.ts
import { Router, type Request, type Response } from 'express';
import type Stripe from 'stripe';
import { stripe } from '../../lib/stripe.js';
import { prisma } from '../../lib/prisma.js';
import { logger } from '../../lib/logger.js';

export const webhookRouter = Router();
const WEBHOOK_SECRET = process.env.STRIPE_WEBHOOK_SECRET!; // whsec_...

webhookRouter.post('/', async (req: Request, res: Response) => {
  const sig = req.headers['stripe-signature'] as string;
  let event: Stripe.Event;

  // —— 验签：失败直接 400，绝不处理 ——
  try {
    // req.body 此处是 raw 中间件给的 Buffer，正是 constructEvent 需要的
    event = stripe.webhooks.constructEvent(req.body, sig, WEBHOOK_SECRET);
  } catch (err) {
    logger.warn({ err }, 'Stripe 签名校验失败，拒绝');
    return res.status(400).send(`Webhook Error: ${(err as Error).message}`);
  }

  // —— 幂等：Stripe 会重试，同一事件可能来好几次，靠 event.id 去重 ——[4]
  const seen = await prisma.processedWebhook.findUnique({ where: { id: event.id } });
  if (seen) return res.json({ received: true }); // 处理过了，直接确认

  try {
    switch (event.type) {
      case 'checkout.session.completed': {
        const session = event.data.object as Stripe.Checkout.Session;
        const userId = session.client_reference_id; // 我们当初塞进去的
        if (userId) {
          await prisma.user.update({ where: { id: userId }, data: { plan: 'pro' } });
          logger.info({ userId }, '用户升级到 pro ✅');
        }
        break;
      }
      case 'customer.subscription.updated': {
        const sub = event.data.object as Stripe.Subscription;
        const plan = sub.status === 'active' || sub.status === 'trialing' ? 'pro' : 'free';
        await prisma.user.updateMany({
          where: { stripeCustomerId: sub.customer as string },
          data: { plan },
        });
        break;
      }
      case 'customer.subscription.deleted': {
        const sub = event.data.object as Stripe.Subscription;
        await prisma.user.updateMany({
          where: { stripeCustomerId: sub.customer as string },
          data: { plan: 'free' }, // 退订 → 降回免费版
        });
        break;
      }
      default:
        logger.debug({ type: event.type }, '未处理的事件类型');
    }

    await prisma.processedWebhook.create({ data: { id: event.id } }); // 记下已处理
    res.json({ received: true }); // 必须尽快回 200，否则 Stripe 会一直重试
  } catch (err) {
    logger.error({ err, type: event.type }, '处理 webhook 出错');
    res.status(500).json({ error: 'handler failed' }); // 回非 2xx → Stripe 自动重试
  }
});
三条 Stripe 官方铁律4：① 验签每个请求；② 按 event.id 幂等去重（Stripe 保证「至少一次」投递，会重发）；③ 尽快返回 2xx——重活儿别堵在响应里。返回非 2xx 会触发 Stripe 重试，这恰好是我们处理失败时想要的「自动补偿」。需要给 schema 加一张 processed_webhooks(id text primary key) 表来记录已处理事件。

4套餐限额：免费版只能建 N 条短链

收费的意义在于免费版有限制、付费版解锁。LinkPulse 最自然的限额是短链数量：免费版最多 5 条，pro 无限。

🌍 实战：用量/套餐限额一律在业务层（service）判断，不要指望数据库约束或前端隐藏按钮。前端隐藏只是体验，真正的闸门必须在服务端——否则用户直接打 API 就绕过了。判断逻辑放在「创建短链」这个动作的最前面，超限就抛业务错误。

我们的 users 表已经有 plan 字段，直接用：

// src/modules/links/links.service.ts（创建短链时加限额校验）
import { prisma } from '../../lib/prisma.js';

// 各套餐的短链数量上限；pro 用 Infinity 表示无限
const LINK_QUOTA: Record<string, number> = { free: 5, pro: Infinity };

export class QuotaExceededError extends Error {
  status = 403; // 让错误中间件据此返回 403
  code = 'QUOTA_EXCEEDED';
}

export async function createLink(userId: string, targetUrl: string) {
  const user = await prisma.user.findUniqueOrThrow({ where: { id: userId } });
  const limit = LINK_QUOTA[user.plan] ?? LINK_QUOTA.free;

  // 已有多少条？超限就拦下，引导升级
  const count = await prisma.link.count({ where: { userId } });
  if (count >= limit) {
    throw new QuotaExceededError(
      `免费版最多创建 ${limit} 条短链，升级到 Pro 即可无限创建`,
    );
  }

  // 没超限，照常创建（沿用之前 nanoid 生成短码的逻辑）
  return prisma.link.create({
    data: { userId, targetUrl, shortCode: generateShortCode() },
  });
}

配合我们已有的 4 参数错误处理中间件，把 QuotaExceededError 翻成干净的 403 响应，前端就能据此弹出「升级」引导。这样「限额 → 引导升级 → Checkout → webhook 开通 pro → 限额解除」形成了完整闭环。

并发小提醒：「先 count 再 create」在高并发下理论上能被刷过一两条（竞态）。对短链这种弱约束业务，问题不大；若要严格，可用数据库事务 + 行锁，或给 user 维护一个计数列并在事务里 UPDATE ... WHERE count < limit。收官阶段先求能跑，知道有这回事即可。

5本地测 Webhook：Stripe test mode + Stripe CLI

Webhook 最难测——它需要一个公网 URL 让 Stripe 能打进来，可你的 localhost 外网访问不到。官方答案是 Stripe CLI：它在你本机和 Stripe 之间建一条隧道，把云端事件转发到你的 localhost。6

🌍 实战：全程在 test mode（测试模式）做——用测试密钥（sk_test_...）、测试卡号 4242 4242 4242 4242（任意未来日期 + 任意 CVC），不会产生真实扣款。把所有事件路径都在 test mode 跑通、能稳定收到并正确更新 plan，再切真实密钥上线。真金白银的流程，宁可多测十遍。
# 1) 登录（一次性，浏览器授权）
stripe login

# 2) 监听并把事件转发到本地 webhook 路由
stripe listen --forward-to localhost:3000/webhooks/stripe
# 启动后会打印一个 whsec_... —— 这是 CLI 专用的本地签名密钥，
# 注意它和 Dashboard 上线上 endpoint 的密钥不同！[6]
# 把它写进本地 .env 的 STRIPE_WEBHOOK_SECRET 再启动你的服务

# 3) 另开一个终端，手动触发一个事件来测试 handler
stripe trigger checkout.session.completed

完整的本地回路：终端 A 跑 stripe listen，终端 B 跑你的 LinkPulse（npm run dev），浏览器走一遍真实的 Checkout（用测试卡），观察 stripe listen 打印事件、你的服务打印「升级到 pro」日志、数据库里 plan 确实变了。三处对上，说明集成成功。

那线上呢？ 部署后在 Stripe Dashboard → Developers → Webhooks 里注册一个正式 endpoint（你的公网域名 + /webhooks/stripe），Stripe 会给你一个线上专用的 whsec_...，配进生产环境变量。本地用 CLI 的密钥，线上用 Dashboard 的密钥，两套别搞混。

6上线清单（Launch Checklist）

能收钱不等于能上线。一个面向真实用户、要收信用卡的 SaaS，上线前还有一串「成年人的责任」。下面这份清单可以直接抄进你的项目 README，逐条勾掉。

🌍 实战：把它当成真正可勾选的 checklist，上线日一条条过。很多事故不是技术不行，而是「忘了开备份」「告警没配」「忘了限流被刷爆」。清单的价值就是把「我以为做了」变成「确认做了」。
  • HTTPS / TLS：全站强制 HTTPS（反向代理 Nginx/Caddy 或平台自带），Cookie 加 Secure + HttpOnly + SameSite。收信用卡的站点没有 HTTPS 是绝对红线。
  • 密钥与环境隔离：生产用真实 Stripe 密钥，且只存在生产环境变量/密钥管理里，绝不进 git；.env 已在 .gitignore。
  • 限流（Rate limiting）：给登录、注册、创建短链、跳转接口加限流（如 express-rate-limit + Redis），防刷防滥用，保护数据库。
  • 错误追踪（Error tracking）：接入 Sentry 之类，未捕获异常能自动上报，别等用户投诉才知道挂了。
  • 监控告警（Monitoring & alerting）：健康检查 /health、关键指标（错误率、响应时延、DB/Redis 连接），异常时能主动通知到人（邮件/Slack）。
  • 数据备份：PostgreSQL 开启自动定时备份并实测恢复一次——没验证过的备份等于没有备份。
  • Webhook 健壮性：验签 ✓、幂等去重 ✓、失败返回非 2xx 让 Stripe 重试 ✓、Dashboard 里能看到投递日志。
  • 隐私政策 & 服务条款：收集邮箱/IP、接 Stripe 收款，法律上需要 Privacy Policy 与 Terms of Service 页面（GDPR/CCPA 等）。
  • 日志合规：结构化日志（pino 已有）里不要记密码、不要记完整卡号/密钥。
  • 优雅关闭：收到 SIGTERM 时停止接新请求、放完在途请求、关掉 DB/Redis 连接（部署滚动更新时不丢请求）。
  • CI/CD 卡口：测试不过不许部署（上一课已搭），上线走流水线而非手动 SSH 改文件。
  • 套餐限额复核：免费版限额在服务端生效、付费后真的解除、退订后真的降级——端到端走一遍。
