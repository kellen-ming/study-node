Lesson 0001 · Node.js 后端之路

Node 的本质：单线程、非阻塞 I/O，
与代码的执行顺序

为什么先学这个？ 你的目标是独立做出 LinkPulse（短链接 + 访问分析 SaaS）。 当成千上万人同时点击短链时，你的服务器凭什么用一个线程就扛得住？答案就藏在「事件循环」里。 搞懂它，你才能写出不会「卡死」整台服务器的后端代码——这是后端并发的第一块基石。

1Node 到底是什么

先纠正一个常见误解：Node.js 不是一门语言，也不是一个框架，而是一个 JavaScript 运行时（runtime）。官方定义：

“Node.js® is a free, open-source, cross-platform JavaScript runtime environment that lets developers create servers, web apps, command line tools and scripts.”1

它把 Chrome 用的 V8 引擎搬到了浏览器之外，让 JS 能在服务器上运行1。 你已经会写的 JS，现在可以用来写服务器了——这就是你从前端跨到后端最大的「现成优势」。

心智模型：V8（执行你的 JS）＋ libuv（一个 C 库，负责事件循环和与操作系统打交道的异步 I/O）＝ Node 运行时。

2核心直觉：单线程，却不阻塞

关键事实：一个 Node 应用跑在单个进程里，不会为每个请求都新建线程1。 这听起来很反直觉——只有一个线程，怎么同时服务很多用户？

诀窍在于：后端的工作大多是 I/O（读数据库、读文件、等网络），而 I/O 是「等」的过程。 如果像传统同步代码那样「等」，这一个线程就被占住、谁也服务不了。Node 的做法是—— 把「等」这件事外包出去，自己立刻去处理下一个请求。等结果回来了，再回头处理。

🔗 映射到 LinkPulse：用户点击短链 → 你要查数据库拿到原始网址 → 这一步要「等」数据库。 在等的这几毫秒里，单线程的 Node 不会干等，而是继续处理别人的点击。这就是它能高并发的原因。

3事件循环：把慢操作「外包」给内核

让这一切运转起来的机制，就叫 事件循环（Event Loop）。官方解释（请逐字读一遍）：

“The event loop is what allows Node.js to perform non-blocking I/O operations — despite the fact that a single JavaScript thread is used by default — by offloading operations to the system kernel whenever possible.”2

翻译成人话：遇到慢操作（读文件、查数据库），Node 把它交给操作系统内核去后台处理，自己继续往下跑。 内核干完后会通知 Node，于是对应的回调函数（callback）被排进队列，等着被执行2。 事件循环就是那个「不停地从队列里取出已就绪的回调来执行」的循环。

4你要掌握的技能：预测执行顺序

既然回调是「排队」执行的，那它们的先后顺序就有讲究。这是面试爱问、写代码也常踩的点。 记住这条优先级链： 

① 同步代码        → 立即执行（在调用栈上，从上到下）
② 微任务 microtask → 同步代码跑完后、立刻清空
      ├─ process.nextTick(...)   ← Node 里优先级最高
      └─ Promise.then/catch/finally
③ 宏任务 macrotask → 微任务清空后，进入事件循环各阶段
      └─ setTimeout / setImmediate / I/O 回调 ...

规则：同步 → 微任务 → 宏任务。微任务（Promise）总是排在下一个宏任务（如 setTimeout）之前被执行完3。 而在 Node 里，process.nextTick 又比 Promise 更优先——它会在「当前这步操作结束后、事件循环继续之前」就被清空2。

问：为什么 Node.js 用单线程也能同时服务成千上万的用户？

参考答案：因为后端工作大多是「等待型」的 I/O，Node 把这些慢操作外包给操作系统内核去后台处理， 单线程不干等、立刻去处理下一个请求；结果就绪后再通过事件循环执行对应回调——所以一个线程也能高并发2。

💡 能用自己的话说出来，比「看懂了」更重要——这是把知识从「短期流畅」变成「长期记住」的关键。
