Lesson 0007 · Node.js 后端之路

输入校验与错误处理
（永远，永远别信用户输入）

你的 API 现在是「裸奔」的。 上一课你给 LinkPulse 写了 POST /api/links，直接读 req.body.url 就当成网址用—— 没有任何校验。但请求体是客户端发来的，谁都能伪造：它可能是 undefined、是个数字、是 "哈哈哈"、甚至是一段攻击载荷。 这一课你会建立后端工程师的第一条铁律——「永不信任输入」——并用 zod 把进门的数据挡在关卡外，再用 Express 的错误处理中间件统一兜住所有意外。这是 LinkPulse 从「玩具」走向「能收费上线」的第一道安全闸门。

1为什么「永不信任输入」是铁律

先看你上一课的真实代码长什么样（简化版）： 

app.post('/api/links', (req, res) => {
  const url = req.body.url           // ← 盲信！万一 body 里根本没有 url？
  const code = Math.random().toString(36).slice(2, 8)
  res.status(201).json({ code, url })
})

作为前端你很熟「请求体（request body）」这个概念。但在前端，你通常能控制自己发什么；在后端，请求来自整个互联网，发送者可能是：手滑的真实用户、写错的另一个团队、自动化爬虫、或者主动找漏洞的攻击者。所以后端的世界观必须反过来：

🌍 OWASP 的官方立场：安全权威 OWASP 在《Input Validation Cheat Sheet》里明确写道——客户端（浏览器里的 JavaScript）校验只是为了体验，“can be circumvented by an attacker who disables JavaScript or uses a web proxy”（攻击者关掉 JS 或用代理工具就能绕过）4。因此校验必须在服务端、在数据被使用之前完成。前端校验拦不住坏人，只有后端校验能。

不校验会出什么事？给上面这段代码喂几个真实请求看看：

客户端发来的 body不校验的后果
{}（漏传 url）url 是 undefined，被当合法短链存进库，前端跳转时 404
{ "url": 12345 }数字混进字符串字段，后续 .startsWith() 之类操作直接抛错崩溃
{ "url": "随便一段文字" }不是合法网址，存下来后用户点击短链跳到一个无效地址
{ "url": "javascript:alert(1)" }潜在的 XSS 跳转载荷，可能伤害点击短链的访客

归纳成两个「为什么」：① 健壮性（Robustness）——脏数据进了系统，轻则脏数据落库、重则进程崩溃；② 安全（Security）——未经校验的输入是注入攻击（SQL 注入、XSS、SSRF）的头号入口。OWASP 把它列为应当贯穿整个应用的主动控制项之一4。

💡 术语：校验（validation）≠ 净化（sanitization）。 校验是判断数据合不合格、不合格就拒绝（本章重点）；净化是把数据洗干净再用（如转义 HTML）。OWASP 提醒：即使一个值「校验通过」，也未必绝对安全（一个合法 URL 里仍可能藏 XSS），所以校验之外还要配合转义/参数化查询等纵深防御4。本章先把第一道闸门——校验——建起来。

2zod 是什么，为什么用它

你当然可以手写一堆 if：if (!req.body.url) ...、if (typeof url !== 'string') ...、if (!/^https?:/.test(url)) ...。但字段一多，这种校验代码会迅速膨胀成难维护的「if 地狱」，而且很容易漏判。

zod 是一个 schema validation（模式校验）库。它的核心思想是：先用一个「schema（模式）」声明式地描述「合格的数据长什么样」，再拿这个 schema 去校验真实数据1。

💡 类比：schema 就像机场安检的「登机标准」清单——「必须有登机牌、证件、行李不超规」。你不需要对每个旅客手写检查逻辑，只要把人往这张「标准」上一套，符合就放行、不符合就拦下并告诉他哪里不对。zod 的 schema 就是这张可复用的标准清单。

先装上它（项目目前还是 JS，但 zod 在纯 JS 里照样工作）：

npm i zod

最小的例子——描述「一个有 username 字段、且必须是字符串的对象」1：

import { z } from 'zod'

const User = z.object({
  username: z.string(),
})

这里出现了三个 zod 的基础构件，先讲清楚再用：

写法含义
z.object({...})描述一个对象，花括号里列出每个字段及其规则。我们的请求体就是对象，所以最外层几乎总是它。
z.string()这个字段必须是字符串。是数字、布尔、缺失都不合格。
z.url()这个字段必须是合法 URL 格式的字符串。正是 LinkPulse 的 url 字段要的2。
🌍 版本提示（真实会踩的坑）：你可能在老教程里看到 z.string().url() 这种链式写法。从 zod v4 起，URL、email 等「字符串格式」校验被提升为顶层函数 z.url()、z.email()，老的链式 .url() 已被标记 deprecated（弃用），未来大版本会移除2。本课统一用新写法 z.url()，照着官方走，别被旧博客带偏。

还可以给规则加约束和自定义错误信息——这一点对「给前端清晰提示」很关键2： 

const LinkSchema = z.object({
  url: z.url({ message: 'url 必须是合法的网址' }),
})

3parse vs safeParse：两种校验姿势

有了 schema，怎么拿它去校验数据？zod 给了两个方法，区别只在「失败时怎么办」1：

parse —— 失败就 throw

// 成功：返回校验后的数据
LinkSchema.parse({ url: 'https://a.com' })
// => { url: 'https://a.com' }

// 失败：抛出 ZodError 异常
LinkSchema.parse({ url: 123 })
// => throw ZodError

safeParse —— 失败不抛，返回结果对象

const r = LinkSchema.safeParse(req.body)
if (!r.success) {
  r.error    // ← ZodError 实例
} else {
  r.data     // ← 校验通过的数据
}

safeParse 返回的是一个带 success 布尔字段的结果对象（官方称之为 discriminated union，可辨识联合）：成功时 { success: true, data }，失败时 { success: false, error }1。它不会抛异常，所以你能用普通的 if 流程优雅地处理失败。

💡 该用哪个？处理「来自外部、本就可能不合格」的数据（比如用户请求体）→ 用 safeParse，因为失败是预期之内的正常分支，不该靠 try/catch。处理「你高度确信格式正确」的内部数据 → 用 parse 更简洁。校验请求体，首选 safeParse。

4给 LinkPulse 的 POST /api/links 加校验

把上面的零件组装起来。当 safeParse 失败时，我们要返回 HTTP 400 Bad Request——这个状态码你在前端见过，语义就是「客户端发来的请求有问题」，正好对应「输入不合格」。

失败时把错误信息整理成对前端友好的结构。zod 提供 z.flattenError(error)，它把一堆错误压平成按字段分组的形状，特别适合表单类请求3： 

z.flattenError(result.error)
// => {
//   formErrors: [],
//   fieldErrors: { url: ['url 必须是合法的网址'] }
// }

改造后的路由（注意这是同步校验，没有任何 await）：

import { z } from 'zod'

const LinkSchema = z.object({
  url: z.url({ message: 'url 必须是合法的网址' }),
})

app.post('/api/links', (req, res) => {
  const result = LinkSchema.safeParse(req.body)

  if (!result.success) {
    // 校验失败 → 400 + 统一错误结构，让前端能逐字段显示
    return res.status(400).json({
      error: 'ValidationError',
      details: z.flattenError(result.error).fieldErrors,
    })
  }

  // 走到这里，result.data 一定是「干净」的数据
  const { url } = result.data
  const code = Math.random().toString(36).slice(2, 8)
  res.status(201).json({ code, url })
})
POST /api/links→ safeParse(req.body) 校验关卡 ✗ 不合格 → 400 + error ✓ 合格 → 201 + 短码

验证一下（确保你的服务器在跑，且上一课已 app.use(express.json())）：

# 缺 url → 应得 400
curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" -d '{}'

# 不是合法网址 → 应得 400
curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" -d '{"url":"哈哈"}'

# 合法 → 应得 201
curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" -d '{"url":"https://example.com/very/long/path"}'
🌍 统一错误响应格式（真实项目的约定）：注意我们失败时固定返回 { error, details } 结构。在真实团队里，「所有错误都长一个样」是一条重要约定——前端只需写一套错误处理逻辑（看到 error 字段就弹提示、看到 details 就标红对应输入框），不用为每个接口猜返回结构。约定一种错误信封（error envelope），是后端 API 设计的基本功。
🌍 zod 的「一鱼两吃」（埋个伏笔）：zod 最被推崇的一点是——同一份 schema 既能在运行时校验数据（本章用的），又能自动推导出 TypeScript 静态类型（用 z.infer<typeof LinkSchema>）1。也就是说写一次 schema，编译期和运行期两种安全一起拿到，再也不会出现「类型说是 string、运行时却来个 number」的撕裂。等 Lesson 0010 项目迁到 TypeScript 后，我们会回来兑现这个红利。这正是 zod 比手写 if 高一个维度的原因。

5错误处理中间件：兜住所有意外

校验挡住了「能预料的坏输入」。但程序里还有预料之外的错误：第三方服务超时、数据库断连、某个 undefined.foo 把代码炸了……总不能在每个路由里都写一大坨 try/catch。Express 为此提供了一种特殊中间件——错误处理中间件（error-handling middleware）。

上一课你知道普通中间件是 3 个参数 (req, res, next)。错误处理中间件的唯一区别是：它有 4 个参数 (err, req, res, next)。Express 就是靠「参数个数是 4」来识别它是错误处理器的——哪怕你用不到 next，也必须把 4 个参数都写全，否则 Express 会把它当成普通中间件5。 

// 放在所有路由 / app.use 的【最后面】
app.use((err, req, res, next) => {
  console.error(err)              // 记录到服务端日志，便于排查
  res.status(err.status || 500).json({
    error: err.name || 'InternalServerError',
    message: err.message,
  })
})

关于错误处理中间件，官方强调三条规则5，每条都要理解：

  • ① 必须 4 个参数 (err, req, res, next) —— 这是它和普通中间件的唯一区分标志。
  • ② 必须放在最后 —— 在所有其它 app.use() 和路由之后定义，这样它才能接住前面任何环节抛出的错误。（回忆 0006：404 兜底放最后，错误处理器放得比 404 还后。）
  • ③ 用 next(err) 触发它 —— 在普通中间件/路由里，只要给 next() 传一个参数，Express 就认为那是个错误，于是跳过所有剩余的普通中间件，直接进到错误处理中间件5。

同步错误 vs 异步错误：谁帮你兜，谁要你自己交

这是新手最容易栽跟头的地方，务必分清。

同步错误（synchronous）——Express 自动接住。 在路由里直接 throw 的错误（包括 zod 的 parse 抛的 ZodError），Express 会自动捕获并送进你的错误处理中间件，你什么都不用做5： 

app.get('/boom', (req, res) => {
  throw new Error('炸了')   // ✅ Express 自动接住，转给错误中间件
})

异步错误（asynchronous）——要看 Express 版本。 这里有个关键分水岭，而且正好和 LinkPulse 相关，因为我们项目装的是 express@^5（看 package.json）：

Express 4 的世界（老教程）

// ❌ 这个 reject 不会被自动接住
app.get('/u/:id', async (req, res) => {
  const u = await getUser(req.params.id)
  res.json(u)
})
// 必须手动 try/catch + next(err)：
app.get('/u/:id', async (req, res, next) => {
  try {
    res.json(await getUser(req.params.id))
  } catch (err) {
    next(err)   // 手动上交
  }
})

Express 5 的世界（我们用的）

// ✅ 返回的 Promise 若 reject，
// Express 5 会【自动】调用 next(err)
app.get('/u/:id', async (req, res) => {
  const u = await getUser(req.params.id)  // 抛错 → 自动转交
  res.json(u)
})
// 不再需要到处包 try/catch

官方原话：在 Express 5 中，“route handlers and middleware that return a Promise will call next(value) automatically when they reject or throw an error.”（返回 Promise 的处理器在 reject 或抛错时会自动调用 next）5。

🌍 真实会踩的坑：大量网上 Express 教程仍是 4.x 写法，到处教你装 express-async-handler 或手包 try/catch 来「补救」异步错误。我们的 LinkPulse 用的是 Express 5，这一痛点已被官方原生解决5，盲目照搬旧方案只会徒增噪音。但请记牢边界：自动接住的前提是「处理器返回了那个 Promise」。如果你在回调里(如老式 fs.readFile(cb))出错、或自己 setTimeout 里抛错——它没被 await/返回，Express 仍接不住，那时还得老老实实 next(err)。

最后补一个官方提到的细节：如果错误发生时响应头已经发出去了（res.headersSent 为真），你的错误处理器应当 return next(err) 把它交还给 Express 内置的默认错误处理器，由后者负责关闭连接——因为这时你已经没法再改状态码了5。

🧠 小检验 校验 + 错误处理

客户端往 POST /api/links 发了 { "url": 12345 }。你用 LinkSchema.safeParse(req.body) 校验。下面哪种做法最正确？

⌨️ 动手：给 LinkPulse 装上「安检关卡」

  1. 在项目根目录运行 npm i zod，确认 package.json 的 dependencies 里出现了 zod。
  2. 在 express-server.js 顶部 import { z } from 'zod'，并定义 LinkSchema = z.object({ url: z.url({ message: 'url 必须是合法的网址' }) })。
  3. 改写 POST /api/links：用 LinkSchema.safeParse(req.body)；失败返回 400 + { error:'ValidationError', details: z.flattenError(result.error).fieldErrors }；成功才用 result.data.url 生成短码。
  4. 在文件最后（404 兜底中间件之后）加一个 4 参数错误处理中间件 app.use((err, req, res, next) => {...})，统一返回 { error, message } 并打印 console.error(err)。
  5. 临时加一个 app.get('/boom', () => { throw new Error('测试错误') })，用浏览器或 curl 访问它，确认请求没有挂起、而是被错误中间件接住返回了 JSON。验证完删掉这行。
  6. 用本课第 4 小节的三条 curl 分别验证 400 / 400 / 201，确认错误响应里能看到针对 url 字段的清晰提示。
