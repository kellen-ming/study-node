Lesson 0006 · Node.js 后端之路

中间件，与处理 POST 请求体
（让你的 API 能「收数据」）

从「只读」到「能写」。 到目前为止，你的路由全是 GET——只往外读数据。 但 LinkPulse 要能创建短链：用户提交一个长网址，你存下来并返回短码。这就需要接收客户端发来的数据（请求体）。 而要把「解析数据」「记日志」这类活儿做得优雅可复用，靠的就是 Express 的灵魂——中间件。

1什么是中间件（Middleware）

官方定义：中间件就是能访问请求对象 req、响应对象 res、以及「下一个中间件」next 的函数1。

最好的心智模型是流水线 / 安检关卡：每个请求进来后，依次穿过一道道中间件，每一道都可以——检查/加工 req、提前响应、或者放行（调用 next()）交给下一道：

请求→ 日志中间件→ express.json()→ 路由处理→ 响应

一个最简单的自定义中间件（每个请求都先打一行日志）：

app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`)
  next()        // 放行：交给下一道中间件 / 路由
})

中间件能做四件事1：执行任意代码、修改 req/res、结束请求（发响应）、或调用 next() 放行。

⚠️ 最关键的一条规则：如果一个中间件既不结束请求（不发响应）、又不调用 next()，那么——官方原话——“the request will be left hanging”（请求会一直挂起，客户端永远转圈）1。这是新手最常见的「接口卡住不返回」的元凶。

💡 其实你早就用过中间件了：app.get('/x', (req, res) => {...}) 里的处理函数，本身就是「绑定到某个路径的中间件」。所谓「路由」和「中间件」在 Express 里是同一套东西。

2顺序很重要

中间件按你定义的先后顺序执行1。一旦某个中间件发了响应、又没调用 next()，后面的就不会再执行。所以摆放顺序有讲究：

  • 日志、解析 body 这类「预处理」→ 放在所有路由之前（请求得先被它们加工过）。
  • 404 兜底 → 放在所有路由之后（前面都没匹配上，才轮到它）。
🌍 原来如此：你在 0005 把 app.use(404) 放在最后，正是这个原理——它是「兜底中间件」，必须排在所有正常路由后面，否则会把正常请求也拦成 404。你当时凭直觉放对了，现在知道为什么了。

3express.json()：接收 POST 请求体

当客户端 POST 一段 JSON 数据时，它在底层是一串原始字节流，Express 默认不会替你解析。你需要一个内置中间件 express.json() 来解析它，解析结果会被放进 req.body2： 

app.use(express.json())   // 放在路由之前！解析 JSON 体 → 填进 req.body

加了它之后，POST 路由里就能直接读 req.body 了。至此，「从请求里取数据」的三兄弟你就集齐了：

三者别搞混

req.params  // 路径里：/links/:code
req.query   // ? 后面：?page=2
req.body    // 请求体里：POST 的 JSON
            // （需 express.json() 解析）

各自的来源

GET /links/abc?page=2
         └─┬─┘ └─┬──┘
        params  query

POST /api/links
{ "url": "..." }   ← body

4写一个 POST 路由：创建短链的雏形

现在可以写 LinkPulse 的第一个「写」接口了——接收一个长网址，返回一个（暂时假的）短码：

app.post('/api/links', (req, res) => {
  const { url } = req.body                      // 从请求体取数据
  if (!url) {
    return res.status(400).json({ error: 'url is required' })   // 校验：没给 url → 400
  }
  const code = Math.random().toString(36).slice(2, 8)   // 临时假短码，以后用数据库生成
  res.status(201).json({ code, url })           // 201 = 已创建
})
🌍 两个实战状态码：201 Created 表示「成功创建了一个新资源」（比笼统的 200 更准确）；400 Bad Request 表示「你传的数据有问题」。真实 API 会认真区分这些状态码——这是专业和业余的分水岭。

5把它们拼起来（注意顺序）

import express from 'express'
const app = express()

app.use((req, res, next) => {          // ① 日志（每个请求）
  console.log(`${req.method} ${req.url}`)
  next()
})
app.use(express.json())                 // ② 解析 JSON body

app.get('/api/health', (req, res) => {  // ③ 路由们
  res.json({ status: 'ok' })
})
app.post('/api/links', (req, res) => {
  const { url } = req.body
  if (!url) return res.status(400).json({ error: 'url is required' })
  const code = Math.random().toString(36).slice(2, 8)
  res.status(201).json({ code, url })
})

app.use((req, res) => {                 // ④ 404 兜底（最后）
  res.status(404).send('Not Found')
})

app.listen(3000, () => console.log('http://localhost:3000'))

🧠 小检验 中间件

一个中间件里，如果你既不发送响应、也不调用 next()，会发生什么？

⌨️ 动手：给 API 加上「记日志」和「收数据」（15 分钟）

  1. 在 express-server.js 里，所有路由之前加两个中间件：① 自定义日志中间件（打印 method + url，记得 next()）；② app.use(express.json())。
  2. 加一个 POST 路由 /api/links，照 §4 写（取 req.body.url、没传就 400、否则返回 201 + 假短码）。
  3. 重启服务器，用 curl 发一个带 JSON 体的 POST： 
    curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com"}'
    应看到 201 和返回的 { code, url }。同时看服务器终端，日志中间件应打印了这次请求。
  4. 故意踩个坑（很重要）：再发一次，但去掉 -H "Content-Type: application/json"。观察 req.body 变空、于是返回 400 —— 体会「express.json() 只解析声明了 JSON 类型的请求体」这个真实坑。
