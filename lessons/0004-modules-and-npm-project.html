Lesson 0004 · Node.js 后端之路

模块系统与你的
第一个正式项目（ESM · npm）

从「零散脚本」到「正式项目」。 到现在你的代码还是几个孤立的 .js 文件。 真实产品（包括 LinkPulse）都是一个由 package.json 管理的项目：能装依赖、能用一条命令启动、代码拆成多个模块互相复用。 这节课就把你的零散文件，升级成专业的项目骨架。

1模块：把代码拆成多个文件

一个应用不可能挤在一个文件里。模块（module）就是把功能拆到不同文件、各管一摊、再互相引用。 这点你在前端早就熟了（import/export）。Node 里有两套模块语法，你都会遇到：

CommonJS（Node 老传统，前几课用的）

// math.js —— 导出
function add(a, b) { return a + b }
module.exports = { add }

// 别处 —— 导入
const { add } = require('./math')

ESM（你前端用的那套，现代标准）

// math.js —— 导出
export function add(a, b) { return a + b }

// 别处 —— 导入
import { add } from './math.js'

👀 一眼对比：require / module.exports（CommonJS） vs import / export（ESM）。功能类似，但下一节你会看到，在 Node 里选哪套是有讲究的。

2在 Node 里到底用 ESM 还是 CommonJS？

在前端，模块的事是打包器（webpack/vite）替你搞定的。但在 Node 里没有打包器，是你自己来定——而且两套各有脾气：

  • ESM（import/export）：JavaScript 的官方标准，支持 tree-shaking、顶层 await；Node 官方文档现在都用它1。
  • CommonJS（require）：Node 的老传统，不用任何配置 node 文件 就能跑；生态里海量老包仍是它。

默认情况下，Node 把 .js 当 CommonJS。要让它用 ESM，在 package.json 里加一行1：

{
  "type": "module"
}

（或者把文件后缀改成 .mjs。）加了之后，.js 文件就能用 import/export 了。

另外，导入内置模块推荐写 node: 前缀，明确「这是 Node 内置的，不是 npm 包」：

import { createServer } from 'node:http'   // 推荐
import fs from 'node:fs'
🌍 实战：新项目用哪个？ 新项目直接上 ESM——它是未来，也和你 MISSION 里「最终用 TypeScript」一致（TS + ESM 是主流组合）。 你只会在维护老项目、或某些工具链里才碰到 CommonJS。
⚠️ 实战中真实会踩的坑（interop）：require() 无法加载纯 ESM 的包，会报 ERR_REQUIRE_ESM。 而现在越来越多新包（如新版本的一些库）只发 ESM。这是 CommonJS 项目升级时最常见的报错之一——你用 ESM 就基本绕开了它。

🔗 还记得你那个未声明的 body 吗？ ESM 默认就是严格模式（strict mode），那种「忘了声明变量」的写法在 ESM 里会直接报错——等于强制你写更规范的代码。这也是 ESM 更「专业」的一点。

3package.json：项目的身份证 + 说明书

前端你见过它。这里快速过一遍，重点看它在后端怎么用2：

{
  "name": "linkpulse",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "start": "node src/server.js",
    "dev": "node --watch src/server.js"
  },
  "dependencies": {},
  "devDependencies": {}
}
  • scripts：把常用命令起个名，用 npm run <名字> 执行2。后端最常见的就是启动服务器：开发用 npm run dev（带 --watch 自动重启），上线用 npm start。
  • dependencies：运行时就需要的包（如 express、prisma）。npm install express 会装到这里。
  • devDependencies：只在开发/构建时用的包（如 typescript、eslint）。npm install -D eslint 会装到这里。
🌍 实战：为什么要分 deps / devDeps？ 上线部署时可以只装 dependencies（npm install --omit=dev），让生产环境更小更快——你的测试框架、lint 工具没必要打进生产镜像。

4一个真实 Node 项目长什么样

这是后端项目的惯用结构（LinkPulse 以后也会长这样）：

linkpulse/ ├── package.json # 项目身份证 + npm 命令 ├── node_modules/ # 依赖（npm 装的，不提交到 git） ├── .gitignore # 告诉 git 忽略哪些（首先就是 node_modules） ├── src/ # 源码都放这 │ ├── server.js # 入口 │ └── health.js # 拆出来的一个模块 └── README.md
🌍 实战必做：.gitignore 一定要忽略 node_modules/。 它动辄几百 MB、能用 npm install 随时重建，绝不该提交到仓库。这是每个 Node 项目的第一条铁律。

🧠 小检验 ESM

要让 Node 把 .js 文件当作 ESM（能用 import/export）来运行，下面哪种做法对？

⌨️ 动手：把零散文件升级成正式 ESM 项目（15 分钟）

  1. 在仓库根目录运行 npm init -y → 生成 package.json（-y 表示全用默认值）。
  2. 打开 package.json，加上 "type": "module"，并把 scripts 改成： 
    "scripts": {
  "dev": "node --watch 01-node-core/first-server.js",
  "start": "node 01-node-core/first-server.js"
}
  3. 把 first-server.js 顶部的 const { createServer } = require('http') 改成 ESM 写法：import { createServer } from 'node:http'。
  4. 练一下拆模块：新建 01-node-core/health.js： 
    export function healthPayload() {
  return { status: 'ok', uptime: process.uptime() }
}
    然后在 first-server.js 里导入它（⚠️ ESM 相对导入必须带 .js 后缀，这点和前端打包器不同，很多人踩）： 
    import { healthPayload } from './health.js'
// ...把 /api/health 那行改成：
res.end(JSON.stringify(healthPayload()))
  5. 新建 .gitignore，写入一行 node_modules/。
  6. 用 npm run dev 启动（带 --watch，以后改代码自动重启，告别手动重启）。
  7. 浏览器/curl 访问 /api/health，确认仍返回那段 JSON —— 说明你的模块拆分成功了。✅
🌍 实战补充：npm run dev 这套「热重启」工作流是日常开发标配。Node 18+ 内置的 --watch 够用了； 你可能还会听到 nodemon（第三方老牌方案，功能更多）。两者都是为了「改完即生效、不用手动重启」。
📖 本课首选精读源： Node.js 官方 — ECMAScript modules（ESM 的完整规则）； 项目配置见 npm 官方 — package.json。
