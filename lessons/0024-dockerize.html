Lesson 0024 · Node.js 后端之路

用 Docker 容器化 LinkPulse

为什么学这个？ 到现在为止，LinkPulse 在你本机能跑：Node 进程 + 本地装的 PostgreSQL + 本地装的 Redis。可一旦换台机器、换个同事、或者推到云服务器，就开始「我这跑得好好的啊」。本章把 LinkPulse 容器化：写一个 Dockerfile 把 app 打包成可移植镜像，再用 docker compose 一条命令同时拉起 app + postgres + redis 整套全栈。这是下一课「CI/CD 与部署」的地基——能部署上线收费，第一步就是「能被打包成镜像」。

1为什么要容器化：消灭「在我机器上是好的」

你现在的运行环境其实是一堆隐式依赖：本机的 Node 版本、本机装的 Postgres 版本、Redis 是不是开着、.env 里那串连接串、甚至某个全局 npm 包。这些东西没有一个写在代码仓库里。换台机器，任何一项对不上，应用就崩——这就是经典的「在我机器上是好的」（works on my machine）。

容器（container）就是为了解决这个问题。它把「你的应用 + 运行它所需的一切（Node 运行时、依赖、系统库、环境配置）」打包成一个标准化的、隔离的运行单元。在哪台装了 Docker 的机器上跑，行为都一致。

前端类比（快速复习）：你熟悉 npm pack / 发布到 npm 的「锁版本」思路，或者 vite build 产出确定的 dist/。容器是把这个思路扩大到整个操作系统层级：不只锁住 JS 依赖，连 Node 版本、OS 库都一起锁进一个不可变的「快照」里。

容器和虚拟机（VM）不同：VM 要模拟一整套硬件 + 跑一个完整 Guest OS，几个 GB、启动几十秒；容器共享宿主机的 Linux 内核，只打包应用层，几十到几百 MB、启动以秒计1。这就是为什么容器能做到「一台机器塞几十个、秒级启停」，非常适合做后端服务和弹性扩缩。

🌍 实战：团队协作里，容器化最直接的收益是「新人入职第二天就能跑起整个项目」。没有容器时，新同事要照着一份过时的 README 手装 Postgres、Redis、配环境变量，常常折腾半天。有了 docker compose up，一条命令把数据库、缓存、应用全拉起来，环境和你完全一致。这也是为什么几乎所有现代后端项目仓库根目录都有 Dockerfile 和 compose.yaml。

2镜像 vs 容器：模板与实例

这两个概念初学最容易混。一句话区分：

  • 镜像（image）：只读的模板/打包产物。它是一层一层叠起来的文件系统快照（基础 OS 层 + Node 层 + 你的依赖层 + 你的代码层），加上启动命令等元数据。镜像是静态的，可以推到镜像仓库（registry）分发。
  • 容器（container）：镜像跑起来后的运行实例。它是动态的——有自己的进程、可写的临时层、网络。一个镜像可以同时跑出多个容器。
类比：镜像 = 你写的 class（蓝图）；容器 = new 出来的实例。或者：镜像 = 一个安装好系统的 U 盘镜像文件；容器 = 用它启动出来的那台正在运行的「机器」。

镜像由 层（layer）叠成，每条 Dockerfile 指令通常产生一层。层会被缓存：如果某一层和它之前的层都没变，Docker 直接复用缓存，不重新构建。这就是后面「指令顺序很重要」的原因。

layer 0 · 基础镜像 node:22-alpine（OS + Node 运行时）
layer 1 · COPY package*.json + npm ci（依赖，很少变 → 缓存命中率高）
layer 2 · COPY 源码 + 编译产物（经常变 → 放最后）

3写第一版 Dockerfile：把 LinkPulse 打包

Dockerfile 是一份「怎么构建镜像」的脚本，放在项目根目录。我们先写一个朴素单阶段版本理解每条指令，再在第 4 节升级成多阶段。LinkPulse 是 TypeScript + ESM 项目，npm run build 把 src/ 编译进 dist/，入口是 dist/index.js。

# ── 朴素单阶段版（仅用于讲解，生产请看第 4 节多阶段版）──
FROM node:22-alpine

# 容器内的工作目录，后续相对路径都基于它
WORKDIR /app

# 先只拷依赖清单，利用层缓存：源码改了也不会触发重装依赖
COPY package.json package-lock.json ./

# npm ci 严格按 lock 文件安装，可复现，比 npm install 更适合构建
RUN npm ci

# 再拷其余源码（含 prisma/、tsconfig.json、src/）
COPY . .

# 生成 Prisma Client，再把 TS 编译成 dist/
RUN npx prisma generate
RUN npm run build

# 文档化：应用监听 3000（EXPOSE 只是声明，不真正发布端口）
EXPOSE 3000

# 容器启动时执行的命令
CMD ["node", "dist/index.js"]

逐条理解几个后端新概念：

  • FROM node:22-alpine：基础镜像。node:22 是官方 Node 镜像，-alpine 表示基于极小的 Alpine Linux（几 MB），相比默认的 Debian 版镜像小很多2。
  • WORKDIR /app：设定容器里的当前目录，并自动创建它。后面的 COPY . . 第二个 . 就是指 /app。
  • 先 COPY package*.json 再 npm ci，最后才 COPY . .：这是层缓存的关键。Docker 官方建议「把最不常变的指令放前面」2。依赖清单很少变，源码天天改；这样排，只改源码时 npm ci 那层直接命中缓存，构建快得多。
  • npm ci 而非 npm install：ci 严格依据 package-lock.json 安装、构建可复现，是 CI/容器构建的标准做法。
  • CMD 用数组（exec form）：["node","dist/index.js"] 不经过 shell，进程是 PID 1，能正确收到 SIGTERM 优雅退出——这点在 K8s/容器编排下很重要。
为什么要 npx prisma generate？ Prisma Client 是「按你的 schema 生成的代码」，不在 node_modules 里现成存在，必须在构建时生成一次，否则容器里 import { PrismaClient } 会报错。注意这只是生成 Client，不是连数据库；prisma migrate 要等容器运行、能连上 DB 时再做。

4多阶段构建 + .dockerignore：把镜像瘦下来

上面那版有个问题：最终镜像里塞着全部 devDependencies（TypeScript 编译器、类型包、测试框架……）和 TS 源码，而运行时只需要 dist/ + 生产依赖。镜像越大，拉取越慢、攻击面越大。

多阶段构建（multi-stage build）是官方推荐的解法2：在同一个 Dockerfile 里写多个 FROM，每个 FROM 是一个独立阶段。build 阶段装全部依赖、编译 TS；最终阶段从一个干净的小镜像起步，只用 COPY --from=... 把「编译产物 + 生产依赖」捞过来。中间阶段的临时文件全部丢弃，不进最终镜像。

① builder 阶段

装全部依赖（含 dev）→ prisma generate → npm run build 编出 dist/。这一层很大，但不会进最终镜像。

② runner 阶段

干净小镜像，只拷 dist/ + 生产依赖 + Prisma 产物。最终镜像可从 1GB+ 降到 ~150MB。

下面是给 LinkPulse 的生产级 Dockerfile，请用它替换第 3 节那版：

# ============ Stage 1: builder（编译 TS） ============
FROM node:22-alpine AS builder
WORKDIR /app

# 缓存友好：先装依赖
COPY package.json package-lock.json ./
RUN npm ci

# 拷源码并构建
COPY prisma ./prisma
RUN npx prisma generate
COPY tsconfig.json ./
COPY src ./src
RUN npm run build      # 产出 dist/

# 把 node_modules 收敛到只剩生产依赖，供下一阶段拷贝
RUN npm prune --omit=dev

# ============ Stage 2: runner（只留运行所需） ============
FROM node:22-alpine AS runner
WORKDIR /app

# 生产环境标志：让依赖/框架走生产路径
ENV NODE_ENV=production

# 官方 node 镜像自带一个非 root 的 node 用户，直接用它
# --chown 保证文件归 node 所有，避免权限问题
COPY --from=builder --chown=node:node /app/node_modules ./node_modules
COPY --from=builder --chown=node:node /app/dist ./dist
COPY --from=builder --chown=node:node /app/package.json ./package.json
# Prisma 运行时需要 schema 与生成的 client（client 在 node_modules 里已带）
COPY --from=builder --chown=node:node /app/prisma ./prisma

# 切换到非 root 用户运行
USER node

EXPOSE 3000
CMD ["node", "dist/index.js"]

配套的 .dockerignore（放项目根目录）告诉 Docker「构建时别把这些发给守护进程、别拷进镜像」。官方明确建议每个项目都加一个，能让构建更快、镜像更小2：

# .dockerignore
node_modules
dist
npm-debug.log*
.git
.gitignore
.env
.env.*
coverage
*.md
Dockerfile
.dockerignore
🌍 实战：node_modules 是 .dockerignore 里必排第一名。原因有两层：一是它体积巨大、白白拖慢构建上下文上传；二是你本机的 node_modules 可能是为 macOS/Windows 编译的二进制（如某些原生模块），拷进 Linux 容器会直接跑不起来——必须让容器内 npm ci 重新装。同理 dist 也排掉，强制容器内重新编译，保证产物干净。
🌍 实战（安全）：两条生产铁律。①密钥绝不打进镜像：不要 COPY .env、不要在 Dockerfile 里写 ENV DATABASE_URL=postgres://...密码。镜像是会被分发、被人 docker history 翻层的，密钥进了镜像等于泄露。正确做法是运行时通过环境变量注入（compose 的 environment / 部署平台的 secret）。②用非 root 用户跑：官方 node 镜像自带 node 用户，加一句 USER node，万一应用被攻破，攻击者拿到的也只是低权限用户而非容器 root2。

5用 docker compose 一键起整套全栈

现在 app 能打成镜像了，但 LinkPulse 还要 PostgreSQL 和 Redis 才能跑。手动 docker run 三次、再手配网络很麻烦。Docker Compose 让你用一个 compose.yaml 声明式地描述「这套应用由哪几个服务组成、怎么连」，然后 docker compose up 一键拉起全部3。

几个 Compose 核心概念（后端新概念，讲透）：

  • service（服务）：一个要跑的容器（或一组副本），比如 app、postgres、redis。
  • 自动网络 + 服务名 DNS：Compose 会建一个内部网络，服务之间可以用服务名当主机名互相访问。所以 app 连数据库的地址不是 localhost，而是 postgres（服务名）。这是新人最常踩的坑。
  • depends_on + condition: service_healthy：声明启动顺序。但注意——depends_on 默认只等容器「启动了」，不等里面的服务「就绪」。Postgres 容器起来 ≠ 数据库能接受连接。要真正等就绪，得给被依赖服务配 healthcheck，再用 condition: service_healthy 等它健康4。
  • volumes（卷）：容器的可写层在删除时会消失。数据库数据要用命名卷持久化，否则 compose down 一次数据就没了。
# compose.yaml （放项目根目录）
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: linkpulse
      POSTGRES_PASSWORD: secret      # 本地开发用；生产请用 secret 注入
      POSTGRES_DB: linkpulse
    volumes:
      - pgdata:/var/lib/postgresql/data   # 持久化数据
    healthcheck:
      # pg_isready 探测数据库是否能接受连接
      test: ["CMD-SHELL", "pg_isready -U linkpulse -d linkpulse"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]   # 返回 PONG 即健康
      interval: 10s
      timeout: 5s
      retries: 5

  app:
    build: .                # 用当前目录的 Dockerfile 构建镜像
    ports:
      - "3000:3000"         # 宿主机 3000 → 容器 3000
    environment:
      NODE_ENV: production
      # 注意主机名用服务名 postgres / redis，不是 localhost
      DATABASE_URL: postgres://linkpulse:secret@postgres:5432/linkpulse
      REDIS_URL: redis://redis:6379
      PORT: "3000"
    depends_on:
      postgres:
        condition: service_healthy   # 等 DB 真正就绪再启动 app
      redis:
        condition: service_healthy

volumes:
  pgdata:                   # 命名卷，数据在 compose down 后仍保留
迁移怎么跑？ 容器内不要在构建期跑 prisma migrate（那时连不上 DB）。两种常见做法：①临时手动 docker compose run --rm app npx prisma migrate deploy；②把 app 的启动命令改成「先迁移再启动」，例如在 compose 里给 app 加 command: sh -c "npx prisma migrate deploy && node dist/index.js"。本地学习用第①种最直观。
🌍 实战：compose 最适合的场景是「本地一键起全栈依赖」和「集成测试环境」。生产环境通常交给 K8s 或云平台的容器服务编排，而不是直接 compose up。但 compose.yaml 仍然极有价值：它是项目「需要哪些依赖、怎么连」的可执行文档，新人 clone 下来 docker compose up 就有完整环境，比任何 README 都靠谱。另外注意：旧教程里的 docker-compose（带横杠、独立 Python 工具）已被官方淘汰，现在统一用 docker compose（作为 Docker CLI 插件）3。

6跑起来 & 验证

在 LinkPulse 项目根目录依次执行：

# 构建并后台启动整套（app + postgres + redis）
docker compose up --build -d

# 看三个服务都起来了，postgres/redis 应显示 (healthy)
docker compose ps

# 首次需要建表：对着容器里的 app 跑迁移
docker compose run --rm app npx prisma migrate deploy

# 跟一下 app 日志，确认监听 3000
docker compose logs -f app

然后验证业务链路（你在 Lesson 0001–0016 做好的功能此刻应整套可用）：

# 创建一条短链
curl -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" \
  -d '{"targetUrl":"https://docs.docker.com"}'

# 用返回的 shortCode 测 302 跳转（-I 只看响应头）
curl -I http://localhost:3000/<shortCode>
# 期望：HTTP/1.1 302 Found + Location: https://docs.docker.com
# 第二次请求应命中 Redis 缓存（看 app 日志里的 cache hit）

# 看完收工：停止并删除容器（pgdata 卷里的数据仍保留）
docker compose down

如果 302 正常、二次请求命中缓存，说明 app 镜像 ↔ postgres 服务 ↔ redis 服务三者已经通过 compose 网络和环境变量正确串起来了——LinkPulse 完成了容器化。

🧠 小检验 Compose 网络

在第 5 节的 compose.yaml 里，app 服务的 DATABASE_URL 主机名为什么写 postgres 而不是 localhost？

⌨️ 动手：把 LinkPulse 完整容器化并 compose up 跑通

  1. 在项目根目录写多阶段 Dockerfile（第 4 节版本）：builder 阶段 npm ci + prisma generate + npm run build + npm prune --omit=dev；runner 阶段从 node:22-alpine 起，COPY --from=builder --chown=node:node 捞 node_modules/dist/package.json/prisma，设 USER node。
  2. 写 .dockerignore，确认 node_modules、dist、.env 都被排除。
  3. 写 compose.yaml 编排 app + postgres(16-alpine) + redis(7-alpine)；给 pg/redis 配 healthcheck，app 用 depends_on ... condition: service_healthy；DB 数据挂命名卷 pgdata。
  4. docker compose up --build -d，再 docker compose run --rm app npx prisma migrate deploy 建表。
  5. 用 curl 创建短链、测 302 跳转、确认二次请求命中 Redis 缓存。
  6. 额外加分：跑 docker images 对比「单阶段 vs 多阶段」镜像体积，记下差了多少 MB。
