SQL 基础：建表与增删改查
（让 LinkPulse 真正「记住」短链）

从「假短码」到「真数据」。 Lesson 0006 里你的 POST /api/links 返回的是一个假短码——服务一重启就全忘了。 上一课（0011）你装好了 PostgreSQL 并连上了它，但库里还空空如也。 这一课你将学会用 SQL 亲手建表（CREATE TABLE）、插入 / 查询 / 更新 / 删除（INSERT / SELECT / UPDATE / DELETE），再加上 约束 和 索引。 目标很具体：建一张 links 表（含唯一短码 code），在里面把增删改查跑一遍——为下一课（0013 正式数据建模）热身。 本课命令全是 SQL，直接敲在数据库里；你的 TypeScript 项目下一课才会去调用它们。

1SQL 是什么：一种「声明式」语言

SQL（Structured Query Language，结构化查询语言）是和关系型数据库（如 PostgreSQL）对话的标准语言。 你用它来定义表、写入数据、查询数据。它是一门正式的、由 ISO 标准化的语言1，PostgreSQL 在标准之上做了不少扩展（后面会遇到几个）。

对前端工程师最关键的一个认知差异是：SQL 是「声明式（declarative）」的，而你熟悉的 JavaScript 大多是「命令式（imperative）」的。

命令式（你写 JS 的习惯）

// 一步步告诉机器「怎么做」
const found = [];
for (const link of allLinks) {
  if (link.code === 'abc123') {
    found.push(link);
  }
}

声明式（SQL 的方式）

-- 只描述「我要什么」，
-- 怎么找由数据库自己决定
SELECT * FROM links
WHERE code = 'abc123';

在 SQL 里，你不写循环、不写「先扫第一行再扫第二行」。你只声明「我想要 code 等于 abc123 的那些行」， 数据库的查询规划器（query planner）会自己决定用什么策略去取——是逐行扫描，还是走索引（本章第 5 节）。这正是声明式的威力：你管「要什么」，引擎管「怎么高效拿到」。

约定：SQL 关键字（SELECT、FROM…）大小写不敏感，写成大写只是社区惯例，方便一眼区分关键字和你自己的表名/列名。每条语句以 分号 ; 结尾。-- 开头是单行注释。

2DDL：用 CREATE TABLE 建表

SQL 语句按用途分两大类，先记住这个分类，后面不会乱：

  • DDL（Data Definition Language，数据定义语言）：定义「数据长什么样」——建表、改表、删表。代表：CREATE TABLE / ALTER TABLE / DROP TABLE。
  • DML（Data Manipulation Language，数据操作语言）：操作「表里的数据」——增删改查。代表：INSERT / SELECT / UPDATE / DELETE（第 3~4 节）。

一张表（table）就像一个有严格表头的电子表格：每一列（column）有固定的名字和数据类型；每一行（row）是一条记录2。建表就是先把「表头」和「每列的类型」定义清楚。

先认识几个常用数据类型

PostgreSQL 类型很多，但 LinkPulse 起步只需要这几个3：

🌍 实战：用 timestamptz 存时间，别用 timestamp。 timestamptz 是 timestamp with time zone 的简写3。它在存入时会把时间归一化为 UTC 保存，读取时再按当前会话时区呈现。 而裸 timestamp（不带时区）只是「墙上的钟点」，不知道是哪个时区的，跨时区一定出错。 LinkPulse 将来要做访问分析，用户可能在全球任何地方点链接——所有时间列一律用 timestamptz，这是后端老手的肌肉记忆。

自增主键：identity 还是 serial？

每张表通常需要一个能唯一标识每行的「身份证号」，最常见的做法是一个自动递增的整数——插入第 1 行它是 1，第 2 行自动变 2，你不用手动管。 PostgreSQL 提供两种写法4：

✅ identity（SQL 标准，推荐新项目）

id bigint
   GENERATED ALWAYS AS IDENTITY
   PRIMARY KEY

serial（PostgreSQL 老写法，仍很常见）

id bigserial PRIMARY KEY
-- 等价于：创建一个序列 +
-- DEFAULT nextval(序列)

官方文档说明：serial / bigserial 并不是真正的类型，只是「创建自增列」的一种书写便利——背后会自动建一个序列（sequence）对象并设默认值4。 而 GENERATED ALWAYS AS IDENTITY 是 SQL 标准的等价能力，且 identity 列会被自动标记为 NOT NULL5。 本课用 identity——它更标准、行为更清晰，PostgreSQL 官方也将其列为推荐方向。

动手：创建 links 表

把上面的知识拼起来，这就是 LinkPulse 的第一张表（先不加约束，下一节逐个加上）：

CREATE TABLE links (
  id          bigint GENERATED ALWAYS AS IDENTITY,
  code        text,
  long_url    text,
  created_at  timestamptz
);

执行后你就有了一张空表，结构如下（还没有任何行）：

links 表的结构（4 列，0 行）

3DML（上）：INSERT 写入与 SELECT 查询

INSERT：往表里塞数据

官方推荐写法是显式列出列名，这样就不依赖列的物理顺序，将来加列也不会错位6：

INSERT INTO links (code, long_url, created_at)
VALUES ('abc123', 'https://example.com/very/long/path', now());

注意我们没写 id——因为它是 identity 列，数据库会自动填 1、2、3…。 now() 是 PostgreSQL 内置函数，返回当前的 timestamptz。一次插多行也可以，用逗号隔开多组 VALUES6： 

INSERT INTO links (code, long_url, created_at) VALUES
  ('gh-repo', 'https://github.com/linkpulse',        now()),
  ('docs01',  'https://www.postgresql.org/docs/',     now()),
  ('blog42',  'https://example.com/blog/2026/launch', now());

SELECT：把数据读出来

这是你今后用得最多的语句。最基本的形态是「选哪些列、从哪张表」7：

SELECT * FROM links;              -- * 表示所有列
SELECT code, long_url FROM links; -- 只取这两列

真正有用的是三个修饰子句，注意它们的书写顺序是固定的：WHERE → ORDER BY → LIMIT。

  • WHERE 过滤行：只保留满足条件的行。条件里 判断相等用单个 =（不是 JS 的 ===），字符串用单引号。
  • ORDER BY 排序：ASC 升序（默认）、DESC 降序。
  • LIMIT 限制条数：只要前 N 行（做分页、取最新几条时必备）。
-- 按 code 精确查找（这正是 LinkPulse 重定向时要做的事）
SELECT id, long_url FROM links
WHERE code = 'abc123';

-- 最近创建的 3 条短链：按时间倒序，只取前 3
SELECT code, created_at FROM links
ORDER BY created_at DESC
LIMIT 3;

其中 WHERE code = 'abc123' 就是 LinkPulse 的核心动作：用户访问 短域名/abc123 时，后端拿 abc123 去查出对应的 long_url，再 301/302 跳过去。下一课你会在 TypeScript 里调用它。

🌍 实战：SQL 注入风险——现在就要拉响警报。 你也许想这样在 TypeScript 里「拼」出这条查询：
const sql = `SELECT * FROM links WHERE code = '${userInput}'`
千万别这么做。 如果 userInput 是 ' OR '1'='1，拼出来就变成 ... WHERE code = '' OR '1'='1'——条件恒真，整张表被拖走；更恶意的输入甚至能删库。这就是经典的 SQL 注入（SQL Injection），常年位列 Web 安全风险榜首。 本章你手敲固定 SQL 没问题，但一旦把用户输入拼进 SQL 字符串就危险。正确解法是参数化查询（占位符 + 参数分离）或用 ORM——这正是下一阶段（ORM/Prisma）要教的，先把这根弦绷上。

4DML（下）：UPDATE 改与 DELETE 删

四个增删改查动作，合称 CRUD（Create / Read / Update / Delete），是几乎所有后端 API 的底层骨架：

INSERTCreate 增 SELECTRead 查 UPDATEUpdate 改 DELETEDelete 删

UPDATE：修改已有行

用 SET 指定改哪些列、改成什么；用 WHERE 圈定改哪些行：

UPDATE links
SET long_url = 'https://example.com/new-destination'
WHERE code = 'abc123';
⚠️ 最经典、最致命的失误：UPDATE / DELETE 忘了写 WHERE。 UPDATE links SET long_url = '...'（没有 WHERE）会把每一行都改成同一个值！ 动手前先把同样的条件用 SELECT 跑一遍，确认命中的正是你想改的那些行，再换成 UPDATE/DELETE。这是从血泪里总结出的安全习惯。

DELETE：删除行

DELETE FROM links
WHERE code = 'blog42';   -- 只删这一行

DELETE FROM links;（不带 WHERE）会清空整张表——再次提醒，WHERE 是你的安全带。 注意 DELETE 删的是「表里的行」；要删整张表（结构都不要了）才用 DDL 的 DROP TABLE links;，别搞混。

5约束（Constraints）：让数据库替你守规矩

约束是你在建表时立下的规则，写错/违规的数据会被数据库当场拒绝。 这是关系型数据库相比「自己在代码里 if 判断」的巨大优势：规则在数据库里集中、强制、永远生效，哪怕将来有别的程序、别的同事直接写库，也绕不过去。下面四个是入门必备8。

四大常用约束一览
约束含义违反时
NOT NULL这列不能为空（不能没有值）插入/更新报错
UNIQUE这列的值在全表不能重复插入重复值报错
PRIMARY KEY主键 = UNIQUE + NOT NULL，每张表至多一个，用作行的唯一身份重复或为空都报错
REFERENCES
（外键）		这列的值必须在另一张表里真实存在指向不存在的行时报错

NOT NULL：必须有值

官方建议：在多数表设计里，大部分列都应该是 NOT NULL8——一个连目标网址都没有的短链毫无意义，所以 long_url 必须有值。

UNIQUE：全表不重复

LinkPulse 的命脉就在这里：短码 code 绝不能重复——否则 abc123 同时指向两个网址，重定向就乱套了。UNIQUE 约束帮你把这条铁律交给数据库强制执行。

🌍 实战：用 UNIQUE 约束防短码碰撞，而不是靠应用层「先查再插」。 新手常这样生成短码：先 SELECT 看 code 在不在，不在就 INSERT。但在并发下（两个请求同一瞬间都查到「不存在」），会双双插入同一个 code——这叫竞态条件（race condition）。 正确姿势：给 code 加 UNIQUE 约束，直接 INSERT；万一撞了，数据库会抛唯一性冲突错误，你的代码捕获它、换个码重试即可。让数据库做最终裁判，比应用层「先查再插」可靠得多。顺带一提：UNIQUE 会自动建一个 B-tree 索引8，所以按 code 查询也顺带变快了。
一个易踩的冷知识：默认情况下，UNIQUE 约束认为两个 NULL 不相等，所以多行 code 同时为 NULL 是被允许的8。但我们给 code 同时加了 NOT NULL，根本不允许 NULL，这个坑就自动避开了。

PRIMARY KEY：行的唯一身份

官方明确：PRIMARY KEY 在语义上等价于 UNIQUE + NOT NULL，并且会自动建唯一索引、自动把列标为 NOT NULL；一张表至多只能有一个主键8。 它的特殊地位在于：当别的表用外键引用本表时，默认就指向主键。我们的 id 列正适合当主键。

把约束加进 links 表

现在重建一张「带约束」的 links 表（若已存在旧表，先 DROP TABLE links;）：

CREATE TABLE links (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  code        text NOT NULL UNIQUE,            -- 短码：非空 + 全表唯一
  long_url    text NOT NULL,                   -- 目标网址：必须有
  created_at  timestamptz NOT NULL DEFAULT now() -- 没传就用当前时间
);

这里还顺手用了 DEFAULT now()：插入时若不提供 created_at，自动填入当前时间——这样应用代码连时间都不用管了。 约束生效后，下面这条会被数据库当场拒绝（因为 abc123 已存在）： 

INSERT INTO links (code, long_url) VALUES ('abc123', 'https://dup.example');
-- ERROR: duplicate key value violates unique constraint "links_code_key"

外键 REFERENCES：跨表的「必须真实存在」

外键（foreign key）让一列的值必须在另一张表里真实存在，从而把两张表「关联」起来9。 这是 LinkPulse 下一课（数据建模）的重头戏，这里先建立直觉。设想未来要记录每次点击： 

CREATE TABLE clicks (
  id        bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  link_id   bigint NOT NULL REFERENCES links (id), -- 必须指向某条真实存在的 link
  clicked_at timestamptz NOT NULL DEFAULT now()
);

link_id ... REFERENCES links (id) 表示：每条点击记录的 link_id，必须等于 links 表里某一行的 id。 如果你想插入一条 link_id = 999 但 links 里没有 id=999，数据库会拒绝——这保证了数据的引用完整性（referential integrity）：不会出现「点击记录指向一个根本不存在的短链」这种脏数据。 （外键被引用列必须是主键或唯一约束9，所以 links.id 作为主键正好满足。）

6索引（Index）初步：让查询飞起来

随着 LinkPulse 上线，links 表可能有几百万行。每次重定向都要 WHERE code = '...' 找一行——如果数据库逐行扫描（sequential scan，从第 1 行翻到第 N 行）整张表，会越来越慢。索引就是为此而生。

官方用了一个绝妙的类比：索引就像书末的术语索引10——你想查某个词，不必从头读到尾整本书，翻到索引页就能直接跳到对应页码。 数据库索引同理：没有索引，系统得逐行扫全表找匹配；有了索引，它「只需在搜索树里往下走几层」就能定位到目标行10。

建索引的语法很简单10。给最常用作查询条件的 code 建一个：

CREATE INDEX links_code_idx ON links (code);
其实你已经有了。 上一节给 code 加 UNIQUE 时，PostgreSQL 已自动为它建了一个唯一索引8，主键 id 也是。所以按 code 或 id 查已经是「走索引」的快路。 上面这条 CREATE INDEX 是演示给「非唯一但常用来过滤」的列加索引时怎么写——比如将来按 clicks.link_id 统计点击数。
🌍 实战：索引加速读，但拖慢写——别给每列都加。 索引不是免费的。官方明确指出：索引创建后，系统必须始终与表保持同步，这会给数据修改操作（INSERT/UPDATE/DELETE）带来额外开销；很少或从不被查询用到的索引应当删除10。 道理是：每次写入，所有相关索引都要跟着更新。所以经验法则是——给「经常出现在 WHERE / ORDER BY / 外键」的列加索引；对写多读少、几乎不拿来查的列别乱加。LinkPulse 的 code（重定向高频查）值得加；而像点击日志这种「写爆但很少单条查」的表，索引要更克制。

⌨️ 动手：在 Postgres 里把 links 表的 CRUD 跑一遍

  1. 用 psql 连进上一课的数据库（如 psql linkpulse）。
  2. 建表：执行第 5 节那条带约束的 CREATE TABLE links (...)（含 id 主键、code 的 NOT NULL UNIQUE、long_url 的 NOT NULL、created_at 的 DEFAULT now()）。用 \d links 确认结构。
  3. 增：用一条多行 INSERT 插入至少 3 条短链（显式写列名，不写 id 和 created_at，让它们自动生成）。
  4. 查：① SELECT * FROM links; 看全部；② 用 WHERE code = '...' 精确查一条；③ 用 ORDER BY created_at DESC LIMIT 2; 取最近 2 条。
  5. 验证 UNIQUE：故意再插一条已存在的 code，确认数据库抛出 duplicate key ... unique constraint 错误。
  6. 改：用 UPDATE ... SET long_url = '...' WHERE code = '...' 改掉某条的目标网址；改前先用同样的 WHERE 跑一次 SELECT 确认命中行。
  7. 删：用 DELETE FROM links WHERE code = '...' 删掉一条，再 SELECT 确认它没了。
  8. （进阶）执行 CREATE INDEX links_code_idx ON links (code);——观察它会不会因为 UNIQUE 已建索引而提示已存在/重复，思考为什么。
