Lesson 0009 · Node.js 后端之路

配置管理（.env）与结构化日志（pino）

为什么学这个？ 到目前为止，LinkPulse 的端口 3000 是写死在代码里的，日志全靠 console.log 随手打。这在玩具阶段没问题，但要上线收费就不行了：上线后端口/域名/数据库密码会变、不同环境（你本机 vs 服务器）配置不同、出故障时你得能在成千上万条日志里按字段检索而不是肉眼翻字符串。本章把两件「业余 → 专业」的事一次做对：① 把配置从代码里抽出来放进 .env（并保证密钥永不进 git）；② 用 pino 输出结构化 JSON 日志，替换掉之前写的 console.log 日志中间件。这两件事是任何能上线的后端服务的「成年礼」。

1为什么不能把配置写死在代码里

先看我们现在 LinkPulse 里的写法（来自 Lesson 0005/0008）：

// 现状：端口写死在代码里
const PORT = 3000;
app.listen(PORT, () => console.log(`listening on ${PORT}`));

这个 3000 叫硬编码（hard-coded）配置——把「会随环境变化的值」直接钉死在源码里。它有三个致命问题：

  • ① 同一份代码跑不了多个环境。 你本机想用 3000，但云服务器可能要求用 8080，或者由平台动态分配。写死意味着每个环境都要改源码、重新发布。
  • ② 改配置 = 改代码 = 重新部署。 想把对外域名从 localhost 换成 https://lnkp.io？本该是「改一行配置重启」，结果变成「改代码、提交、走 CI、重新构建」。配置和代码本是两种东西，变化频率完全不同。
  • ③ 最危险——密钥会被写进代码，然后进 git。 数据库密码、Stripe 支付密钥、JWT 签名密钥……一旦写进源码并 git push，就永久留在 git 历史里（即使后来删掉，历史里还在）。仓库一旦泄露或开源，密钥就泄露了。
🌍 实战：12-Factor App 第三条「Config」。 业界公认的云原生应用方法论 The Twelve-Factor App 明确规定：「在环境中存储配置（Store config in the environment）」。原文要点4：配置是「likely to vary between deploys（在不同部署间会变的东西）」——数据库连接、外部服务凭据、每次部署的域名等；把它们写成代码里的常量是「a violation of twelve-factor」。它给了一个极其好用的试金石（litmus test）：「能不能在任意时刻把代码库开源，而不泄露任何凭据？」如果不能，说明你的配置还没和代码分干净。LinkPulse 将来要放 GitHub、要让别人 code review，这条必须从第一天就遵守。
区分两个词。「配置（config）」= 随环境变化的值（端口、域名、密钥）；「常量（constant）」= 永远不变的业务值（比如「免费版最多 50 个短链」这种产品规则）。常量可以留在代码里；配置必须抽出去。判断标准就一句：本机和生产服务器上，这个值会不会不一样？ 会，就是配置。

2环境变量与 process.env

那配置放哪？答案是环境变量（environment variable）。

定义 + 类比。 环境变量是操作系统层面的键值对，由「启动进程的环境」提供给程序，而不是写在程序里。
类比：你（程序）去酒店入住，房间号（端口）、Wi-Fi 密码（密钥）不是你出生就带着的，而是前台（操作系统/部署平台）在你入住时塞给你的一张卡。同一个你，在不同酒店拿到不同的卡——这正是「一份代码、多个环境」的关键：代码不变，环境把配置「喂」进来。

在 Node.js 里，所有环境变量都挂在全局对象 process.env 上。process 是 Node 的全局对象（不用 import 就能用），代表「当前这个 Node 进程」。官方定义：process.env「returns an object containing the user environment（返回一个包含用户环境的对象）」2。

// 读取环境变量（注意：在 ESM 里 process 仍是全局可用的）
console.log(process.env.HOME);     // 你的家目录，比如 /Users/you
console.log(process.env.PORT);     // 我们待会儿要自己设的

先在终端临时设一个变量跑跑看，建立直观感受：

macOS / Linux（zsh/bash）

# 在命令前面临时设置，仅对这条命令生效
PORT=4000 node -e "console.log(process.env.PORT)"
# 输出：4000

Windows（PowerShell）

$env:PORT=4000; node -e "console.log(process.env.PORT)"
# 输出：4000
⚠️ 后端新手最容易踩的坑：环境变量永远是字符串。 官方明确：「Assigning a property on process.env will implicitly convert the value to a string（赋值会被隐式转成字符串）」2。所以 process.env.PORT 读出来是字符串 "4000"，不是数字 4000。如果你拿它去和数字比较、或某些 API 要求传 number，就会出诡异 bug。养成习惯：读数字配置就 Number(...)，读布尔就 === 'true'。
const port = Number(process.env.PORT);          // 字符串 → 数字
const isProd = process.env.NODE_ENV === 'production'; // 字符串 → 布尔

你可能注意到上面出现了 NODE_ENV。它是一个约定俗成（不是 Node 强制，但整个生态都认）的环境变量，用来标记当前运行环境。它正是下一节的主角。

3不同环境：development vs production

同一份 LinkPulse 代码，会跑在至少两种环境里，配置和行为都该不同：

维度development（你本机开发）production（线上给用户用）
PORT3000由平台决定，如 8080
BASE_URL（生成的短链前缀）http://localhost:3000https://lnkp.io
日志格式人类可读、带颜色（方便你看）纯 JSON（方便机器检索）
日志级别debug（啥都打，方便排查）info（少打点，省存储/降噪）
报错信息返回完整堆栈（你要调试）只返回「服务器错误」（不能泄露内部细节给攻击者）

所以后端代码里经常出现「根据环境分叉」的写法，靠的就是读 NODE_ENV：

const isProd = process.env.NODE_ENV === 'production';
// 例：开发环境用漂亮日志，生产环境用纯 JSON（第 6 节会真正用到）
🌍 实战：NODE_ENV=production 不只是「标记」，它会真的让你的程序变快。 很多库（包括我们在用的 Express5）会检测这个变量：设为 production 时，Express 会缓存视图模板、减少冗长错误输出，官方文档称这能带来数倍性能提升。所以线上务必设 NODE_ENV=production——忘了设是新手最常见的「线上比本地慢」原因之一。

4把配置放进 .env 文件并加载它

临时在命令前敲 PORT=4000 太累，配置一多就管不过来。业界标准做法是把它们集中写进一个名为 .env 的文本文件。在 LinkPulse 根目录新建：

# .env  —— LinkPulse 本地开发配置
PORT=3000
BASE_URL=http://localhost:3000
NODE_ENV=development

格式规则（来自 Node 官方文档，dotenv 也基本一致）1：

  • 一行一个 KEY=VALUE。
  • # 之后是注释：PORT=3000 # 本地端口 也合法。
  • 值可以用 "、' 或反引号包起来，引号会被去掉：NAME="LinkPulse" 得到 LinkPulse。
  • 支持多行值；export KEY=... 里的 export 会被忽略（方便从 shell 脚本复制粘贴）。

加载方式 A（推荐）：Node 20+ 内置 --env-file

从 Node v20.6.0 起，Node 内置了加载 .env 的能力，不需要装任何包。在启动命令加一个 --env-file 标志即可1：

node --env-file=.env 01-node-core/express-server.js

官方对它的描述：「Loads environment variables from a file relative to the current directory, making them available to applications on process.env」1。加载流程是这样的：

.env 文件──①读取──▶ --env-file 解析──②注入──▶ process.env──③读取──▶ 你的代码
好消息：从 Node 24.10 / 22.21 起，--env-file 不再是「实验性（experimental）」特性1，可以放心在生产用。如果你用的是更早的版本，运行时会看到一行 experimental 警告，功能仍可用。查版本：node -v，本课程要求 Node 20+。

把它写进 package.json 的 scripts，以后一个 npm run dev 就行（注意我们项目是 ESM，配合 --watch 自动重启）：

// package.json（节选）—— 让 dev 启动时自动读 .env
{
  "scripts": {
    "dev":   "node --watch --env-file=.env 01-node-core/express-server.js",
    "start": "node --env-file=.env 01-node-core/express-server.js"
  }
}
两个好用的兄弟标志。① --env-file-if-exists=.env（v22.9.0+）：文件不存在也不报错，适合「本地有 .env、生产靠平台注入环境变量」的场景1。② 可以传多个：node --env-file=.env --env-file=.env.local ...，后面的文件覆盖前面的同名变量1，方便用 .env.local 覆盖个人配置。

加载方式 B（了解）：dotenv 第三方包

在 Node 20 内置这功能之前，全世界都用 dotenv 这个包（至今仍是 npm 上下载量最大的包之一）。你会在无数老项目和教程里见到它，所以要认得：

dotenv：代码里加载

// 装包：npm install dotenv
import 'dotenv/config'; // 放在最顶上，越早越好

console.log(process.env.PORT);

内置：命令行加载（推荐）

// 不用装包，不用改代码
// 只在启动命令加标志：
// node --env-file=.env app.js

console.log(process.env.PORT);

怎么选？新项目（如我们的 LinkPulse）优先用内置 --env-file：零依赖、更干净。dotenv 的优势在于「在代码里加载」（某些工具链不方便改启动命令时有用）和一些高级特性（如变量插值、加密的 dotenvx）。本课程统一用内置方案。

⚠️ 最重要的一步：.env 必须进 .gitignore

真实事故 把含密钥的 .env 提交进 git → push 到 GitHub → 爬虫几分钟内扫到密钥 → 有人用你的 AWS/Stripe 密钥刷出天价账单。这类事故每天都在发生。

dotenv 官方对「该不该提交 .env」的回答只有一个字：「No.（不该）」，并要求把它加进 .gitignore3。在 LinkPulse 根目录创建/编辑 .gitignore：

# .gitignore
node_modules/
.env
.env.local
.env.*.local
🌍 实战：用 .env.example 当「配置说明书」。 .env 本身不进 git，但队友拉下代码后不知道要配哪些变量怎么办？通行做法是提交一个 .env.example（只有键、没有真值）进 git：
PORT= / BASE_URL= / NODE_ENV=。
新人 cp .env.example .env 再填上自己的值即可。这样「需要哪些配置」是公开文档，「配置的真实值」是私密的——LinkPulse 的仓库就该这么组织。记住 12-factor 那条试金石：仓库随时能开源而不泄露任何凭据。
如果不小心已经提交了怎么办？ 光删文件不够——它还在 git 历史里。dotenv 文档建议：「Remove it, remove git history（删文件，再清理 git 历史）」3，并且立刻把泄露的密钥作废、重新生成。已经暴露过的密钥，必须当成永久失效处理。

5为什么 console.log 不够：结构化日志

我们之前在 LinkPulse 里写过类似这样的日志中间件（Lesson 0006）：

app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);   // GET /api/links
  next();
});

本机看着挺好，但它扛不住生产环境。问题在于 console.log 打出来的是非结构化文本（unstructured text）——对人友好，对机器极不友好。

GET /api/links 201 in 13ms ← console.log 打出来的：一行字符串，没有结构
{"level":30,"time":1718000000000,"msg":"request completed","method":"GET","url":"/api/links","status":201,"ms":13} ← 结构化 JSON 日志：每个字段可被检索

结构化日志（structured logging）的定义：把每条日志写成机器可解析的格式（通常是 JSON），每个信息点都是一个有名字的字段（method、status、ms……），而不是糊成一行文字。为什么这事在生产环境是刚需？

  • ① 能按字段检索/过滤/聚合。 线上一天几百万条日志，你想查「所有 status>=500 的请求」或「ms>1000 的慢请求」。JSON 日志可以直接 WHERE status >= 500；纯文本只能用正则瞎抓，又慢又容易抓错。
  • ② 能携带上下文。 可以给每条日志附带 userId、requestId 等字段。出故障时，按 requestId 就能把同一次请求的所有日志串起来看，而不是在海量日志里大海捞针。
  • ③ 机器读得懂 = 能自动化。 监控系统能基于「error 级别日志数」自动告警、画图、统计。文本日志做不到这些。
🌍 实战：生产日志会被送进集中式日志平台。 真实的 SaaS 不会让你 SSH 上服务器 tail -f 看日志（服务器可能有几十台）。日志会通过采集器汇总到 ELK（Elasticsearch + Logstash + Kibana）、Grafana Loki、Datadog 这类平台，在网页上搜索、画图、告警。这些平台天然吃 JSON：你打的每个字段都会自动变成可搜索、可聚合的维度。LinkPulse 上线后排查「为什么某个用户的短链跳转失败」，靠的就是按 linkId 检索结构化日志。从今天起就打 JSON，等于为将来铺好了路。

6pino 入门：给 LinkPulse 装上专业日志

结构化日志当然可以自己 JSON.stringify 手搓，但有现成的高质量库就别造轮子。我们选 pino。

pino 是什么？ 一个专注于「快」的 Node.js JSON 日志库，官网一句话定位：「Super fast, all natural JSON logger for Node.js」6。它默认就输出 JSON，几乎零配置就能用。
🌍 实战：为什么选 pino 而不是 winston？ Node 生态最有名的两个日志库是 pino 和 winston。winston 历史更久、功能/可配置项更多；pino 的杀手锏是「快」和「低开销」——它的设计哲学是「Logging should never slow down an application（打日志绝不该拖慢应用）」6。它做法很聪明：尽量少在主线程里干活，重活（格式化、写文件）尽量挪到别处。对一个要扛流量、对延迟敏感的 SaaS 来说，「快」往往比「花哨配置」更重要，所以 pino 是近几年新项目的主流选择之一。

第一步：安装并创建 logger

npm install pino

新建一个文件集中创建 logger（这样全项目共用同一个实例）。注意：我们项目是 ESM（package.json 里 "type":"module"），所以用 import。

// 01-node-core/logger.js
import pino from 'pino';

// pino() 接受可选的 options，返回一个 logger 实例
const logger = pino();

export default logger;

用起来就这么简单——调用级别方法，传一句话：

import logger from './logger.js';

logger.info('hello world');

它打印到标准输出的是一行 JSON（官方示例输出）6：

{"level":30,"time":1531257112193,"msg":"hello world","pid":55956,"hostname":"x"}

注意几个 pino 自动加上的字段：time（时间戳）、pid（进程号）、hostname（主机名）——这些在多进程/多机器排查时极有用，console.log 可没有。

第二步：理解日志级别（log levels）

那个 "level":30 是什么？pino 把日志按严重程度分级，每级有方法名和一个数字值7：

方法数值什么时候用
logger.trace()10最啰嗦的调试细节
logger.debug()20开发期排查用
logger.info()30正常运转信息（默认最低级别）
logger.warn()40不致命但要留意
logger.error()50出错了（比如请求失败）
logger.fatal()60致命，程序可能要退出

级别的意义在于过滤：你设一个最低级别，低于它的日志不输出。pino 默认最低级别是 info（30），所以默认 trace/debug 不会打。这正好对应第 3 节的环境差异——开发环境调成 debug（啥都看），生产调成 info（降噪）：

// 01-node-core/logger.js —— 升级版：级别随环境变化
import pino from 'pino';

const logger = pino({
  // 生产 info，开发 debug；还能用环境变量 LOG_LEVEL 覆盖
  level: process.env.LOG_LEVEL
    ?? (process.env.NODE_ENV === 'production' ? 'info' : 'debug'),
});

export default logger;

第三步：记录错误 & 携带上下文字段

每个级别方法都接受「先一个对象，再一句话」的形式：logger.info([mergingObject], [message])。对象里的每个键都会被合并进这条 JSON 日志，成为可检索字段7。这就是「携带上下文」的实现方式：

// 记一次短链创建，带上业务字段
logger.info({ shortCode: 'aZ3x9', url: 'https://example.com' }, 'short link created');
// → {"level":30,...,"shortCode":"aZ3x9","url":"https://...","msg":"short link created"}

// 记错误：把 error 对象传进去，pino 会自动展开成 type/message/stack
try {
  // ... 某个会抛错的操作
} catch (err) {
  logger.error({ err }, 'failed to create short link');
}

第四步：用 pino 替换 LinkPulse 的日志中间件

现在把 Lesson 0006 那个 console.log 中间件换成 pino。注意响应结束后才知道状态码和耗时，所以我们监听 res 的 finish 事件来记一条完整的请求日志：

// 01-node-core/express-server.js（节选：日志中间件）
import express from 'express';
import logger from './logger.js';

const app = express();
app.use(express.json());

// ✅ 用 pino 记录每个请求（替换掉旧的 console.log 中间件）
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    logger.info(
      {
        method: req.method,
        url: req.originalUrl,
        status: res.statusCode,
        ms: Date.now() - start,
      },
      'request completed'
    );
  });
  next();
});

// 启动时也用 logger，并从 .env 读 PORT / BASE_URL
const PORT = Number(process.env.PORT) || 3000;
const BASE_URL = process.env.BASE_URL ?? `http://localhost:${PORT}`;

app.listen(PORT, () => {
  logger.info({ port: PORT, baseUrl: BASE_URL }, 'LinkPulse server started');
});
进阶（知道有这回事就行）： 生产中通常不自己手写请求中间件，而是用 pino 官方配套的 pino-http 中间件——它自动记请求/响应、自动给每个请求生成 requestId，省心。本课先用「手写 + finish 事件」把原理讲透；等 LinkPulse 进入正式阶段我们会换成 pino-http。

第五步：开发时把 JSON 变好看（pino-pretty）

纯 JSON 适合机器，但你本地开发时盯着一行行 JSON 很累。pino 官方提供 pino-pretty，开发环境把日志格式化成带颜色、人类可读的样子6。它是个独立工具，用管道接在后面即可——不污染生产（生产仍输出纯 JSON）：

npm install -D pino-pretty

# 本地开发：把 pino 的 JSON 通过管道喂给 pino-pretty 美化
node --env-file=.env 01-node-core/express-server.js | npx pino-pretty
[12:00:00.000] INFO: LinkPulse server started port: 3000 baseUrl: "http://localhost:3000" ← pino-pretty 美化后，本地看着舒服
🌍 实战：生产环境绝不要在 Node 进程里跑 pino-pretty。 官方建议把美化放在「进程之外」用管道做（就像上面），因为格式化很耗 CPU——让它在你的应用进程里跑，等于把 pino「快」的优势抵消掉。生产就让 pino 直吐 JSON，由日志采集器接走。规律：开发图好看（pretty），生产图快和可检索（raw JSON）。

  1. 确认 Node 版本 ≥ 20：终端跑 node -v。
  2. 在项目根目录新建 .env，写入：PORT=3000、BASE_URL=http://localhost:3000、NODE_ENV=development（各占一行）。
  3. 新建/编辑 .gitignore，把 .env 加进去（再加 node_modules/）。跑 git status 确认 .env 没出现在待提交列表里——这一步最重要。
  4. 顺手建一个 .env.example（只有键没有值：PORT= 等），这个可以 git add。
  5. 装日志库：npm install pino，再 npm install -D pino-pretty。
  6. 新建 01-node-core/logger.js，按第 6 节创建并 export default logger（级别随 NODE_ENV 变化）。
  7. 改 01-node-core/express-server.js：删掉旧的 console.log 日志中间件，换成 pino 版（监听 res.on('finish')）；app.listen 的端口改成读 Number(process.env.PORT)，并用 logger.info 打启动日志。
  8. 修改 package.json 的 dev 脚本为 node --watch --env-file=.env 01-node-core/express-server.js。
  9. 跑 npm run dev，再用浏览器或 curl 访问 http://localhost:3000/api/links，确认终端打出的是带 method/url/status/ms 字段的 JSON 日志。
  10. 加分项：把启动命令接上 | npx pino-pretty，对比美化前后的差别。
📖 本课首选精读源：pino 官方文档（getpino.io） 与 Node.js 官方 --env-file 文档。
