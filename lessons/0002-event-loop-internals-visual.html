Lesson 0002 · Node.js 后端之路

事件循环的内部机制：
调用栈、队列，与各个阶段（图解）

承接你的疑问。 你在动手题里遇到了 A E C 666 B 的诡异顺序——这正是深入事件循环的最好契机。 搞懂这套机制，你以后写 LinkPulse 的异步代码（查库、调外部 API、后台聚合）时，才能准确预判执行时序，不被「为什么这段先跑」坑到。

1调用栈（Call Stack）

调用栈是 JS 引擎里记录「此刻正在执行哪个函数」的一叠盒子，遵循后进先出（LIFO）： 调用一个函数就把它压入栈顶，函数返回就弹出。

console.log('A')　← 栈顶，正在执行
main()　← 主脚本
图 1：调用栈是 LIFO。因为 JS 是单线程，所以全程只有一个调用栈。

🔑 最关键的一条规则：只有当调用栈空了（同步代码全跑完），事件循环才会去各个队列里取回调来执行。 这就是为什么任何异步回调都得排队等同步代码先跑完——也是上节课 A、E 总在最前面的原因1。

2这些队列，到底存在哪？

你问得很好：「微任务队列存在哪？」答案是——它们不在你的代码里，而是运行时内部的数据结构，由运行时的不同部分各自管理：

Node.js 运行时（一个进程 · 一个主线程）
🟦 V8 引擎（执行 JS）
调用栈 Call Stack
Promise 微任务队列（microtask queue）
🟪 Node 核心
process.nextTick 队列（优先级最高）
🟩 libuv（C 库 · 事件循环）
各阶段的宏任务队列：timers / poll / check …
图 2：Promise 微任务队列由 V8 管理；nextTick 队列由 Node 核心管理； 定时器 / I/O 等宏任务队列由 libuv 管理2。它们都是内存里的内部结构，你只是往里「塞回调」。

3事件循环的 6 个阶段

当同步代码跑完、调用栈空了，libuv 的事件循环就开始一轮一轮（tick）地转，每轮按固定顺序经过 6 个阶段，每个阶段有自己的回调队列3：

  1. ① timers　执行到期的 setTimeout / setInterval 回调
  2. ② pending callbacks　执行少数被延迟到本轮的系统 I/O 回调
  3. ③ idle, prepare　仅 Node 内部使用
  4. ④ poll　取回新的 I/O 事件、执行绝大多数 I/O 回调（读文件/网络完成等）；必要时会在此阻塞等待
  5. ⑤ check　执行 setImmediate() 的回调
  6. ⑥ close callbacks　执行关闭类回调，如 socket.on('close', …)
  7. ↻ 回到 ①，进入下一轮 tick
图 3：6 个阶段按序轮转。重点：每执行完一个回调、每次切换阶段之间，都会先把 nextTick 队列清空，再把 Promise 微任务队列清空——这就是微任务「插队」的原理3。 阶段 4（poll）是后端最常打交道的，你查数据库的回调通常在这里被执行。
