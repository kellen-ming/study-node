Prisma 上手：类型安全地操作数据库
（让 LinkPulse 的表「长出」TypeScript 类型）

从「手写 SQL 字符串」到「类型安全的对象操作」。 上一课（0013）你把 LinkPulse 设计成了三张表——users / links / clicks，画好了字段、主键、外键和关系。 但那只是纸面设计：你还没在 TypeScript 代码里真正「拿到」这些表。 这一课你将认识 Prisma——一个 TS-first 的 ORM。你会把 0013 的设计写进一个 schema.prisma 文件， 用一条命令 prisma migrate dev 把它变成真实的数据库表， 然后用带自动补全、拼错字段编译期就报错的 Prisma Client 跑一个查询验证。 跑通后，下一课（0015）我们就把 Lesson 0006 那个返回「假短码」的 POST /api/links 改造成真正写库的 CRUD 接口。
版本说明（很重要）：本课基于 Prisma ORM 7（2025-11 起成为默认版本6）。 Prisma 7 和网上很多旧教程有三处关键差异：① 连接数据库要装一个 driver adapter（驱动适配器）； ② 生成的 Client 从你指定的 output 目录导入，不再从 @prisma/client 直接 import { PrismaClient }； ③ 数据库连接串配在 prisma.config.ts 里。本课代码全部按 v7 写，照着敲就能跑。

1ORM 是什么：对象 ↔ 表的「翻译官」

上一课你直接写 SQL：SELECT * FROM links WHERE code = 'abc123'。 能用，但有两个痛点你很快会撞上：

  1. SQL 是「字符串」，编辑器帮不上忙。 你把 code 拼成 cdoe，TypeScript 完全不知道——程序照样编译，跑到那一行才在运行时炸（甚至悄悄返回空结果）。
  2. 结果是「无类型的一坨」。 数据库返回的行，在 TS 里往往是 any 或 Record<string, unknown>，你访问 row.shortCode 时编辑器既不补全也不报错。

ORM（Object–Relational Mapping，对象关系映射）就是来解决这个落差的。 它是一层位于「你的代码」和「数据库」之间的库，干一件核心的事：把数据库里的「表 / 行 / 列」翻译成你代码里的「类型 / 对象 / 属性」，反过来也一样。

数据库世界（SQL）

表 links 一行记录 列 short_code (text) 外键 owner_id
⇄

你的代码世界（TS）

类型 Link 一个对象 {…} 属性 shortCode: string 关系 owner: User

有了这层翻译，你就不再手写 SQL 字符串，而是调方法：

上一课：手写 SQL 字符串

// 字段名是字符串，拼错也不报错
const sql =
  "SELECT * FROM links WHERE code = $1";
const rows = await db.query(sql, ["abc123"]);
const link = rows[0]; // 类型是 any 😟

本课：ORM 的方法调用

// 字段名有补全，拼错编译期就红线
const link = await prisma.link.findUnique({
  where: { shortCode: "abc123" },
});
// link 的类型是 Link | null ✅

类比：ORM 像「中英翻译官」

把数据库想成只会说「SQL 语」的老专家，你的 TS 代码只会说「对象语」。 ORM 就是站在中间的同声传译：你说「给我 shortCode 等于 abc123 的那条 link」， 它实时翻成 SELECT ... WHERE short_code = 'abc123' 递给数据库；数据库吐回一行，它再翻成一个带类型的 Link 对象塞回你手里。 你和数据库都不用学对方的语言。

🌍 实战：ORM 天然防 SQL 注入。 还记得上一课强调的「永远不要把用户输入拼进 SQL 字符串」吗？ 比如 "... WHERE code = '" + userInput + "'"，用户传个 ' OR '1'='1 就能把你整张表读走——这就是 SQL 注入。 用 ORM，你从源头就不写 SQL 字符串，而是传一个值 { shortCode: userInput }。 Prisma 在底层用参数化查询（parameterized query）把这个值作为「数据」而非「代码」发给数据库7，注入攻击因此几乎被结构性地杜绝。安全不再依赖你「记得转义」。

2为什么 LinkPulse 选 Prisma：schema 自动生成类型

ORM 有很多（Node 生态里还有 TypeORM、Sequelize、Drizzle 等）。我们选 Prisma，因为它对你这种 TypeScript 项目有一个杀手锏： 一份 schema 文件，自动生成全套 TypeScript 类型和一个带补全的查询客户端。

Prisma 的工作方式分三块，先建立全景，后面逐一动手：

  1. Prisma Schema（schema.prisma）：一个用 Prisma 自家语法写的文件，唯一的真相来源（single source of truth）。你在这里描述「有哪些表、每张表有哪些字段、表之间什么关系」2。
  2. Prisma Migrate：读你的 schema，生成 SQL 迁移文件并应用到数据库——也就是真正把表建出来 / 改结构。
  3. Prisma Client：根据 schema 自动生成的一个查询库。它知道你有哪些表、哪些字段、什么类型，所以 prisma.link.findUnique(...) 全程有补全，写错就报错。
「自动生成」是关键词。 你不用手写一行 TypeScript 接口去描述 Link 长什么样。 你只维护 schema.prisma 这一份文件，跑一下 prisma generate，对应的 Link、User、Click 类型和查询方法就全有了。 schema 改了，重新生成，类型自动跟着变——数据库结构和代码类型永远不会对不上。
🌍 实战：类型安全意味着「拼错字段，编译期就报错」。 这是 Prisma 最让人上瘾的体验。假设你查 link 时把字段名写成了 shortCdoe： 
// ❌ 字段名拼错
await prisma.link.findUnique({
  where: { shortCdoe: "abc123" },
  //       ~~~~~~~~~ TS 报错：Object literal may only
  //       specify known properties, 'shortCdoe' does not
  //       exist in type 'LinkWhereUniqueInput'
});
你还没运行，编辑器就划了红线、CI 的 tsc 直接构建失败。 对比手写 SQL 字符串——拼错要等到线上用户访问那条短链、接口 500 了才发现。 在一个要收费的 SaaS 里，这种「把运行时事故提前到编译期」的能力，直接换算成更少的线上故障和更短的排查时间。

3安装 Prisma 并初始化

回到你的 study-node 项目根目录（就是有 package.json、"type":"module" 的那个）。先装依赖。

这里要装四个包，各司其职（前端复习：-D / --save-dev 表示只在开发期用的依赖）：

# prisma：命令行工具（init / migrate / generate / studio），只开发期用 → -D
npm i -D prisma

# @prisma/client：运行时查询库，线上也要用 → 普通依赖
npm i @prisma/client

# 连接 PostgreSQL 的「驱动适配器」+ 底层驱动（Prisma 7 必需）
npm i @prisma/adapter-pg pg
npm i -D @types/pg            # pg 的 TypeScript 类型
为什么 v7 多了 @prisma/adapter-pg 和 pg？ Prisma 7 起，Client 必须通过一个 driver adapter（驱动适配器）连数据库； pg 是 Node 生态最常用的 PostgreSQL 驱动，@prisma/adapter-pg 是把 Prisma 接到 pg 上的「转接头」6。 你不用直接调它们，配一次就行（见第 6 节）。用别的库（如 MySQL）适配器名字会不同。

装完，初始化 Prisma：

npx prisma init --datasource-provider postgresql

这条命令在项目里生成几样东西1：

study-node/ ├─ prisma/ │ └─ schema.prisma ← 你的数据模型（核心，接下来要写它） ├─ prisma.config.ts ← Prisma CLI 配置（连接串在这里读） ├─ .env ← 放 DATABASE_URL（密码别提交到 git！） └─ package.json
把 .env 和生成目录加进 .gitignore。 .env 里有数据库密码，绝不能进版本库；下一节会指定 Client 生成到 src/generated/prisma，那是构建产物，也不必提交。 在你的 .gitignore 里确保有这两行：
.env
src/generated/prisma

4把 0013 的三张表写成 model

打开 prisma/schema.prisma。文件顶部有两个块：generator（告诉 Prisma 生成什么客户端、放哪）和 datasource（用什么数据库）。 把它们改成 v7 的写法： 

// prisma/schema.prisma

generator client {
  provider = "prisma-client"          // v7 的新生成器
  output   = "../src/generated/prisma" // 生成的 Client 放这里
}

datasource db {
  provider = "postgresql"             // 用 PostgreSQL（连接串在 prisma.config.ts）
}
和旧教程的差异：旧版这里写 provider = "prisma-client-js" 且不需要 output，连接串也写在 datasource 里（url = env("DATABASE_URL")）。 v7 改成了 "prisma-client"、必须指定 output，连接串移到了 prisma.config.ts6。看到旧写法不要照抄。

写第一个 model：User

在 schema 里，一个 model 就对应一张表。语法是 model 名字 { 字段... }。 每个字段写成 字段名 类型 [可选属性...]。先看 users 表怎么落地： 

model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  createdAt DateTime @default(now())
  links     Link[]   // 一个 User 拥有多个 Link（关系字段，下面解释）
}

逐项拆解这些「属性（attribute）」——它们以 @ 开头，是 Prisma 给字段加的修饰2：

  • Int / String / DateTime：Prisma 的字段类型，会映射成对应的 SQL 类型（Int→integer，String→text，DateTime→timestamp）。
  • @id：把这个字段设为主键（对应上一课的 PRIMARY KEY）。
  • @default(autoincrement())：插入时数据库自动生成递增整数 ID，你不用自己填。
  • @unique：唯一约束。两个用户不能用同一个 email。（这也让 email 能用于 findUnique 查询。）
  • @default(now())：插入时自动填当前时间。
  • links Link[]：注意它的类型是 Link[]（数组）。这不是数据库里的一列，而是一个关系字段——它在告诉 Prisma「一个 User 关联着多个 Link」。下面讲关系。

写关系：Link 属于某个 User（一对多）

上一课你用外键（foreign key）owner_id 把 link 关联到 user。 在 Prisma 里，一对多关系要在两边都写：父表（User）写一个数组字段，子表（Link）写「外键标量字段 + 关系字段」4。 

model Link {
  id        Int      @id @default(autoincrement())
  shortCode String   @unique          // 短码，唯一
  longUrl   String                    // 原始长链接
  createdAt DateTime @default(now())

  // —— 关系：每个 Link 属于一个 User ——
  ownerId   Int                       // 外键「标量字段」(真实存在于表里的列)
  owner     User     @relation(fields: [ownerId], references: [id])
  //         ↑关系字段      ↑用本表 ownerId   ↑指向 User.id

  // —— 关系：一个 Link 有多次点击 ——
  clicks    Click[]
}

关键理解（这是 Prisma 关系最容易绕晕的地方）：

  • ownerId Int 是真实的数据库列（就是外键），存着所属 user 的 id。
  • owner User @relation(...) 是虚拟的关系字段，数据库里没有这一列；它只存在于 Prisma / TS 层面，方便你写 link.owner.email 这样的代码。@relation(fields: [ownerId], references: [id]) 告诉 Prisma：用本表的 ownerId 去匹配 User 表的 id。
  • 对应地，上面 User 里的 links Link[] 是这条关系的「另一头」。两边合起来，Prisma 才知道这是一对多。

第三张表：Click（点击记录）

把 0013 的 clicks 也写上——每条点击属于一个 Link：

model Click {
  id        Int      @id @default(autoincrement())
  clickedAt DateTime @default(now())
  ipAddress String?                   // ? 表示可空（nullable）
  userAgent String?

  // 每条点击属于一个 Link
  linkId    Int
  link      Link     @relation(fields: [linkId], references: [id])
}
新语法点：String? 末尾的问号 表示这个字段可空（nullable），对应 SQL 里没有 NOT NULL。 没有问号的字段默认是必填的。这正好对上你写 TS 时熟悉的 ip?: string 可选属性直觉——拿到的对象里它的类型会是 string | null。

到这里，0013 的三张表 + 它们的关系，已经完整地变成了一份 schema.prisma。 这就是你整个数据层的「真相来源」——以后改结构，改这里就对了。

5配置连接串，跑迁移建表

先把数据库连接信息填好。打开根目录的 .env，写上你上一课用的那个 PostgreSQL 连接串5：

# .env  （格式：postgresql://用户名:密码@主机:端口/数据库名?schema=public）
DATABASE_URL="postgresql://postgres:你的密码@localhost:5432/linkpulse?schema=public"

然后看 prisma init 生成的 prisma.config.ts。v7 在这里读环境变量并告诉 CLI 连哪个库。确认它长这样（按需补全）6： 

// prisma.config.ts
import "dotenv/config";                         // 让它能读到 .env
import { defineConfig, env } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: { path: "prisma/migrations" },
  datasource: {
    url: env("DATABASE_URL"),                   // 从 .env 读连接串
  },
});
前端复习 + 后端语境：process.env / .env 你在前端构建里见过（如 VITE_*）。 后端用它的理由更硬核：数据库密码这类机密绝不能写进代码。 "dotenv/config" 这一行的作用就是在程序启动时把 .env 的内容加载进 process.env，env("DATABASE_URL") 再去取。 线上则由部署平台注入真正的环境变量，.env 只用于本地。

一条命令，把 schema 变成真实的表

万事俱备，跑迁移：

npx prisma migrate dev --name init

migrate dev 是开发期的迁移命令。--name init 给这次迁移起个名字（首次一般叫 init）。 它会一次性帮你做完这条流水线3：

读 schema对比当前库 → 生成 .sql写进 migrations/ → 在库里执行真正建表 → prisma generate生成 Client+类型

跑完你会看到 prisma/migrations/ 下多了一个带时间戳的文件夹，里面是一份 migration.sql——里面正是你没手写、由 Prisma 替你生成的 CREATE TABLE 语句。打开瞄一眼，会发现它和上一课你手敲的 SQL 长得很像，只是现在由 schema 自动产出。

🌍 实战：migrate 是团队协作与「schema 演进」的命脉。 真实项目里，表结构会不停变：这周给 links 加个 expiresAt（过期时间），下周给订阅功能加个 plan 字段。 你每改一次 schema、跑一次 migrate dev --name add_expires_at，Prisma 就新增一个带时间戳的迁移文件，记录「这一步把数据库从什么状态改成了什么状态」。 这些文件跟代码一起提交到 git。于是：① 同事拉下你的代码后跑 prisma migrate dev，他的本地库就精确地变成和你一样的结构，不用口头同步「记得加个字段哦」； ② 上线时用 prisma migrate deploy 把同一套迁移按顺序应用到生产库，可重复、可追溯、可回滚思考。 没有迁移工具的年代，改线上表结构靠手敲 SQL、靠人记，是事故高发区。
开发 vs 生产：migrate dev 只用于本地开发（它可能在检测到「漂移」时提示重置数据库）。 线上千万别用它——生产环境用 npx prisma migrate deploy，它只老老实实地应用已有迁移、绝不重置数据3。这一课先把本地跑通即可。

6用 Prisma Client 跑第一个查询

迁移成功的同时，Prisma 已经把 Client 生成到了你指定的 src/generated/prisma（第 4 节配的 output）。 现在写一段 TS 来用它。先建一个共享的 client 实例文件——这一步关乎性能，请认真看实战框。 

// src/db.ts —— 全项目共享的 Prisma Client 单例
import { PrismaClient } from "./generated/prisma";  // 注意：从生成目录导入，不是 @prisma/client
import { PrismaPg } from "@prisma/adapter-pg";

// 驱动适配器：把 Prisma 接到 pg 上，读 .env 里的连接串
const adapter = new PrismaPg({
  connectionString: process.env.DATABASE_URL!,
});

export const prisma = new PrismaClient({ adapter });
🌍 实战：为什么要「单例」？避免连接耗尽。 每 new PrismaClient() 一次，就会开一个数据库连接池（connection pool）。 数据库能同时接受的连接数是有上限的（PostgreSQL 默认 100 上下）。 如果你在每个路由、每个请求里都 new 一个 Client，连接数会爆炸式增长，很快把数据库连接占满，新请求直接报「too many connections」——这是 Node + Prisma 最经典的线上事故之一8。 正确做法：整个应用只创建一个 prisma 实例，到处 import 复用它。 所以我们把它单独放在 src/db.ts 里 export 出去。 （Prisma 官方还提示：长期运行的服务不需要每次查完手动 $disconnect()，那样反而因频繁重连拖慢性能8。）
关于「生成目录放哪 / 要不要提交」： v7 把 Client 生成到你 output 指定的目录（我们放在 src/generated/prisma），而非旧版的 node_modules。 这个目录是构建产物，通常不提交到 git（已加进 .gitignore）；团队成员拉代码后跑一次 npx prisma generate 即可重新生成。

现在写一个验证脚本，把三类基本查询都跑一遍：

// src/try-prisma.ts —— 验证脚本，跑一遍就删
import "dotenv/config";
import { prisma } from "./db.js";  // ESM 里相对导入要带 .js 后缀

async function main() {
  // ① create：建一个 user，再给它建一条 link（嵌套写入，一次搞定）
  const user = await prisma.user.create({
    data: {
      email: "founder@linkpulse.dev",
      links: {
        create: [{ shortCode: "abc123", longUrl: "https://example.com/very/long" }],
      },
    },
  });
  console.log("已创建 user:", user.email);   // user.email 有补全，类型是 string

  // ② findUnique：用唯一字段 shortCode 精确查一条 link
  const link = await prisma.link.findUnique({
    where: { shortCode: "abc123" },
  });
  console.log("查到 link 指向:", link?.longUrl); // 类型是 Link | null，所以用 ?.

  // ③ findMany：查所有 link，并把每条所属的 owner 一起带出来
  const all = await prisma.link.findMany({
    include: { owner: true },     // include：把关系数据一并查出
  });
  console.log("共有", all.length, "条 link，第一条属于", all[0]?.owner.email);
}

main()
  .catch((e) => { console.error(e); process.exit(1); })
  .finally(() => prisma.$disconnect());  // 一次性脚本跑完断开是对的

逐个认识这三个方法（它们是每个 model 都自动拥有的 CRUD 方法9）：

  • create({ data })：插入一条记录。上面还演示了嵌套写入——建 user 的同时，用 links: { create: [...] } 顺手把它的 link 也建了，Prisma 自动把外键 ownerId 填对。
  • findUnique({ where })：按唯一字段查一条（只能用 @id 或 @unique 字段）。查不到返回 null，所以 TS 类型是 Link | null，你必须用 link?.longUrl 处理空值——这正是类型安全在帮你提前想到「查不到怎么办」。
  • findMany({ ... })：查多条，返回数组。include: { owner: true } 让它把每条 link 关联的 user 一并查出来（对应 SQL 的 JOIN），于是 all[0].owner.email 直接可用且有类型。

用 tsx 跑这个脚本（一个直接执行 .ts 的运行器，比手动编译方便）：

npm i -D tsx
npx tsx src/try-prisma.ts

看到打印出 user 邮箱、link 指向和条数，说明你的代码已经真正读写到了数据库。恭喜——LinkPulse 第一次有了「会记事」的数据层。

🌍 实战：用 Prisma Studio 可视化看数据。 脚本跑完，想亲眼确认数据进库了？不用再去敲 SELECT。Prisma 自带一个可视化工具： 
npx prisma studio
它会在 localhost:5555 打开一个网页界面，像 Excel 一样浏览、筛选、增删改你的每张表10。 你能直接看到刚建的 User / Link 行，点进去还能看到它们的关系。 日常开发里它特别适合：快速核对数据是否写对、手动造几条测试数据、给非技术同事演示。比反复手写查询语句快得多。

7小结：这一层做对了什么

你这一课为 LinkPulse 搭起了一条类型安全的数据通道：

  • schema.prisma = 数据库结构的唯一真相来源，0013 的三张表 + 关系全在这。
  • migrate dev = 把 schema 变成真实的表，并留下可提交、可协作、可上线的迁移历史。
  • Prisma Client（单例） = 带补全、拼错即报错的查询入口；create / findUnique / findMany 覆盖了基本读写。
  • Prisma Studio = 随时可视化核对数据。
