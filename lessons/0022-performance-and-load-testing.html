Lesson 0022 · Node.js 后端之路

性能优化与压测：先测量，再优化

为什么学这个？ LinkPulse 已经能「创建短链 → 存库 → GET /:code 做 302 跳转 → Redis 缓存」。但要「上线收费」，你必须能回答一个收费用户一定会问的问题：「我每秒能扛多少跳转？慢的时候有多慢？」 本章教你不靠感觉、靠数据地回答它：用 autocannon 压测拿到 QPS 与 p50/p99 延迟，定位真正的瓶颈（N+1、缺索引、连接池、阻塞事件循环），再针对性优化——并用「加 Redis 前 vs 后」的压测对比，亲眼看见数量级的吞吐提升。

1先测量，再优化：别凭感觉动手

你大概率听过这句话：「过早优化是万恶之源」（premature optimization is the root of all evil）。它的真正含义不是「不要优化」，而是「不要在没有数据的情况下优化」。后端性能优化的第一原则只有一句：

原则：先测量（measure），后优化（optimize）。 没有测量数据就改代码，等于闭着眼睛拧螺丝——你会把时间花在「看起来很慢但其实没人调用」的代码上，而真正的瓶颈纹丝不动。

这里有两个互补的工具概念，前端的你可能对前者很熟：

  • 压测（load testing / benchmarking）：从外部，用大量并发请求轰炸一个接口，测出它的吞吐量和延迟分布。回答「这个接口在压力下表现如何」。
  • 剖析（profiling）：从内部，记录代码每个函数耗时多少、事件循环是否被阻塞、哪行 SQL 最慢。回答「时间到底花在哪了」。

典型工作流是个闭环：压测发现「慢」 → 剖析定位「为什么慢」 → 改一处 → 再压测确认「真的快了」。每次只改一个变量，否则你分不清是哪个改动起了作用。本章聚焦压测这一头，剖析会在排查时顺带用到。

🌍 实战：先 profile/压测定位真瓶颈，再动手。 真实项目里最常见的浪费是：工程师凭直觉认定「ORM 慢」，花两天把 Prisma 换成手写 SQL，结果 p99 没动——因为真正的瓶颈是一个没加索引的 WHERE short_code = ?，加一行索引就解决了。先量，能帮你省下 90% 的无效努力。

读懂延迟分布：为什么看 p99，不看平均

压测会给你一组百分位延迟（latency percentiles）：p50、p95、p99……p99 = 23ms 的意思是「99% 的请求在 23ms 内返回，只有最慢的 1% 超过它」。平均值（average）会骗人，因为它被大量快请求拉低，掩盖了那条长尾。

p505 ms
p9012 ms
p99180 ms

上图这种「平均 7ms 但 p99 高达 180ms」的接口，意味着每 100 个用户就有 1 个等了快 0.2 秒。在收费 SaaS 里，这 1% 往往是付费大客户的批量请求。SLA（服务等级协议）几乎都按 p99/p99.9 来签，从不按平均签。

🌍 实战：关注 p99，而非平均。 一个流传很广的判断：「平均延迟告诉你系统平时好不好用；尾部延迟（p99/p99.9）告诉你系统最坏时有多糟，而最坏的体验才是用户记住并退订的那个。」LinkPulse 的跳转接口被嵌在别人网站里，一次 p99 卡顿就可能让对方页面整体变慢——你必须盯 p99。

2用 autocannon 压测 LinkPulse 的跳转接口

autocannon 是一个用 Node.js 写的 HTTP/1.1 压测工具，由 Fastify 核心作者维护，CLI 和编程式 API 都有，能打出很高的并发并给出 p50/p99 延迟与 Req/Sec 吞吐1。它和我们的技术栈天然契合——直接 npm i -D 装到开发依赖即可。

npm i -D autocannon
# 它附带类型，无需单独装 @types

2.1 CLI 三个核心参数

最常用的三个 flag1：

  • -c / --connections：并发连接数（默认 10）。模拟「同时有多少个客户端」。
  • -d / --duration：压测持续秒数（默认 10）。
  • -p / --pipelining：每个连接上的流水线请求数（默认 1）。先用默认 1 即可。

假设 LinkPulse 跑在 http://localhost:3000，库里已有短码 abc123。压测跳转接口：

# 100 并发，压 20 秒，打 GET /abc123
npx autocannon -c 100 -d 20 http://localhost:3000/abc123
坑：302 会被「跟随」吗？ 跳转接口返回 302。autocannon 默认不跟随重定向，它把 302 当成一次正常完成的请求来计数——这正是我们想要的（我们测的是 LinkPulse 自己的耗时，不是目标网站的）。但要注意：3xx 不是 2xx，autocannon 会把它统计进 non2xx 计数，这是正常现象，不是错误。真正的错误看 errors / timeouts 字段。

2.2 读懂输出表

autocannon 跑完打印两张表1：Latency（延迟，含 2.5%/50%/97.5%/99% 等百分位、平均、标准差、最大值，单位 ms）和 Req/Sec + Bytes/Sec（吞吐，每秒采样一次）。重点盯三处：

  • Req/Sec 的 Avg：平均每秒处理多少请求 ≈ 你的 QPS。
  • Latency 的 p99：尾部延迟。
  • 非预期的 errors / timeouts / non2xx：有大量超时说明系统被压垮了。

2.3 编程式 API：把压测脚本化（TypeScript / ESM）

CLI 适合临时一压；要做「优化前后对比」并把数字记下来，写成脚本更靠谱。autocannon 的编程式 API 返回一个结果对象，含 requests / latency / throughput 三个直方图（histogram）字段，每个都带 average、p99 等属性1。

// scripts/bench-redirect.ts —— 压测 LinkPulse 跳转接口
import autocannon from 'autocannon';

async function bench(label: string, url: string) {
  // autocannon 返回 Promise（不传 callback 时）
  const result = await autocannon({
    url,
    connections: 100,   // -c
    duration: 20,       // -d
    pipelining: 1,      // -p
  });

  console.log(`\n=== ${label} ===`);
  console.log(`Req/Sec (avg): ${result.requests.average.toFixed(0)}`);
  console.log(`Latency p50 : ${result.latency.p50} ms`);
  console.log(`Latency p99 : ${result.latency.p99} ms`);
  console.log(`Errors/Timeouts: ${result.errors} / ${result.timeouts}`);
  return result;
}

const code = process.argv[2] ?? 'abc123';
await bench('GET /:code 跳转', `http://localhost:3000/${code}`);

# package.json 里加一条脚本（ESM + tsx 直接跑 TS）
# "bench:redirect": "tsx scripts/bench-redirect.ts"
npm run bench:redirect -- abc123
压测纪律： ① 用生产构建（NODE_ENV=production），别用带 source-map / 热重载的 dev server，否则数字毫无意义。② 压测机和被压服务分开更准（同机会互抢 CPU），本地学习同机也行，但别和数字较真到个位数。③ 每次只改一个变量。

3缓存的威力：跳转接口「直查库」vs「Redis 缓存」压测对比

你在 Lesson 0017 已经为跳转接口加了 cache-aside（旁路缓存）：先查 Redis，命中（hit）就直接用；未命中（miss）才查 Postgres 再回填 Redis。本节用压测量化这层缓存到底值多少钱。

先快速复习两个指标的定义：

  • 缓存命中率（hit rate） = 命中次数 ÷ 总请求数。短链跳转是「读多写极少 + 热点集中」的典型场景，命中率通常能到 95%+。
  • 为什么缓存快：Redis 是内存键值存储，一次 GET 通常是亚毫秒级；而 Postgres 即使走索引，也要经过连接、查询计划、磁盘/页缓存、网络往返，单次往往是毫秒级。差一个数量级。

为做对比，临时给跳转接口加一个「绕过缓存」的开关（仅压测用）：

// src/links/redirect.controller.ts （节选，对比用）
import type { Request, Response } from 'express';
import { redis } from '../redis.js';
import { prisma } from '../prisma.js';

export async function redirect(req: Request, res: Response) {
  const { code } = req.params;
  const bypass = req.query.nocache === '1'; // 压测开关：?nocache=1 强制查库

  if (!bypass) {
    const cached = await redis.get(`link:${code}`);
    if (cached) return res.redirect(302, cached); // 缓存命中
  }

  const link = await prisma.link.findUnique({
    where: { shortCode: code },
    select: { targetUrl: true }, // 只取需要的列，见第 5 节
  });
  if (!link) return res.status(404).send('Not found');

  if (!bypass) await redis.set(`link:${code}`, link.targetUrl, 'EX', 3600);
  return res.redirect(302, link.targetUrl);
}

# 两轮对比压测
npx autocannon -c 100 -d 20 http://localhost:3000/abc123              # 走缓存
npx autocannon -c 100 -d 20 "http://localhost:3000/abc123?nocache=1" # 直查库

你会看到类似这样的差距（具体数字随机器变化，但数量级差异是真实且可复现的）：

方案Req/Sec (avg)p50 延迟p99 延迟
直查 Postgres（?nocache=1）~3,00028 ms120 ms
Redis 缓存命中~18,0004 ms15 ms
🌍 实战：跳转接口加 Redis 前后压测对比能看到巨大差距。 这正是短链产品的核心竞争力——跳转必须快且稳。缓存把吞吐拉高数倍、把 p99 压低一个数量级。但别忘了缓存的代价：数据一致性。当用户改了 targetUrl，你必须主动 redis.del(\`link:\${code}\`) 让旧值失效（write 时删 key），否则跳转会指向旧地址。压测看的是性能，但上线前一致性必须想清楚。

4头号 ORM 坑：N+1 查询问题

现在做一个 LinkPulse 的新功能：「我的链接」列表页——展示当前用户的所有短链，每条还要显示「创建者邮箱」（来自 users 表）。一个前端转后端最容易写出的「天真版本」是这样：

// ❌ 天真写法：制造一个 N+1
const links = await prisma.link.findMany({
  where: { userId },
}); // ① 一条查询：拿到 N 条 link

const rows = [];
for (const link of links) {
  const user = await prisma.user.findUnique({   // ② 循环里每条再查一次 user！
    where: { id: link.userId },
    select: { email: true },
  });
  rows.push({ ...link, ownerEmail: user?.email });
}

这段代码看起来人畜无害，运行也完全正确——这正是它危险的地方。N+1 查询问题（N+1 problem）的定义：先执行 1 条查询拿到 N 条记录，然后对每条记录再各发 1 条查询去取关联数据，总共发出 1 + N 条 SQL2。

类比： 你去图书馆借 100 本书，不是推一辆车一次搬走，而是跑 100 趟、每趟搬一本。每趟（每条 SQL）的固定开销——网络往返、查询解析、连接占用——单看很小，乘以 100 就成了灾难。列表只有 10 条时你根本察觉不到；某个大客户有 5000 条链接时，这个接口直接超时。
🌍 实战：N+1 是 ORM 的头号坑——「查了一个列表，又对列表里每一条去查它的关联」。 它在开发期几乎不可见（本地数据少、延迟低），一上生产就爆。排查信号有三个：① 接口延迟随列表行数线性增长；② 数据库 QPS 异常高但每条都简单；③ 打开 Prisma 的查询日志，看到刷屏的重复 SELECT ... FROM users WHERE id = $1。

4.1 用 Prisma 的 include / select 一次取回关联

解法是把「循环里的 N 次查询」合并成「跟主查询一起完成」。Prisma 官方推荐用 include 或 select 做预加载（eager loading），让关联数据随主查询一并取回2：

✅ include：在结果上挂载完整关联对象

const links = await prisma.link.findMany({
  where: { userId },
  include: { user: { select: { email: true } } },
});
// links[0].user.email 可直接用

✅ select：精确挑选要的字段（更省）

const links = await prisma.link.findMany({
  where: { userId },
  select: {
    id: true, shortCode: true, targetUrl: true,
    user: { select: { email: true } }, // 嵌套关联
  },
});

这样从 1 + N 条 SQL 降到常数条。默认情况下，Prisma（v5+）对带 include/select 的关联查询使用数据库层 JOIN 策略（relationLoadStrategy: "join"）：在 PostgreSQL 上用 LATERAL JOIN + JSON 聚合，把关联拼接放在数据库里一次完成3。你也可以显式选择 relationLoadStrategy: "query"（应用层多查询再合并），但官方建议默认用 join，遇到特殊数据分布再用 profile 验证哪个更快3。

include vs select 怎么选？ 想在已有字段基础上额外加关联 → 用 include；想精确控制返回哪些列（少传数据、少占带宽） → 用 select。注意两者不能在同一层级混用。SaaS 接口建议优先 select，只取前端真正要用的列。

5更深一层：索引、连接池、不阻塞事件循环

修掉 N+1 后，如果某条接口还慢，瓶颈通常在这三处之一。

5.1 索引：用 EXPLAIN 看查询计划（复习 0013）

跳转接口的核心是 WHERE short_code = $1。short_code 在建表时已加 unique，Postgres 会自动为唯一约束建索引——但当你新增其它高频过滤列（比如按 user_id 查列表）时，必须确认它走了索引。用 EXPLAIN ANALYZE 看真实执行计划4：

-- 在 psql 里直接跑
EXPLAIN ANALYZE SELECT * FROM links WHERE user_id = 42;

读计划只看一处关键词4：

  • Seq Scan（顺序扫描）：逐行扫全表。大表上出现它通常意味着缺索引——表越大越慢。
  • Index Scan（索引扫描）：先查 B-tree 定位，再回表取数据。从大表里取一小部分行时，比 Seq Scan 快得多。

给高频过滤列加索引（Prisma schema 里声明，再 migrate）：

// prisma/schema.prisma
model Link {
  id        Int      @id @default(autoincrement())
  shortCode String   @unique          // 自动有索引
  targetUrl String
  userId    Int
  user      User     @relation(fields: [userId], references: [id])
  createdAt DateTime @default(now())

  @@index([userId])                    // ← 为「按用户查列表」加索引
}
别盲目加索引。 索引加速读，但拖慢写（每次 INSERT/UPDATE 都要维护索引）并占空间。小表上 Postgres 的规划器会故意选 Seq Scan，因为全表扫比走索引还快——这是对的，不是 bug4。所以：先 EXPLAIN 确认真有 Seq Scan 拖慢了大表，再加索引。又是「先测量」。

5.2 连接池（connection pool）

连接池是一组预先建好、可复用的数据库连接。为什么需要它？建立一条新的 Postgres 连接很贵（TCP 握手 + 认证 + 后端进程创建），如果每个请求都新建一条，高并发下会瞬间打满数据库的 max_connections 而报错。池子的作用是「借了用完还、循环复用」。

好消息：Prisma Client 内置连接池，你已经在用了。默认池大小公式是 physical_cpus × 2 + 1（4 核机 = 9 条连接）5。可以在连接串里用 connection_limit 显式覆盖：

# .env —— 例如把池设为 20 条
DATABASE_URL="postgresql://user:pass@localhost:5432/linkpulse?connection_limit=20"
🌍 实战：连接池太小或太大都坑。 太小 → 高并发时请求排队等连接，你会看到 Prisma 报 connection pool timeout，p99 飙升但 CPU 却很闲（在等）。太大 → 多个应用实例 × 各自的池，加起来超过数据库 max_connections，直接 too many connections 崩库。多实例部署时，官方建议每实例的 connection_limit ≈ 默认池大小 ÷ 实例数5。压测时若发现 QPS 上不去且 DB CPU 没满，先怀疑连接池。

5.3 别阻塞事件循环（呼应 0020）

复习 0020 的核心：Node.js 用单线程事件循环（event loop）处理所有 JS 回调与非阻塞 I/O。官方《Don't Block the Event Loop》指出：如果某个回调长时间占用 CPU 不返回，这条线程就被阻塞，期间所有其它请求都得排队等——轻则吞吐暴跌，重则整个服务拒绝响应6。

注意区分：跳转接口里的 await redis.get(...)、await prisma... 都是 I/O，不会阻塞事件循环（等待期间线程去干别的）。真正危险的是同步 CPU 密集计算。LinkPulse 未来会有这种场景：导出某条链接的全部点击记录为大 CSV、或对 user-agent 跑复杂正则解析。如果直接在请求里同步算：

// ❌ 在请求处理里做 CPU 密集计算，阻塞所有人
app.get('/links/:id/export', async (req, res) => {
  const clicks = await prisma.click.findMany({ where: { linkId } });
  const csv = heavySyncFormat(clicks); // 假设这步 CPU 密集、跑 800ms
  res.type('text/csv').send(csv);       // 这 800ms 里，全站其它请求全卡住
});

官方给出两类解法6：分片（partitioning）——把大循环拆成小块、用 setImmediate 在块之间让出事件循环，适合简单遍历；卸载（offloading）——把整块重计算交给 worker_threads 工作线程跑，主线程只收结果，适合复杂逻辑。LinkPulse 的 CSV 导出属于后者：

// ✅ 卸载到 worker（呼应 0020 的 worker_threads）
import { Worker } from 'node:worker_threads';

app.get('/links/:id/export', async (req, res) => {
  const clicks = await prisma.click.findMany({ where: { linkId } });
  const csv = await runInWorker('./workers/csv.js', clicks); // 重活在别的线程
  res.type('text/csv').send(csv); // 主线程全程不被阻塞，跳转接口照常飞快
});

function runInWorker<T>(file: string, data: unknown): Promise<T> {
  return new Promise((resolve, reject) => {
    const w = new Worker(new URL(file, import.meta.url), { workerData: data });
    w.once('message', resolve);
    w.once('error', reject);
  });
}
🌍 实战：怎么发现事件循环被阻塞？ 压测时如果所有接口（包括最简单的健康检查）的 p99 同时无规律飙高，而 CPU 单核打满，几乎可以断定有同步重计算在堵路。生产里可用 perf_hooks 的 monitorEventLoopDelay() 或 pino 配合事件循环延迟指标做监控，超阈值告警。规律是：请求处理路径里不要出现耗时的同步代码。

🧠 小检验 性能

压测「我的链接」列表接口时，发现接口延迟随返回行数线性增长，且数据库日志里刷屏出现大量相同的 SELECT ... FROM users WHERE id = $1。最可能的原因与修法是？

⌨️ 动手：压测跳转 + 制造并修复一个 N+1

  1. npm i -D autocannon tsx，确认 LinkPulse 以生产模式跑在 :3000，库里有一个可用短码（如 abc123）。
  2. 给跳转 controller 加 ?nocache=1 绕过开关（见第 3 节），用 npx autocannon -c 100 -d 20 分别压「走缓存」与「直查库」，把两组 Req/Sec、p50、p99 记到一张表里，确认缓存带来数量级差距。
  3. 写出第 4 节的「我的链接」列表接口的天真版本（N+1），打开 Prisma 查询日志（new PrismaClient({ log: ['query'] })），压测它并数一数发了多少条 SQL。
  4. 用 include 或 select 改成预加载版本，重新压测，对比 SQL 条数与 p99——记录前后差异。
  5. （加分）给 links 表的 userId 加 @@index，迁移后用 EXPLAIN ANALYZE 看列表查询是否从 Seq Scan 变成 Index Scan。
