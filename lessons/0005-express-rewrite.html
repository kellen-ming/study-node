Lesson 0005 · Node.js 后端之路 · 阶段 2 开始

用 Express 重写你的服务器
（告别 if/else 路由）

欢迎进入阶段 2。 你已经亲手感受过原生 http 的「痛」——一堆 if/else 判断 method 和 url、手动 JSON.stringify、手动设响应头。 现在用 Express（Node 最流行的 web 框架）把它收拾干净。LinkPulse 真正的 API，就从这一课开始搭。

1为什么需要框架？Express 是什么

原生 http 能用，但你已经体会到它的麻烦：路由靠手写 if/else、返回 JSON 要自己 stringify + 设 Content-Type、方法判断也得自己来……项目一大就乱成一团。

框架（framework）就是把这些重复、易错的活儿封装好，给你一套干净的 API 和结构，让你专注写业务。 Express 是 Node 最流行的极简 web 框架：生态最大、文档最全、招聘最常见1。它是你最好的起点。

🌍 实战选型：除了 Express，还有 Fastify（更快、更现代）、NestJS（更结构化、适合大型项目，咱们阶段 4 会用到它）。 但 Express 概念最简单、资料最多，先用它把 web 框架的核心吃透，换别的框架都是触类旁通。

2第一次装真实依赖：npm install express

Express 不是 Node 内置的，要从 npm 装。在项目根目录运行：

npm install express

这条命令做了两件事：

  • 把 express 及其依赖下载到 node_modules/ 文件夹；
  • 在 package.json 里记下这条依赖（还记得 0004 讲的 dependencies 吗？——运行时要用的包就在这）：
"dependencies": {
  "express": "^5.1.0"        // 版本号可能不同
}
🌍 实战：这正是你 0004 给 node_modules/ 加 .gitignore 的原因——它不进 git。别人拿到你的项目，只要 package.json 在，一句 npm install 就能把所有依赖重建出来。

3用 Express 重写

同样的服务器，看看 Express 版有多干净。先感受「同一个 /api/health，两种写法」：

😩 原生 http（你之前写的）

if (url === '/api/health'
    && method === 'GET') {
  res.writeHead(200, {
    'Content-Type': 'application/json'
  })
  res.end(JSON.stringify(payload))
  return
}

😌 Express

app.get('/api/health', (req, res) => {
  res.json(payload)
})

完整的 Express 版服务器：

import express from 'express'

const app = express()
const port = 3000

app.get('/', (req, res) => {
  res.send('Hello Guys!')                       // 发文本
})

app.get('/api/health', (req, res) => {
  res.json({ status: 'ok', uptime: process.uptime() })   // 发 JSON
})

// 没匹配到任何路由 → 404（放在所有路由的最后）
app.use((req, res) => {
  res.status(404).send('Not Found')
})

app.listen(port, () => {
  console.log(`Server running at http://localhost:${port}/`)
})

对照着读，你会发现 Express 帮你包办了三件原来要手写的事：

  • 路由分发：app.get(path, handler) 直接按「方法 + 路径」匹配，不用再写 if/else 判断 method/url1。app.post、app.put、app.delete 同理。
  • 发 JSON：res.json(obj) 自动帮你设好 Content-Type: application/json 并序列化——告别手动 writeHead + JSON.stringify。
  • 状态码：res.status(404).send(...) 链式调用，清爽。

4路由参数 :param —— 这就是 LinkPulse 的核心

真实 API 的路径里常有动态部分，比如「获取第 42 号用户」是 /users/42。 在 Express 里，用 : 定义这种路由参数（route param），捕获到的值会放进 req.params1： 

app.get('/greet/:name', (req, res) => {
  const { name } = req.params      // 访问 /greet/kellen → name 就是 "kellen"
  res.json({ hello: name })
})
🌍 这正是 LinkPulse 的心脏：用户访问短链 /abc123，你用 app.get('/:code', …) 配 req.params.code 拿到 "abc123"，再去数据库查它对应的原始网址、然后跳转。 你今天写的 /greet/:name，就是短链跳转的雏形。

📌 顺带认识两个「兄弟」（下节细讲）：req.query 接收 ?key=value 这种查询参数；req.body 接收 POST 提交的请求体。三者别搞混：params 在路径里、query 在 ? 后、body 在请求体里。

🧠 小检验 路由参数

你定义了路由 app.get('/links/:code', handler)。当有人访问 /links/abc123 时，在 handler 里怎么拿到 abc123？

⌨️ 动手：用 Express 重写（15 分钟）

  1. 在项目根目录运行 npm install express，然后打开 package.json 确认 dependencies 里多了 express。
  2. 新建 01-node-core/express-server.js，照 §3 写出 Express 版（/ 和 /api/health）。
  3. 加一个路由参数路由：/greet/:name，返回 { hello: name }。
  4. 加上 §3 末尾的 404 处理（app.use 放最后）。
  5. 运行 node --watch 01-node-core/express-server.js（或把 package.json 的 dev 脚本指向它，再 npm run dev）。
  6. 挨个访问验证：/、/api/health、/greet/你的名字、/不存在的路径（应返回 404）。

💡 对比一下：实现同样的功能，Express 版是不是短了一大截、也好读多了？这就是框架的价值。
