Lesson 0008 · Node.js 后端之路

用 express.Router 拆分路由
+ 分层架构（route / controller / service）

为什么学这个？ 你的 express-server.js 现在还塞得下：几条 app.get、一条 app.post /api/links、404 兜底。 但 LinkPulse 真要上线收费，路由会爆炸式增长——创建短链、查列表、查详情、删除、访问统计、用户、订阅、计费回调…… 全堆在一个文件里，会变成谁也不敢碰的「意大利面条」。 这一章你将学会两件让项目长期可维护的核心功夫：① 用 express.Router() 把路由按模块拆出去；② 把一个接口拆成 route → controller → service 三层。 做完，LinkPulse 的 links 功能就从「一坨」变成「三个各司其职、可单独测试」的文件。

1先认清痛点：单文件为什么会失控

目前你的服务器大概长这样（沿用 Lesson 0006 写的假短码逻辑）： 

// express-server.js —— 所有东西都挤在这里
import express from 'express'
const app = express()
app.use(express.json())

app.get('/', (req, res) => res.send('LinkPulse OK'))
app.get('/greet/:name', (req, res) => res.send(`Hi ${req.params.name}`))

app.post('/api/links', (req, res) => {
  const { url } = req.body
  // TODO 校验、TODO 生成短码、TODO 存库……全写这？
  const code = Math.random().toString(36).slice(2, 8)
  res.status(201).json({ code, url })
})

app.use((req, res) => res.status(404).json({ error: 'Not Found' }))
app.listen(3000)

只有一个接口时它没问题。但随着功能变多，会撞上四类真实麻烦：

  • 文件爆炸：一个文件几百上千行，找一条路由要翻半天。
  • 职责混乱：「解析参数」「写业务规则」「拼数据库语句」「拼 HTTP 响应」全揉在一个回调里，改一处怕碰坏另一处。
  • 无法复用：「生成短码」这段逻辑，定时任务、CLI 脚本、单元测试都想用，但它被 req/res 绑死在路由里，拿不出来。
  • 难以测试：想测「生成短码是否 6 位」，却被迫先伪造一个 HTTP 请求——本末倒置。

解决思路是两步走，本章按顺序攻破：(A) 横向拆——把同一类路由（比如所有 links 接口）抽成一个独立路由模块；(B) 纵向拆——把单个接口按职责切成 route / controller / service 三层。

💡 提醒：从 Lesson 0010 起项目会迁到 TypeScript。本章先用你已经熟悉的 ESM JavaScript（import/export） 把结构思想讲透；结构一旦学会，换成 .ts 只是加类型标注，骨架完全一样。

2横向拆：express.Router() 是什么

定义（后端新概念）：官方说 express.Router() 创建出的 router 对象，是「一个由中间件和路由组成的实例」，你可以把它当成一个「迷你 App（mini-application）」——它能做的事就两样：跑中间件、做路由匹配12。

类比：把主 app 想成公司总机，每个 Router 是一个部门分机组。总机不需要知道「市场部内部怎么转接」，只要把「拨 1 字头的电话」整体甩给市场部那台分机即可。同理，主 App 只需说一句「凡是 /api/links 开头的请求，全交给 links 路由处理」，至于 links 内部有多少条子路由，主文件一概不管。

关键在于：Router 本身就是一个中间件，所以它能被 app.use() 挂载2。官方 birds 示例的精髓如下：

birds.js（一个路由模块）

import express from 'express'
const router = express.Router()

// 这个中间件只对本路由生效
router.use((req, res, next) => {
  console.log('Time:', Date.now())
  next()
})

router.get('/', (req, res) =>
  res.send('Birds home page'))
router.get('/about', (req, res) =>
  res.send('About birds'))

export default router

app.js（主文件挂载它）

import express from 'express'
import birds from './birds.js'

const app = express()

// 挂在 /birds 前缀下
app.use('/birds', birds)

app.listen(3000)

挂载后，App 会处理 /birds 和 /birds/about 两个地址1。注意一个极易踩的坑👇

🌍 实战：前缀只写一次，子路由写「相对路径」。 上面 router.get('/about', …) 写的是 '/about'，不是 '/birds/about'。因为 app.use('/birds', birds) 已经把 /birds 这段前缀「吃掉」了——官方原话：挂载路径会在请求传给该中间件前被剥离（stripped）2。新手常犯的错是在子路由里把前缀又写一遍，结果接口变成 /birds/birds/about 死活访问不到。口诀：前缀归 app.use 管，路径细节归 router 管。

把它套到 LinkPulse

我们把所有 links 接口搬进一个 linksRouter，主文件只剩一行挂载：

// routes/links.routes.js
import express from 'express'
const router = express.Router()

router.post('/', (req, res) => { /* 创建短链 */ })
router.get('/',  (req, res) => { /* 列表    */ })
router.get('/:code', (req, res) => { /* 查详情 */ })

export default router
// express-server.js（主文件，瞬间清爽）
import express from 'express'
import linksRouter from './routes/links.routes.js'

const app = express()
app.use(express.json())

app.use('/api/links', linksRouter)   // ← 一行接管所有 links 接口
app.use((req, res) => res.status(404).json({ error: 'Not Found' }))
app.listen(3000)

💡 router.post('/', …) 配合 app.use('/api/links', …)，最终命中的就是 POST /api/links；router.get('/:code', …) 就是 GET /api/links/:code。前缀 + 相对路径 = 完整路径。

3纵向拆：route → controller → service 三层

光把路由挪个文件还不够——router.post('/', …) 的回调里如果继续塞「校验 + 生成短码 + 存库 + 拼响应」，那只是把意大利面条换了个盘子。真正的功夫是分层（Layered Architecture）：把一个请求的处理过程，按「关注点」切成三段独立的代码。

route 路由层 只声明「什么方法 + 什么路径 → 交给哪个 controller」。认路的
▼
controller 控制器 只跟 HTTP 打交道：从 req 取数据、调 service、把结果塞进 res。接待员
▼
service 服务层 纯业务逻辑，完全不碰 req/res，只接收普通参数、返回普通值/抛错。真正干活的

每一层到底负责什么、不负责什么——这张表是本章的核心，建议记牢：

层它该做它绝不做
route 定义路径与 HTTP 方法，把请求路由到 controller 函数 不写业务、不读 req.body、不发响应
controller 从 req 取参数 → 调 service → 决定状态码 → res.json() 返回 不写核心业务规则（短码怎么生成、怎么算重复）
service 核心业务：生成短码、校验唯一性、（以后）读写数据库 不引入 express、不碰 req/res、不关心 HTTP 状态码

类比：餐厅。 route 是门口的指示牌（「点餐请往里走」）；controller 是服务员——接你的单、把菜端给你，但不下厨；service 是后厨——只管按菜谱做菜，根本不知道前厅长什么样、客人坐几号桌。后厨（service）不依赖前厅（HTTP），所以同一个后厨可以同时给堂食、外卖、自助餐厅供菜。

🌍 实战金句：薄控制器，厚服务（thin controller, fat service）。 业界公认的好结构是——controller 越薄越好，理想状态只有「取参数 → 调 service → 返回」三行；真正的重量级逻辑全沉到 service 里34。判断标准很简单：如果哪天你想从 Express 换成别的框架（Fastify、Koa），service 层应该一行都不用改——因为它根本不知道 Express 的存在。controller 才是那层「HTTP 适配器」。

4为什么值得这么麻烦：关注点分离（SoC）

「关注点分离（Separation of Concerns, SoC）」是软件工程的一条基本原则：把因不同原因而变化的代码分开放。HTTP 协议怎么收发（controller 关注）和「短码业务规则」（service 关注）是两件会各自独立演进的事——它们凭什么要纠缠在同一个函数里？拆开后你立刻收获三样东西：

  • 可测试（testable）：service 是个普通函数，输入参数、输出结果。测「生成的短码是 6 位」直接 createLink({url}) 调一下断言即可，不用启动服务器、不用伪造 HTTP 请求。这是分层最大的红利3。
  • 可维护（maintainable）：改业务规则只动 service，改响应格式/状态码只动 controller，改路径只动 route。改动半径小 = 不容易误伤。
  • 可复用（reusable）：同一个 createLink()，HTTP 接口能用，「批量导入 CSV 的脚本」「定时清理任务」也能用——因为它不绑 HTTP4。
🌍 实战：真实 Node 项目的目录长这样。 几乎所有成熟的 Express 项目都会有这套分层目录，新人一看就懂去哪找代码：
src/ ├─ express-server.js # 创建 app、挂中间件、挂各路由 ├─ routes/ │ └─ links.routes.js # 只声明 /api/links 下的路径 ├─ controllers/ │ └─ links.controller.js# 取 req、调 service、发 res └─ services/ └─ links.service.js # 纯业务，不依赖 express ★最值钱
关键信号：services/ 里的文件不应该出现 import express。哪天你在 service 里看到 req 或 res，就是分层漏水了，得堵回去。

5动手重构 LinkPulse：三个文件落地

下面把 links 完整拆成三层。从下往上看（先看最纯的 service，再看 controller，最后看 route），最符合数据流向的理解顺序。

① service —— 纯业务，零 Express

// src/services/links.service.js
// 注意：整个文件没有 import express，也没有 req / res
const store = new Map()   // 临时内存「数据库」，第 11 课换成真 DB

function generateCode() {
  return Math.random().toString(36).slice(2, 8)  // 6 位随机短码
}

// 输入普通对象，返回普通对象 —— 像数学函数一样可预测
export function createLink({ url }) {
  const code = generateCode()
  const link = { code, url, createdAt: new Date().toISOString() }
  store.set(code, link)
  return link
}

export function findByCode(code) {
  return store.get(code) ?? null
}

export function listLinks() {
  return [...store.values()]
}

② controller —— 只做 HTTP 翻译

// src/controllers/links.controller.js
import * as linksService from '../services/links.service.js'

export function create(req, res) {
  const { url } = req.body              // 1) 从 HTTP 请求取数据
  const link = linksService.createLink({ url })  // 2) 调 service 干活
  res.status(201).json(link)            // 3) 把结果翻译成 HTTP 响应
}

export function getOne(req, res) {
  const link = linksService.findByCode(req.params.code)
  if (!link) return res.status(404).json({ error: 'Link not found' })
  res.json(link)
}

export function list(req, res) {
  res.json(linksService.listLinks())
}

💡 看见没？每个 controller 几乎都是「取参数 → 调 service → 发响应」三步——这就是薄控制器。它只懂 HTTP（状态码 201/404、req.body、req.params），不懂短码怎么生成。

③ route —— 只声明路径

// src/routes/links.routes.js
import express from 'express'
import * as linksController from '../controllers/links.controller.js'

const router = express.Router()

router.post('/',      linksController.create)   // POST /api/links
router.get('/',       linksController.list)     // GET  /api/links
router.get('/:code',  linksController.getOne)   // GET  /api/links/:code

export default router

这里有个容易忽略的细节：传给 router.post('/', linksController.create) 的是 函数本身（create，不带括号），不是 create()。我们把函数交给 Express，等请求真正来了，Express 才会用 (req, res) 调用它。写成 create() 会立刻执行、传错东西。

④ 主文件 —— 组装

// src/express-server.js
import express from 'express'
import linksRouter from './routes/links.routes.js'

const app = express()
app.use(express.json())                 // 全局：解析 JSON 请求体
app.use('/api/links', linksRouter)      // 挂载 links 模块
app.use((req, res) => res.status(404).json({ error: 'Not Found' }))

app.listen(3000, () => console.log('LinkPulse on :3000'))

整条调用链一目了然——一个 POST 请求是这样流过三层的：

步骤发生了什么在哪一层
1POST /api/links 进来，express.json() 解析出 req.body主文件中间件
2app.use('/api/links', …) 剥掉前缀，把 / 交给 linksRouterroute
3router.post('/', …) 命中，调用 create(req, res)route → controller
4controller 取出 req.body.url，调 createLink({url})controller → service
5service 生成短码、存进 Map、return 新链接对象service
6controller 拿到对象，res.status(201).json(link) 返回controller
💡 你可能注意到 controller 里还没做 url 校验（空值、非法格式）。那是上一课（Lesson 0007 校验与错误处理）的主题——通常以中间件形式插在 controller之前，或在 service 里抛业务错误。本章先专注「结构」，校验逻辑你已经会写，正好回填进这套分层里。

🧠 小检验 分层架构

在 route / controller / service 三层结构里，下面哪段代码绝对不应该出现在 service 层？

⌨️ 动手：把 links 拆成 routes + controller + service 三个文件

  1. 在项目里新建目录：src/routes、src/controllers、src/services。
  2. 创建 src/services/links.service.js，写 createLink、findByCode、listLinks 三个函数，用一个 Map 当临时存储。确认整个文件没有 import express。
  3. 创建 src/controllers/links.controller.js，写 create / getOne / list 三个函数，import * as linksService 来调用上一步。
  4. 创建 src/routes/links.routes.js，用 express.Router() 把三个路径接到对应 controller 函数，export default router。
  5. 改造主文件 express-server.js：删掉原来那条 app.post('/api/links', …)，换成一行 app.use('/api/links', linksRouter)。
  6. 用 npm run dev 启动，挨个验证：
    • curl -X POST localhost:3000/api/links -H "Content-Type: application/json" -d '{"url":"https://example.com"}' → 应返回 201 + 含 code 的 JSON
    • 把返回的 code 填进 curl localhost:3000/api/links/那个code → 应返回该链接
    • curl localhost:3000/api/links → 应返回数组
    • curl localhost:3000/api/links/nope → 应返回 404
  7. 体会复用：临时在主文件最上方加一行 import * as svc from './services/links.service.js' 再 console.log(svc.createLink({url:'https://test.com'}))，启动看终端——你没经过任何 HTTP 就用上了业务逻辑。验证完记得删掉这行。
