用 Prisma 实现真正的短链 CRUD（替换假短码）

为什么学这个？ 到目前为止，LinkPulse 的 POST /api/links 只是把请求体读出来、拼一个假短码返回——刷新一下就什么都没了。本章是 LinkPulse 第一次真正把数据存进数据库：我们用 0014 接好的 Prisma Client，在 0008 的分层结构里写一个真正的 links service，落库一条 { code, longUrl, userId } 并返回 201；再补齐列表 / 查单条 / 删除，组成完整 CRUD。过程中你会解决三个真实后端工程师每天都在处理的问题：怎么生成不会撞、又不泄露内部信息的短码、万一短码真撞了怎么办、数据库报错如何优雅地变成 HTTP 响应。学完这一章，LinkPulse 就从「玩具 demo」迈进了「能持久化的真应用」。

1CRUD 是什么？把它对上 LinkPulse 的四个接口

CRUD 是后端最基础的四个动作：Create（增）、Read（查）、Update（改）、Delete（删）。几乎所有「管理某种资源」的接口，本质都是这四件事的组合。对前端来说你早就熟悉它对应的 HTTP 方法——这里只快速对一下后端语境：

动作HTTP（你已熟）Prisma 方法LinkPulse 接口

所谓「把假短码换成真 CRUD」，就是把每个接口里那行「假装干活」的代码，换成对应的 Prisma 方法真正读写 Postgres。整章我们都在做这一件事。

2短码怎么生成？为什么不能直接用自增 id

短链的核心就是那段 code（比如 linkpulse.io/aK3xY9）。它必须满足两点：唯一（两条短链不能撞同一个 code，否则跳转就乱了），并且短、URL 友好（只含 URL 里安全的字符）。直觉上你可能想：数据库不是有自增主键 id = 1, 2, 3... 吗，把它转成短字符串不就行了？

这条路（自增 id → base62）确实可行，但把数据库自增 id 暴露给外部是后端的一个经典坏味道，原因值得讲透：

  • 可枚举（enumeration）= 信息泄露。 如果 /1、/2、/3 都能访问，任何人写个循环就能遍历你全站所有短链，顺带推断出「你一共创建了多少条」「增长有多快」——对一个收费 SaaS 来说这是商业机密。
  • 可预测 = 安全隐患。 自增意味着「下一个 id」是可猜的。
  • id 应当是内部实现细节。 主键怎么生成是数据库的事，不该和「对外的公开标识」绑死。今天用 Postgres 自增，明天分库分表想换雪花 id，外部 URL 不该跟着变。

所以业界的主流做法是：对外标识用一段随机生成的不透明字符串，和内部自增 id 分开。我们用 nanoid 来生成它。

nanoid 是什么

nanoid 是一个极小（官方称约 118 字节）、安全、URL 友好的随机字符串 ID 生成器。默认调用 nanoid() 返回 21 个字符，用的字符集是 URL 友好的 A-Za-z0-9_-；官方明确这个长度的碰撞概率与 UUID v4 相当——「要有十亿分之一的重复概率，需要生成约 103 万亿个 ID」3。

import { nanoid } from 'nanoid'

nanoid()    //=> "V1StGXR8_Z5jdHi6B-myT"  （21 字符，默认）
nanoid(10)  //=> "IRFa-VaY2b"             （传 size 控制长度）

对短链来说 21 个字符太长了——我们想要 linkpulse.io/aK3xY9 这种 6～8 位的短码。这就用到 nanoid 的 customAlphabet(alphabet, size)：自定义字符集和默认长度，它返回一个新的生成函数3。

import { customAlphabet } from 'nanoid'

// 自定义字母表（去掉容易看错的 0/O、1/l/I），默认长度 7
const generateCode = customAlphabet(
  '23456789abcdefghjkmnpqrstuvwxyzABCDEFGHJKLMNPQRSTUVWXYZ',
  7
)

generateCode()  //=> "aK3xY9p"
generateCode(8) //=> 也可临时覆盖成 8 位
🌍 实战：短码生成策略，随机 vs 自增转 base62。 两条路线各有取舍——
  • 随机（nanoid，我们的选择）：无状态、不可枚举、不泄露规模、天然适合分布式（多台机器同时生成不打架）。代价是理论上可能撞（概率极低但非零），所以必须配合下一节的「唯一约束 + 冲突处理」。
  • 自增转 base62：短码最短、保证不撞、还能按时间排序。代价是可枚举、需要一个全局递增源（分布式下要发号器），且暴露增长数据。很多大厂内部用变体（如先自增再用「加盐 + 可逆置换」打乱顺序），工程复杂度高。
对一个刚起步、要快速上线的 SaaS，随机短码 + 长度可调是性价比最高的默认选择。位数也是个增长杠杆：起步用 6～7 位足够，库里短链多了把默认长度调到 8 即可，碰撞概率指数级下降。
🌍 实战坑：nanoid 从 v5 起是「纯 ESM」。 官方在 v5 把 nanoid 改成了 ESM-only，CommonJS 项目用 require('nanoid') 会直接报 ERR_REQUIRE_ESM4。对我们是好消息：LinkPulse 早在 0005 就是 "type":"module" 的 ESM 项目，所以 import { customAlphabet } from 'nanoid' 开箱即用，无需任何处理。但如果你将来在某个老的 CommonJS 项目里踩到这个错，记住两条出路：要么把项目转成 ESM，要么把 nanoid 锁在 3.x（最后一个 CJS 版本）。

3把 schema 接上：在 Link 表里放一个唯一的 code

承接 0014，我们的 schema.prisma 里已经有了 User 和 Link。本章对 Link 只做一件关键的事：给 code 字段加 @unique。这一行是后面「冲突处理」的地基——它让数据库在底层强制「不可能存在两条相同 code」，是真正的最后防线（比应用层任何检查都可靠）。

// prisma/schema.prisma （在 0014 基础上确认这几行）
model Link {
  id        Int      @id @default(autoincrement())  // 内部主键，绝不对外暴露
  code      String   @unique                        // ← 对外短码，数据库级唯一
  longUrl   String
  userId    Int
  user      User     @relation(fields: [userId], references: [id])
  createdAt DateTime @default(now())

  @@index([userId])   // 按用户查列表会频繁用到，建个索引
}

改完 schema 别忘了像 0014 那样把它同步到数据库（迁移）：

npx prisma migrate dev --name add_unique_code
为什么 @unique 是「真正的」唯一保证？ 你可能想：那我在写入前先 findUnique 查一下 code 在不在不就行了？不行——这是经典的竞态条件（race condition）。 两个请求可能在「同一瞬间」都查到「code 不存在」，然后都去 create，结果撞车。只有数据库的 UNIQUE 约束是原子的：并发写入时它保证只有一个成功、另一个报错。所以正确姿势永远是直接写，撞了再处理（下一节），而不是「先查再写」。

4核心：用 Prisma Client 在 service 层实现 CRUD

现在进入主菜。我们把 LinkPulse 的 links 业务逻辑集中写在 src/services/links.service.ts。先认识本章要用的 Prisma Client 方法，全部来自官方 CRUD 文档1：

方法作用关键选项
create({ data })插入一条，返回新记录data 要写入的字段
findUnique({ where })按唯一字段查一条；查不到返回 nullwhere 必须是唯一字段（id/code）
findMany({ where, skip, take, orderBy, select })查多条（列表）skip/take 分页
update({ where, data })改一条；查不到抛 P2025where + data
delete({ where })删一条；查不到抛 P2025where

4.1 Create：真正落库 + 不泄露内部 id

这是本章最重要的一段。注意两点：用 select 明确选出要返回的字段（把内部 id 排除在外），以及生成 code 后调用 prisma.link.create。

// src/services/links.service.ts
import { customAlphabet } from 'nanoid'
import { prisma } from '../db.js'   // 0014 里导出的单例 PrismaClient

const generateCode = customAlphabet(
  '23456789abcdefghjkmnpqrstuvwxyzABCDEFGHJKLMNPQRSTUVWXYZ',
  7
)

// 对外返回时统一只暴露这些字段——内部 id 永不出现在响应里
const publicLinkSelect = {
  code: true,
  longUrl: true,
  createdAt: true,
} as const

export async function createLink(input: { longUrl: string; userId: number }) {
  const code = generateCode()
  return prisma.link.create({
    data: { code, longUrl: input.longUrl, userId: input.userId },
    select: publicLinkSelect,            // ← 关键：只返回安全字段
  })
}
🌍 实战：永远不要在响应里泄露内部 id（和其它内部字段）。 默认情况下 Prisma 的 create/findMany 会把整行都返回给你，包含自增 id、userId 这种内部字段。如果你直接 res.json(link)，就把第 2 节我们费力避免的「可枚举自增 id」又从响应里漏出去了。两种防漏手段：(1) 像上面那样用 select 白名单显式挑字段（推荐，最稳）；(2) 用 omit 反向排除。养成习惯：API 的「对外形状」由你主动决定，而不是默认把数据库表结构原样吐出去。

4.2 处理唯一约束冲突：捕获 P2002 并重试

随机短码极少撞，但「极少」不是「不会」。当 code 撞上已存在的值，@unique 约束会让 create 失败，Prisma 抛出一个 PrismaClientKnownRequestError，其 code 字段为 "P2002"（unique constraint failed），meta.target 会告诉你是哪个字段撞了2。正确策略：捕获 P2002 → 换个新 code 重试，重试几次仍失败才真正报错。

// 推荐用 Prisma 命名空间来拿错误类型（见下方实战说明）
import { Prisma } from '@prisma/client'

export async function createLink(
  input: { longUrl: string; userId: number },
  maxRetries = 5,
) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const code = generateCode()
    try {
      return await prisma.link.create({
        data: { code, longUrl: input.longUrl, userId: input.userId },
        select: publicLinkSelect,
      })
    } catch (err) {
      // 只有「code 撞了」才重试；其它错误直接往上抛
      if (
        err instanceof Prisma.PrismaClientKnownRequestError &&
        err.code === 'P2002'
      ) {
        continue   // 换个 code 再试一次
      }
      throw err
    }
  }
  throw new Error('生成唯一短码失败，请重试')   // 极端情况：连撞 5 次
}
🌍 实战：「乐观写入 + 捕获冲突重试」是处理唯一冲突的标准模式。 注意我们没有「先 findUnique 查一下再 create」——前面说过那会引入竞态、还多一次查询。正确做法是乐观地直接写，撞了（P2002）再重试，把判断交给数据库的原子约束。这个模式在后端无处不在：注册时 email 撞、生成订单号撞、幂等键撞……都是同一招。重试上限设个 5 次足矣：用 7 位、50+ 字符的字母表，连撞 5 次的概率小到可以忽略，真撞到说明该把短码加长了。
🌍 实战：错误类型从哪 import？ 老教程常写 import { PrismaClientKnownRequestError } from '@prisma/client'，但随着 Prisma 客户端生成方式演进，更稳妥的写法是通过 Prisma 命名空间访问：import { Prisma } from '@prisma/client'，再用 err instanceof Prisma.PrismaClientKnownRequestError5。如果你用了自定义 generator 输出目录，import 路径要相应改成你生成的 client 路径。判断时一定要先 instanceof 收窄类型，再读 .code——因为 catch 到的 err 是 unknown，不能直接当成功对象用（这正是 TypeScript 帮你避免的坑）。

4.3 Read：列表（带分页）与查单条

列表接口几乎一定要分页——总不能一个用户有十万条短链就一次性全查出来。Prisma 用 skip（跳过几条）+ take（取几条）做偏移分页1，配合 where 只查「当前用户的」短链、orderBy 排序。

// 列出某用户的短链，第 page 页、每页 pageSize 条
export async function listLinks(opts: {
  userId: number
  page?: number
  pageSize?: number
}) {
  const page = opts.page ?? 1
  const pageSize = opts.pageSize ?? 20
  return prisma.link.findMany({
    where: { userId: opts.userId },          // 只看自己的
    orderBy: { createdAt: 'desc' },          // 最新的在前
    skip: (page - 1) * pageSize,             // 跳过前面几页
    take: pageSize,                          // 本页取多少条
    select: publicLinkSelect,                // 同样不泄露 id
  })
}

// 查单条：findUnique 查不到会返回 null（不报错）
export async function getLinkByCode(code: string) {
  return prisma.link.findUnique({
    where: { code },
    select: publicLinkSelect,
  })
}
🌍 实战：分页用 skip/take 还是 cursor？ skip/take（偏移分页）写起来最直观，适合「跳到第 N 页」的后台管理界面，是 LinkPulse 起步阶段的正确选择。但要知道它的短板：skip 越大数据库越慢（要先数过前面 N 条），而且翻页时若有新数据插入，会出现「重复或漏看」。当数据量很大、或做「无限下拉」时，业界会换成游标分页（cursor，基于 cursor + take）——按上一页最后一条的位置往后取，稳定且快。先用 skip/take 上线，等真有性能问题再升级，是务实的顺序。

4.4 Delete（与 Update）

delete 删一条，update 改一条。两者的 where 都要定位到具体记录；如果记录不存在，Prisma 会抛 P2025（依赖的记录未找到）2——这正好对应 HTTP 的 404 Not Found，下一节统一处理。

export async function deleteLink(code: string) {
  await prisma.link.delete({ where: { code } })
  // 删除成功通常不返回 body，路由层回 204 No Content
}

// 延伸：改短链指向的长链接
export async function updateLink(code: string, data: { longUrl: string }) {
  return prisma.link.update({
    where: { code },
    data,
    select: publicLinkSelect,
  })
}

5把它们串起来：Zod 校验 + 路由 + 错误处理中间件

service 写好了，现在回到 router 层，把 0007 的 Zod 校验、本章的 service、async/await + try/catch 串成完整链路。

5.1 用 Zod 守住入口（复习 0007）

快速复习：Zod 是运行时校验库，定义一个 schema，用 .parse() 校验数据；不合法会抛 ZodError。短链最该校验的就是 longUrl 必须是合法 URL——否则我们会把垃圾存进库。

// src/schemas/link.schema.ts
import { z } from 'zod'

export const createLinkSchema = z.object({
  longUrl: z.string().url('longUrl 必须是合法的 URL'),
})
export type CreateLinkInput = z.infer<typeof createLinkSchema>

5.2 路由：薄薄一层，把活交给 service

// src/routes/links.routes.ts
import { Router } from 'express'
import { createLinkSchema } from '../schemas/link.schema.js'
import * as linksService from '../services/links.service.js'

export const linksRouter = Router()

// CREATE → 201
linksRouter.post('/', async (req, res, next) => {
  try {
    const { longUrl } = createLinkSchema.parse(req.body)   // 校验
    // userId 暂时写死；真实鉴权会在后续章节接入
    const link = await linksService.createLink({ longUrl, userId: 1 })
    res.status(201).json(link)
  } catch (err) {
    next(err)            // ← 关键：错误交给下面的错误处理中间件
  }
})

// LIST → 200
linksRouter.get('/', async (req, res, next) => {
  try {
    const page = Number(req.query.page) || 1
    const links = await linksService.listLinks({ userId: 1, page })
    res.json(links)
  } catch (err) { next(err) }
})

// READ ONE → 200 / 404
linksRouter.get('/:code', async (req, res, next) => {
  try {
    const link = await linksService.getLinkByCode(req.params.code)
    if (!link) return res.status(404).json({ error: '短链不存在' })
    res.json(link)
  } catch (err) { next(err) }
})

// DELETE → 204
linksRouter.delete('/:code', async (req, res, next) => {
  try {
    await linksService.deleteLink(req.params.code)
    res.status(204).end()
  } catch (err) { next(err) }
})

5.3 async/await + try/catch 与错误处理中间件的配合

这是很多人没讲透的一环。Express 4 的路由处理器如果是 async 函数，里面 await 的 Promise 一旦 reject，Express 不会自动捕获——它不认识被拒绝的 Promise，请求会挂起。所以每个 async 路由都要 try/catch，在 catch 里调 next(err) 把错误手动「丢」给 Express 的错误处理中间件（你在 0006 见过中间件，错误中间件就是带 4 个参数 (err, req, res, next) 的那种特殊中间件）。

// src/middleware/error.ts —— 注意是 4 个参数！Express 靠这个识别它是错误中间件
import type { ErrorRequestHandler } from 'express'
import { ZodError } from 'zod'
import { Prisma } from '@prisma/client'

export const errorHandler: ErrorRequestHandler = (err, _req, res, _next) => {
  // 1) 校验失败 → 400
  if (err instanceof ZodError) {
    return res.status(400).json({ error: '请求参数有误', details: err.issues })
  }
  // 2) Prisma 已知错误 → 按 code 映射成合适的 HTTP 状态
  if (err instanceof Prisma.PrismaClientKnownRequestError) {
    if (err.code === 'P2025') {                 // 记录不存在（删/改时）
      return res.status(404).json({ error: '短链不存在' })
    }
    if (err.code === 'P2002') {                 // 唯一冲突兜底
      return res.status(409).json({ error: '资源已存在' })
    }
  }
  // 3) 其它没预料到的错误 → 500，且不把内部细节泄露给客户端
  console.error(err)                            // 内部日志留全量
  res.status(500).json({ error: '服务器内部错误' })
}
// src/app.ts —— 装配顺序很重要
import express from 'express'
import { linksRouter } from './routes/links.routes.js'
import { errorHandler } from './middleware/error.js'

const app = express()
app.use(express.json())              // 解析 JSON body（0006）
app.use('/api/links', linksRouter)   // 业务路由
app.use(errorHandler)                // ← 错误中间件必须放在最后

export { app }
为什么错误中间件放最后？ Express 按 app.use 注册顺序逐个匹配中间件。错误中间件要能接住前面所有路由抛出（经 next(err) 传来）的错误，所以必须注册在它们之后。它是整个应用的「错误兜底总闸」——好处是：业务代码只管 throw / next(err)，把「错误长什么样的 HTTP 响应」这件事收口到一个地方，再也不用在每个路由里重复写 try/catch 拼 JSON。
🌍 实战：500 错误绝不能把原始报错原文返回给客户端。 上面第 3 条特意只回一句「服务器内部错误」，而把真实 err 打到 console.error。原因：数据库报错、堆栈信息里常含表名、字段名、SQL 片段甚至连接串线索——直接 res.json(err) 等于把内部结构送给攻击者侦察。规则：对内日志写全，对外响应只给「能安全公开、对用户有用」的信息。 已知的、可控的错误（Zod 400、Prisma P2025 404）才给具体提示；未知错误一律 500 + 通用文案。

6跑通：curl 验证 + Prisma Studio 眼见为实

启动 npm run dev，然后用 curl 走一遍完整 CRUD。重点是建完之后去数据库里确认那行真的存在——这是 LinkPulse 第一次持久化，值得亲眼看一次。

# 1) 创建一条短链（注意：longUrl 必须是合法 URL，否则被 Zod 挡下返回 400）
curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" \
  -d '{"longUrl":"https://example.com/a/very/long/path?x=1"}'
# 期望：HTTP/1.1 201 Created
# {"code":"aK3xY9p","longUrl":"https://example.com/...","createdAt":"..."}
# ← 注意响应里【没有 id、没有 userId】

# 2) 列出（分页）
curl -s "http://localhost:3000/api/links?page=1"

# 3) 查单条（把 aK3xY9p 换成上面真实返回的 code）
curl -s http://localhost:3000/api/links/aK3xY9p

# 4) 删除 → 期望 204 No Content（无 body）
curl -i -X DELETE http://localhost:3000/api/links/aK3xY9p

# 5) 再查已删的 → 期望 404
curl -i http://localhost:3000/api/links/aK3xY9p

# 6) 故意发非法 url → 期望 400（Zod 拦截）
curl -i -X POST http://localhost:3000/api/links \
  -H "Content-Type: application/json" -d '{"longUrl":"not-a-url"}'

最后用 Prisma Studio（0014 装过）打开可视化界面，直接看 Link 表里那一行真实数据：

npx prisma studio   # 浏览器打开 http://localhost:5555，点 Link 表查看
看到数据了 = 里程碑达成。 在 Prisma Studio 里你能看到完整的行：id（内部自增）、code（对外短码）、longUrl、userId、createdAt。对照一下：数据库里有 id，但你的 API 响应里没有 id——这正是第 4.1 节 select 白名单的功劳。从这一刻起，LinkPulse 不再是「刷新就丢」的玩具了。

🧠 小检验 P2002 / 唯一冲突

随机生成的短码 code 字段加了 @unique。新建短链时，处理「短码可能撞车」最稳妥的做法是？

⌨️ 动手：把假短码换成真 CRUD，并落进 Postgres

  1. 装 nanoid：在项目根目录 npm i nanoid（ESM 项目直接可用）。
  2. 确认 schema：在 prisma/schema.prisma 的 Link 模型上确保 code String @unique 与 @@index([userId])，然后 npx prisma migrate dev --name add_unique_code。
  3. 写 service：新建 src/services/links.service.ts，用 customAlphabet 生成 7 位短码；实现 createLink（含 P2002 重试 + select 白名单不返回 id）、listLinks（skip/take 分页）、getLinkByCode、deleteLink。
  4. 改路由：把原来返回「假短码」的 POST /api/links 改成调 createLink 并回 201；补 GET /api/links（列表）、GET /api/links/:code（单条，查不到回 404）、DELETE /api/links/:code（回 204）。每个 async 路由都 try/catch + next(err)。
  5. 挂错误中间件：新建 src/middleware/error.ts（4 参数版），映射 ZodError→400、P2025→404、P2002→409、其它→500；在 app.ts 里 最后 app.use(errorHandler)。
