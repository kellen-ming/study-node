数据建模：为 LinkPulse 设计表结构

为什么学这个？ 上一课你学会了用 SQL 对单张表做增删改查。但真实的 LinkPulse 不只有一张表——它有用户、有用户创建的短链、还有每条短链被点击的访问记录，三者环环相扣。本章不写多少代码，而是做一件后端工程师上线前必做、且最容易被新手跳过的事：数据建模（data modeling）。我们会用「实体—关系」的思维，把 LinkPulse 拆成 users / links / clicks 三张表，定义清楚每个字段的类型、主键、外键和它们之间的关系。这份设计就是下一课用 Prisma 落地的「蓝图」——蓝图错了，后面写再多代码都是返工。

1数据建模是什么？为什么先建模再写代码

数据建模（data modeling），就是在动手写表、写代码之前，先想清楚：我的系统里到底有哪些「东西」（实体），每个东西要记录哪些信息（字段），这些东西之间是什么关系（一个用户能有几条短链？）。然后把这套想法画成图、写成表结构。它是「需求」和「数据库表」之间的一座桥。

类比：盖房子。你不会买来砖头就直接砌墙——你会先画施工图纸：几个房间、承重墙在哪、水电管线怎么走。数据建模就是数据库的施工图纸。前端同学更熟悉的类比：你在写 React 组件前，往往会先设计 props 和 state 的形状（{ user: {...}, links: [...] }）；数据建模就是把这种「形状设计」做在数据库这一层，而且要考虑得更严谨，因为它要被长期持久化、被很多功能共享。

为什么必须「先建模」？这要从前后端一个关键差异说起：

  • 前端代码改起来便宜。一个组件写错了，删掉重写，刷新页面就好，没有「历史包袱」。
  • 数据库的表结构改起来昂贵。一旦上线，表里就装满了真实用户的真实数据。这时你想加一列、改一个类型、把一张表拆成两张，都得在「不丢数据、不停服务」的前提下小心翼翼地迁移。字段名起错了、关系设计错了，可能要拖着这个错误跑很久。
🌍 实战：建模决定后续一切，改表代价大。 这是本章最重要的一句话。后端老手有个共识：代码可以重构，数据模型很难重构。一个真实例子：早期很多产品把「用户名」直接当主键存（username 当 PK，到处用它做外键）。后来要支持「改用户名」功能时才发现——主键被几十张表引用，根本动不了，最后被迫加一列数字 id 做真正的主键、再花几个版本慢慢迁移。建模阶段多花一小时思考，能省下上线后几周的迁移噩梦。这也是为什么我们这一整章都在「想」，下一章才动手「写」。
建模产物长什么样？ 通常是两样东西：① 一张 ER 图（实体关系图），画出有哪些表、表之间怎么连；② 一份字段清单，列出每张表的列名、类型、约束。本章结束时，你会为 LinkPulse 产出这两样东西。

2实体与关系：用 ER 思维拆解 LinkPulse

建模的核心工具叫 ER 模型（Entity-Relationship Model，实体—关系模型）。它把世界看成两种成分：

  • 实体（Entity）：系统里独立存在的「东西」，通常对应一张表。LinkPulse 里的实体：用户、短链、点击记录。
  • 属性（Attribute）：实体身上要记录的信息，对应表里的列（字段）。比如「短链」有「目标 URL」「短码」「创建时间」。
  • 关系（Relationship）：实体之间的关联。这是 ER 模型的精华，分三种基数（cardinality）。

三种关系基数

① 一对一（1—1）：A 的一条最多对应 B 的一条。例如「用户」和「用户的实名认证信息」。LinkPulse 暂时用不到。

② 一对多（1—N）：A 的一条对应 B 的多条，但 B 的一条只属于一个 A。这是最常见的关系。Prisma 官方的定义是：「一对多（1-n）关系把一侧的一条记录连接到另一侧的零条或多条记录。」3 比如「一个用户拥有零到多条短链，而每条短链只属于一个用户」。

③ 多对多（M—N）：双方都能对应对方的多条。Prisma 官方定义：「多对多（m-n）关系把一侧的零条或多条记录连接到另一侧的零条或多条记录。」3 经典例子是「文章」和「标签」：一篇文章可有多个标签，一个标签也可贴到多篇文章上。

怎么判断是 1—N 还是 M—N？ 念两句话，两边都念：「一个 ___ 能有几个 ___？」和「一个 ___ 又能属于几个 ___？」。两边都是「多」→ M—N；一边「多」一边「一」→ 1—N。例：「一个用户能有多条短链」+「一条短链只属于一个用户」→ 1—N。

把 LinkPulse 的关系念一遍：

users

id用户
1 ── N拥有

links

id短链
1 ── N产生

clicks

id点击
  • 「一个 user 能创建多条 link，一条 link 只属于一个 user」→ user 1—N link。
  • 「一条 link 能被点击多次（产生多条 click），一条 click 只来自一条 link」→ link 1—N click。

所以 LinkPulse 是一条「链式 1—N」结构。它不需要多对多，建模因此相对干净——这是 SaaS 早期常见的形态。

🌍 实战：先画 ER 图，再开会，再写代码。 在真实团队里，ER 图是和产品经理对齐需求的语言。当 PM 说「我想看每条短链的访问来源」，工程师立刻在 ER 图上指出：这需要 clicks 表多一个「来源」字段——一张图就避免了「做完才发现没存来源」的悲剧。很多公司用 dbdiagram.io、Figma/FigJam 或直接在 Prisma schema 里画模型来沟通。建模不是一个人闷头做，是团队达成共识的过程。

3关系怎么落地：主键、外键与引用完整性

ER 图上画一条「1—N」的线很容易，但数据库里没有「线」这种东西，只有表和列。这条关系最终要靠两个机制变成真实可用的结构：主键和外键。

主键（Primary Key, PK）

PostgreSQL 官方定义：「主键约束（primary key）表示某一列或一组列可以作为表中各行的唯一标识符，这要求这些值既唯一又非空（NOT NULL）。」1 一张表最多只有一个主键。1

类比：主键就是每一行的「身份证号」。哪怕两个用户同名同姓，他们的 id 也绝不相同。前端里你给列表渲染时写的 key={item.id}，那个 id 的灵魂就来自数据库主键。

外键（Foreign Key, FK）—— 关系的真身

PostgreSQL 官方定义：「外键约束（foreign key）规定某一列（或一组列）的值必须匹配另一张表某一行中出现的值。我们说这维护了两张相关表之间的引用完整性（referential integrity）。」1

说人话：外键就是「我属于谁」的那一列。在 1—N 关系里，外键放在「多」的那一侧。比如 links 表里放一列 user_id，它的值必须是 users 表里真实存在的某个 id。这一列就把「这条短链属于哪个用户」钉死了。

❌ 没有外键概念（前端易犯）

// 想「关联」用户和短链，
// 却把整个用户对象塞进短链里
{
  shortCode: "ab12cd",
  url: "https://...",
  user: {            // 冗余！
    name: "Kellen",
    email: "k@x.com"
  }
}
// 用户改名 → 每条短链里
// 的副本都得改，必然不一致

✅ 用外键关联（后端正解）

// 短链只存「我属于谁」的 id
links:  { id, short_code, url,
          user_id: 7 }   // ← FK

users:  { id: 7, name: "Kellen",
          email: "k@x.com" }

// 用户改名只改 users 一行，
// 所有短链自动「看到」新名字
// 因为它们只存了 id，没存副本

引用完整性（referential integrity）为什么重要？ 它是数据库帮你守住的一条铁律：外键指向的那一行必须真实存在。如果 links.user_id = 7，数据库就保证 users 里一定有 id=7 这一行——你无法插入一条「属于不存在用户」的短链，这种「悬空」的脏数据从源头就被拦住了。

🌍 实战：删除时连带数据怎么办？（ON DELETE） 如果用户 id=7 注销了，他名下的短链该怎么处理？外键允许你声明删除行为，PostgreSQL 给了几种选项2：
• ON DELETE CASCADE：「当被引用的行被删除时，引用它的行也自动一并删除。」2 ——删用户，他的短链和点击记录跟着全删（适合「组成部分」关系）。
• ON DELETE RESTRICT：「阻止删除被引用的行。」2 ——只要用户还有短链，就不让删这个用户，逼你先处理干净。
• ON DELETE SET NULL：把引用列「设为 null」。2 ——短链变成「无主」状态保留下来。
LinkPulse 里，删用户连带删 links、删 link 连带删 clicks，用 CASCADE 最自然。但请记住：CASCADE 是把双刃剑——它能一键清理，也能因为一次误删而连环删掉海量数据。涉及钱、订单的表通常故意用 RESTRICT，强迫人工确认。
用什么类型做主键？ 两大流派：① 自增整数（PostgreSQL 的 BIGSERIAL / 标准 SQL 的 IDENTITY）——简短、有序、索引快，但会暴露规模（看到 /users/5 就知道你只有几个用户），且不便在多库间合并。② UUID（如 550e8400-...）——全局唯一、不暴露数量、适合分布式，但更长、随机插入对索引略不友好。SaaS 早期用自增整数完全够用，本章 LinkPulse 就用它；等你需要把 id 暴露到 URL 上时，可再考虑 UUID。

4规范化（Normalization）：把冗余和不一致挤出去

有了实体和关系，下一个问题是：每个字段到底该放在哪张表？这正是 规范化（normalization，也叫范式化） 要解决的。微软官方文档给的定义很精准：「规范化是组织数据库中数据的过程，它依照规则创建表、建立表间关系，目的是消除冗余（redundancy）和不一致依赖（inconsistent dependency），从而保护数据并让数据库更灵活。」4

两个要躲的坑：

  • 冗余数据（redundant data）：同一份信息存在多个地方。官方原话：「冗余数据浪费磁盘空间并制造维护问题。如果一份数据存在于多处，那么要修改它时，必须在所有位置以完全相同的方式修改。」4（回想第 3 节那个「把用户名抄进每条短链」的反例——那就是冗余。）
  • 不一致依赖（inconsistent dependency）：把不该放一起的信息硬塞进一张表。官方举例：去 Customers 表查某客户的地址很自然，但去那里查「负责该客户的员工的薪水」就不合理——薪水依赖的是员工，应该放进 Employees 表。4

规范化有一套层层递进的规则，每一层叫一个范式（Normal Form）。官方说：「满足前三条规则就称为第三范式（3NF）；虽然还有更高的范式，但对大多数应用，3NF 已是所需的最高级别。」4 所以我们只需要培养到 3NF 的直觉即可，够用。

范式一句话规则（官方表述）直觉 / LinkPulse 对照
1NF
第一范式		消除「重复组」，每列只放单个原子值；用主键标识每行。4别在 links 里搞 tag1, tag2, tag3 这种列；一格里也别塞 "a.com,b.com"。
2NF
第二范式		把「适用于多条记录的值集合」拆成单独的表，用外键关联；记录不应依赖主键之外的东西。4用户的 email、name 属于用户，不该在每条 click 里重复抄一遍——把它们留在 users 表，靠 user_id 关联。
3NF
第三范式		「消除不依赖于主键的字段。」4 非键字段必须直接依赖主键，不能依赖另一个非键字段（传递依赖）。在 links 表里别存「该短链所属用户的注册邮箱」——邮箱依赖的是 user_id（非键），不是 link 的主键，应留在 users。

3NF 的核心直觉，记一句话就够：表里每一个非主键字段，都应该「只描述这一行的主键所代表的那个东西」，不多也不少。微软文档把传递依赖讲得很形象：在一张 Students 表里同时塞「导师 Advisor」和「导师办公室 Adv-Room」，问题在于办公室其实依赖导师、而非学生，所以应把导师及其办公室拆到 Faculty 表去。4

🌍 实战：规范化 vs 适度反规范化（denormalization）。 规范化是默认起点，但不是越纯越好。微软官方专门给了「例外」提醒：「严格遵守 3NF 在理论上可取，但并非总是实际可行……太多小表可能拖慢性能，甚至超出文件/内存上限。」4 落到 LinkPulse 一个真实场景：仪表盘要显示「这条短链被点了多少次」。完全规范化的做法是每次都 COUNT(*) FROM clicks WHERE link_id=?——当一条爆款短链有几百万次点击时，每刷一次仪表盘都全表数一遍，很慢。适度反规范化的做法：在 links 表加一列 click_count 缓存这个数字，每次点击时 +1。这是故意制造的冗余——用「写入时多维护一个字段」换「读取时快百倍」。
权衡的本质：规范化优化「写」与「一致性」，反规范化优化「读」与「速度」。 原则是：先规范化，遇到实测的性能瓶颈再针对性反规范化，并清楚自己在用一致性风险换性能。
🌍 实战：命名规范，团队的隐形契约。 字段名一旦上线就很难改，所以一开始就要统一。主流后端约定：① 表名用复数小写（users、links），② 列名用 snake_case（created_at 而非 createdAt，因为很多 SQL 数据库对大小写不敏感，下划线更稳），③ 外键统一叫 <单数表名>_id（user_id、link_id），一眼就知道指向谁，④ 时间戳统一 created_at / updated_at，⑤ 布尔字段用 is_ 前缀（is_active）。命名一致，半年后接手的人（很可能就是你自己）才读得懂。（下一课 Prisma 里我们会在代码层用 camelCase，再用 @map 映射到数据库的 snake_case 列名，两全其美。）

5什么时候加索引（Index）？

这是建模时顺带要想、但常被新手忽略的一环。索引（index）是数据库为某一列额外维护的一份「目录」，让按这一列查找时不必扫全表。

类比：一本没有目录的书，要找某一章只能从第一页翻到最后一页（全表扫描，O(n)）；加了目录，翻到目录直接定位页码（索引查找，接近 O(log n)）。代价是：目录本身占纸张（占磁盘），而且书的内容一变（增删改），目录也得跟着更新（写入变慢一点点）。

好消息：主键和外键大多自带或强烈建议加索引。 PostgreSQL 官方指出，加主键会自动创建一个唯一 B-tree 索引。1 而外键列虽然不会自动建索引，却是最该手动加索引的地方——因为你几乎总会用它来做关联查询（「查这个用户的所有短链」= WHERE user_id = ?）。

🌍 实战：索引加在「你经常用来查找或关联的列」上。 判断口诀——这一列是否经常出现在 WHERE、JOIN、ORDER BY 里？是 → 考虑加索引。落到 LinkPulse：
• links.short_code：每次有人访问短链，都要 WHERE short_code = 'ab12cd' 查目标 URL，这是系统最高频的查询，必须加索引（而且它天然唯一，用 唯一索引/UNIQUE 约束，顺便保证不会撞码）。
• links.user_id、clicks.link_id：外键，频繁用于「查某用户的短链 / 某短链的点击」，加索引。
• clicks.created_at：仪表盘常按时间范围统计，可加索引。
反过来——别给每一列都加索引。 索引让读变快但写变慢、占更多空间。一张写入极频繁的 clicks 表如果加满索引，每次插入点击都要更新一堆索引，反而拖累。加索引的本质，是用「写慢一点、占空间」换「特定查询快很多」——只为真实高频的查询付这个代价。
本章只需建立「该加索引的直觉」，具体的 CREATE INDEX 语法和性能调优会在后续讲查询优化时深入。现在你只要在设计表时，能指出「哪几列将来要加索引」即可。

6迁移（Migration）：表结构也会演进，且必须可追踪

最后一个关键概念。你可能想：「既然建模这么重要，我一次设计到位不就行了？」——现实是不可能。需求会变：明天 LinkPulse 要加「付费订阅」，users 表就得加 plan 列；后天要支持「短链有效期」，links 就得加 expires_at。表结构（schema）会随产品不断演进。

迁移（migration），就是「对数据库结构做的一次有记录、可重放、可回溯的变更」。每一次「加一列、改类型、建一张新表」，都被写成一个带版本号的迁移文件，按顺序保存在代码仓库里。

类比：迁移之于数据库，就像 Git 提交之于代码。 你不会直接去线上服务器上手改代码——你写代码、git commit、可追溯、可在每台机器重放、出错能回滚。数据库结构同理：你不该登上生产库手敲 ALTER TABLE（改完没人知道改了啥、本地和线上还对不上），而是写一个迁移文件，让它在本地、测试、生产环境按同一顺序、同一套变更跑一遍。

没有迁移（手改）有迁移（工程化）
直接在数据库敲 ALTER TABLE，改完无记录每次变更生成一个版本化文件，进 Git 仓库
本地、线上结构悄悄不一致，「在我机器上是好的」所有环境按同一串迁移重放，结构必然一致
新同事拉下代码，建不出和你一样的库新人跑一条命令，迁移自动建出完整最新结构
改错了只能凭记忆手动改回去有清晰的变更历史，能定位、能回滚
🌍 实战：永远不要手改生产数据库结构。 这是后端铁律。真实事故模式：开发在本地 ALTER TABLE 加了列、功能跑通就上线了，但忘了在生产库执行同样的变更——结果新代码一上线就报「column does not exist」，全站 500。迁移机制就是为了根除这类「环境漂移（schema drift）」。下一课的 Prisma 自带强大的迁移系统（prisma migrate）：你只改 schema.prisma 这份「单一事实来源」，它自动生成 SQL 迁移文件并应用——你这一章设计的三张表，正是要喂给它的输入。所以现在把模型想清楚，下一课会异常顺滑。

图例：橙色 = 主键（PK） · 蓝色 = 外键（FK） · ──< 朝向「多」的一侧（鱼尾/crow's foot 记法）。读法：users 1 — N links 1 — N clicks。

users —— 用户

字段类型约束 / 说明
idBIGSERIAL主键。自增整数，每个用户的身份证号。
emailVARCHAR(255)UNIQUE + NOT NULL。登录账号，必须唯一（靠唯一约束兜底，防止两人注册同邮箱）。
password_hashVARCHAR(255)NOT NULL。存的是密码的哈希值，绝不存明文密码（哈希怎么做留待鉴权章节）。
planVARCHAR(20)订阅档位，如 'free' / 'pro'。默认 'free'。SaaS 收费的依据。
created_atTIMESTAMPTZ注册时间，默认 now()。带时区（TZ）更安全。

links —— 短链（属于某个 user）

字段类型约束 / 说明
idBIGSERIAL主键。
user_idBIGINT外键 → users.id，ON DELETE CASCADE。这一列就是「user 1—N link」关系的真身：删用户连带删其短链。建索引。
short_codeVARCHAR(16)UNIQUE + NOT NULL。短码，如 ab12cd。最高频查询列 → 唯一索引（既保证不撞码，又让访问跳转飞快）。
target_urlTEXTNOT NULL。原始长链接。用 TEXT 因 URL 可能很长，不必限长。
click_countBIGINT默认 0。适度反规范化的缓存列：点击数。属故意冗余，换仪表盘读取速度（见第 4 节）。
created_atTIMESTAMPTZ创建时间，默认 now()。

clicks —— 点击记录（属于某条 link）

字段类型约束 / 说明
idBIGSERIAL主键。
link_idBIGINT外键 → links.id，ON DELETE CASCADE。「link 1—N click」关系的真身：删短链连带删其点击。建索引。
clicked_atTIMESTAMPTZ点击发生时间，默认 now()。按时间出报表会常用 → 可建索引。
ip_addressINET访客 IP。PostgreSQL 有原生 INET 类型，比当字符串存更省、可按网段查询。（注意：IP 属个人数据，留意隐私合规。）
user_agentTEXT浏览器/设备标识，用于分析「来自手机还是桌面」。
referrerTEXT来源页（从哪个网站点进来的），分析流量来源。
这份设计满足 3NF 吗？ 基本满足：每列都是原子值（1NF）；每张表的非键列都只依赖自己的主键（2NF/3NF）——用户信息只在 users、短链信息只在 links、点击信息只在 clicks，没有把 email 抄进 clicks 这类传递依赖。唯一一处刻意的反规范化是 links.click_count——这是清醒的权衡，不是疏忽。这正是「先规范化、按需反规范化」的现实落地。

下面把这份设计写成可执行的 SQL（仅供你直观感受最终形态；下一课我们用 Prisma 来更优雅地表达它并自动生成迁移，不用手写这些）：

-- LinkPulse 数据模型（PostgreSQL 方言，预览用）
CREATE TABLE users (
  id            BIGSERIAL    PRIMARY KEY,
  email         VARCHAR(255) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  plan          VARCHAR(20)  NOT NULL DEFAULT 'free',
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE links (
  id          BIGSERIAL    PRIMARY KEY,
  user_id     BIGINT       NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  short_code  VARCHAR(16)  NOT NULL UNIQUE,   -- UNIQUE 自带唯一索引
  target_url  TEXT         NOT NULL,
  click_count BIGINT       NOT NULL DEFAULT 0, -- 适度反规范化的缓存列
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_links_user_id ON links(user_id);  -- 外键列手动加索引

CREATE TABLE clicks (
  id         BIGSERIAL   PRIMARY KEY,
  link_id    BIGINT      NOT NULL REFERENCES links(id) ON DELETE CASCADE,
  clicked_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ip_address INET,
  user_agent TEXT,
  referrer   TEXT
);
CREATE INDEX idx_clicks_link_id    ON clicks(link_id);
CREATE INDEX idx_clicks_clicked_at ON clicks(clicked_at);

🧠 小检验 实体关系 & 范式

在 LinkPulse 里，「一条短链能被点击很多次，而每条点击记录只来自一条短链」。这是什么关系？它在表结构里应该怎么落地？

⌨️ 动手：写出 LinkPulse 三张表的结构设计

本章不写运行代码，产出「设计文档」。在项目根目录新建一个文件 study-node/03-data/schema-design.md（先用 Markdown 表达想法，下一课再用 Prisma 落地），完成：

  1. 画 ER 关系：用一行文字或简单 ASCII 写出三张表的关系链，标明基数与外键方向，例如：users (1) ──< (N) links (1) ──< (N) clicks，并各注明「外键在哪一侧、叫什么名字」。
  2. 列字段清单：为 users / links / clicks 各做一张表格，列出字段名 + 类型 + 约束（PK / FK / UNIQUE / NOT NULL / DEFAULT）。可直接参考第 7 节，但请用自己的话把「为什么是这个类型/约束」写进备注。
  3. 标注主外键与索引：明确每张表的主键；每个外键写清楚指向哪张表的哪一列、用什么 ON DELETE 行为；并标出你打算加索引的列，每个都用一句话说明「因为它会高频出现在 WHERE / JOIN / ORDER BY」。
  4. 指出反规范化决策：找出你设计里故意冗余的字段（提示：click_count），写一句话说明你用什么换什么（一致性维护成本 ↔ 读取速度）。
  5. （加分）想一次演进：假设下周要支持「短链有效期」，你会给哪张表加什么字段？这个改动将来需要一次迁移（migration）——用一句话描述为什么不能直接上生产库手敲 ALTER TABLE。
