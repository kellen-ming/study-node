Lesson 0023 · Node.js 后端之路

自动化测试：单元、集成与测试金字塔

为什么学这个？ 你的 LinkPulse 已经能「创建短链 → 存库 → 跳转 → 缓存」，代码越来越多。下一步你要 Docker 化、上 CI、加认证和计费——每改一处都可能悄悄弄坏旧功能。自动化测试就是你重构和加功能时的安全网：一条命令 npm test 就能在几秒内告诉你「哪里坏了」。本章给 LinkPulse 的 LinkService 写单元测试（用 0019 的依赖注入 DI 注入假 repo，不碰数据库），给 POST /api/links 写 supertest 集成测试（真实 HTTP，覆盖 400 与 201）。

1为什么要写测试：从「手动点」到「机器跑」

作为前端，你大概体验过：改了个组件，本地点几下没问题就提交了，结果上线发现另一个页面崩了。后端的「悄悄弄坏」更隐蔽——一个被多处调用的 service 函数改了返回结构，可能影响三个接口，而你只测了一个。

测试解决的核心痛点有三个：

  • 防回归（regression）：「回归」指本来好的功能，被新改动弄坏了。一套测试在每次改动后自动重跑，坏了立刻红灯。
  • 重构有底气：你敢把一坨乱代码拆开重写，是因为有测试保证「行为没变」。没有测试的重构 = 闭眼开车。
  • 测试即文档：一份好测试用例，读起来就是「这个函数在什么输入下应该返回什么」，比注释更可靠（因为它会被执行验证）。
类比：手动测试像每次出门前挨个检查门窗水电；自动化测试像装了一套传感器，按一下面板就知道全屋状态。前者随项目变大而线性变慢，后者几乎恒定。
🌍 实战：不要追求「先把功能全做完再补测试」。真实团队里，测试是和代码一起长出来的——你写完 LinkService.createLink，顺手就写它的单测。等功能堆了几千行再补测试，你会发现很多代码根本没法测（耦合太死、依赖到处 new），反过来逼你回去改设计。这也是 0019 强调依赖注入的原因之一。

2测试金字塔：多少单元、多少集成、多少 E2E

测试金字塔（Test Pyramid）是 Mike Cohn 提出、Martin Fowler 在《The Practical Test Pyramid》中系统阐述的策略3。它回答一个工程问题：测试有很多种，每种各写多少才划算？

E2E / UI 测试（少量）启动整个系统，模拟真实用户全链路。慢、脆、贵
集成测试（适量）多个模块一起跑，比如打真实 HTTP 端点、连测试数据库
单元测试（大量）测单个函数/类的纯逻辑，毫秒级，隔离、稳定、便宜

越往上：越接近真实用户、越能发现集成问题，但越慢、越脆弱、越难定位。越往下：越快越稳越好定位，但离真实场景越远。Fowler 的核心论断是：「你应该有远多于高层测试的低层单元测试」3。

层级测什么LinkPulse 例子速度
单元单个函数纯逻辑，依赖全 mockLinkService.createLink 是否拒绝非法 URL、是否生成短码~ms
集成多模块串起来，真路由+真校验supertest 打 POST /api/links，验 400/201~10-100ms
E2E完整系统，真库真 Redis起服务 → 创建短链 → 真访问 /:code → 验 302~秒
🌍 实战：金字塔不是教条，是「投资回报」指南。本章给 LinkPulse 写的是金字塔下两层：大量单元测 LinkService 逻辑 + 几个集成测覆盖关键 HTTP 路径。E2E（起真库真 Redis 跑全链路）留到 0024 Docker 化、0025 CI 之后再补，因为那时你有了 docker compose 能一键拉起依赖环境。

3装 Vitest，写第一个单元测试

Vitest 是基于 Vite 的新一代测试框架，原生支持 TypeScript / ESM，开箱即用，API 与 Jest 高度兼容（describe / it / expect）1。对我们这种已全面 TS + ESM 的项目，它几乎零配置。

npm i -D vitest supertest @types/supertest

Vitest 默认会读你的 vite.config.*；没有也没关系，单独建一个 vitest.config.ts 即可，它优先级更高2：

// vitest.config.ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    globals: false,        // 我们显式 import，不污染全局
    environment: 'node',   // 后端测试跑在 node 环境
    include: ['src/**/*.test.ts'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
    },
  },
});

在 package.json 配置脚本（vitest run 跑一次后退出，适合 CI；不加 run 是 watch 模式，适合本地开发）1：

{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest",
    "test:cov": "vitest run --coverage"
  }
}

先尝个鲜——测一个纯函数。假设 LinkPulse 有个工具校验目标 URL：

src/utils/url.ts（被测代码）

export function isValidHttpUrl(raw: string): boolean {
  try {
    const u = new URL(raw);
    return u.protocol === 'http:' || u.protocol === 'https:';
  } catch {
    return false;
  }
}

src/utils/url.test.ts（测试）

import { describe, it, expect } from 'vitest';
import { isValidHttpUrl } from './url.js';

describe('isValidHttpUrl', () => {
  it('接受 https 链接', () => {
    expect(isValidHttpUrl('https://a.com')).toBe(true);
  });
  it('拒绝非 http 协议', () => {
    expect(isValidHttpUrl('ftp://a.com')).toBe(false);
  });
  it('拒绝乱字符串', () => {
    expect(isValidHttpUrl('not a url')).toBe(false);
  });
});

describe 把相关用例分组；it（等同 test）是一条用例；expect(x).toBe(y) 是断言。跑 npm test 就能看到三条绿勾。

ESM import 路径要带 .js：因为项目是 ESM（NodeNext），编译后引用的是 .js 文件，所以测试里 import 源码也写 ./url.js。Vitest 走 Vite 的解析，能正确处理这一点。

4mock 与 stub：让 LinkService 单测不碰数据库

单元测试要隔离：测 LinkService 的逻辑时，不该真去连 PostgreSQL。这就需要「替身」。两个常被混用但有区别的术语：

  • Stub（桩）：提供预设返回值，让被测代码能跑下去。关注「给出什么」。例：让假 repo 的 findByCode 固定返回 null，模拟「短码没被占用」。
  • Mock（模拟）：除了能返回值，还记录自己被怎么调用，供你断言。关注「有没有被正确地调用」。例：断言 repo.create 被调用了一次、且参数里的 shortCode 是 6 位。

Vitest 用 vi.fn() 创建 mock 函数（可记录调用、可设返回值），vi.mock() 替换整个模块，vi.spyOn() 监视已有方法4。但对我们最优雅的方式是依赖注入——回想 0019：LinkService 通过构造函数接收一个 repository，所以测试时直接注入一个假 repo 对象，连 vi.mock 都不需要。这正是 DI 让代码可测的价值。

src/links/link.service.ts（回顾 0019 的 DI 结构）

import { nanoid } from 'nanoid';
import { isValidHttpUrl } from '../utils/url.js';

// service 只依赖这个接口，不关心背后是 Prisma 还是别的
export interface LinkRepository {
  findByCode(code: string): Promise<{ id: string } | null>;
  create(data: { shortCode: string; targetUrl: string; userId: string })
    : Promise<{ id: string; shortCode: string }>;
}

export class LinkService {
  constructor(private readonly repo: LinkRepository) {}

  async createLink(targetUrl: string, userId: string) {
    if (!isValidHttpUrl(targetUrl)) {
      throw new Error('INVALID_URL');
    }
    // 生成短码，极小概率撞码就重试
    let shortCode = nanoid(6);
    if (await this.repo.findByCode(shortCode)) {
      shortCode = nanoid(6);
    }
    return this.repo.create({ shortCode, targetUrl, userId });
  }
}

src/links/link.service.test.ts（注入 mock repo）

import { describe, it, expect, vi, beforeEach } from 'vitest';
import { LinkService, type LinkRepository } from './link.service.js';

// 用 vi.fn() 造一个全是 mock 方法的假 repo
function makeRepo(): LinkRepository {
  return {
    findByCode: vi.fn().mockResolvedValue(null), // stub：默认没撞码
    create: vi.fn().mockImplementation(async (d) => ({
      id: 'fake-id', shortCode: d.shortCode,
    })),
  };
}

describe('LinkService.createLink', () => {
  let repo: LinkRepository;
  let service: LinkService;

  beforeEach(() => {
    repo = makeRepo();
    service = new LinkService(repo);
  });

  it('非法 URL 抛 INVALID_URL，且不写库', async () => {
    await expect(service.createLink('xxx', 'u1'))
      .rejects.toThrow('INVALID_URL');
    expect(repo.create).not.toHaveBeenCalled(); // mock：断言没被调用
  });

  it('合法 URL 生成 6 位短码并落库', async () => {
    const link = await service.createLink('https://a.com', 'u1');
    expect(link.shortCode).toHaveLength(6);
    expect(repo.create).toHaveBeenCalledTimes(1); // mock：断言调用次数
    expect(repo.create).toHaveBeenCalledWith(
      expect.objectContaining({ targetUrl: 'https://a.com', userId: 'u1' }),
    );
  });
});

注意这套测试零数据库、零网络、毫秒级完成。findByCode 是 stub（给个固定返回让逻辑往下走），create 既是 stub（返回假对象）又当 mock 用（toHaveBeenCalledWith 断言它被怎么调用）。

记得清理 mock 状态：官方文档强调「务必在每次测试前后清理或还原 mock，否则上一条用例的调用记录会污染下一条」4。我们用 beforeEach 每次都重建 repo，自然就隔离了；也可在 config 里设 clearMocks: true。

🌍 实战：DI 让单测不碰真数据库。如果 LinkService 内部直接 import prisma 并调用，你单测时就被迫连库（慢、脏数据、CI 难配）或被迫用 vi.mock('../prisma') 去 hack 模块。0019 把 repo 从构造函数注入，使得测试只要传个普通对象即可——可测性是好架构的副产品，不是负担。这也是为什么「测试驱动」常常反过来改善设计。

5集成测试：用 supertest 打真实 HTTP 端点

单元测试保证「逻辑对」，但接不接得起来要看集成。supertest 专门测 HTTP 服务：你把 Express app 传给它，它会在临时端口起一个测试服务器并发真实请求，你无需手动管理端口5。它覆盖的正是单元测不到的部分：路由是否挂对、zod 校验中间件、错误处理中间件、状态码与响应体格式。

关键前提：导出 app 但不要在同一文件里 app.listen()，否则测试一 import 就会真的开始监听端口。把「建 app」和「listen」拆开：

src/app.ts（只建 app，不 listen）

import express from 'express';
import { linksRouter } from './links/link.router.js';
import { errorHandler } from './middleware/error.js';

export function createApp() {
  const app = express();
  app.use(express.json());
  app.use('/api/links', linksRouter);
  app.use(errorHandler); // 4 参数错误处理中间件
  return app;
}

src/server.ts（生产入口才 listen）

import { createApp } from './app.js';

const app = createApp();
app.listen(3000, () => {
  console.log('LinkPulse on :3000');
});

现在给 POST /api/links 写集成测试，覆盖校验失败 400 与成功 201两条关键路径：

// src/links/link.router.test.ts
import { describe, it, expect } from 'vitest';
import request from 'supertest';
import { createApp } from '../app.js';

const app = createApp();

describe('POST /api/links', () => {
  it('缺少 targetUrl → 400，且带错误信息', async () => {
    const res = await request(app)
      .post('/api/links')
      .send({})                       // 故意不传 body
      .set('Content-Type', 'application/json');

    expect(res.status).toBe(400);     // zod 校验失败应返回 400
    expect(res.body).toHaveProperty('error');
  });

  it('合法请求 → 201，返回短码', async () => {
    const res = await request(app)
      .post('/api/links')
      .send({ targetUrl: 'https://example.com' });

    expect(res.status).toBe(201);
    expect(res.body.shortCode).toHaveLength(6);
    // 验「行为」：拿到的短码确实能再查到
    const got = await request(app).get(`/api/links/${res.body.shortCode}`);
    expect(got.status).toBe(200);
  });
});

request(app) 链式发请求，.send() 带 body，res.status / res.body 拿响应做断言。它会真正经过你的 JSON 解析、zod 校验、controller、service、错误处理——这正是单元测试故意跳过的「胶水层」。

6隔离测试数据：别让测试互相污染

上面集成测试有个问题：service 真去写库了吗？集成测试通常需要真实数据库（这正是它价值所在），但绝不能用开发库或生产库——会留下脏数据、用例之间互相影响。三种主流隔离方案5：

方案怎么做权衡
独立测试库单独一个 linkpulse_test 库，每次跑前 prisma migrate reset简单；但要手动准备环境
事务回滚每条用例包在一个事务里，结束 ROLLBACK极快、天然隔离；但跨连接/嵌套事务较难配
Testcontainers用代码在测试启动时拉一个临时 Postgres 容器，结束销毁环境最干净、最接近生产；但需要 Docker（0024 后顺理成章）

对当前阶段，最朴素可行的是独立测试库 + 每个测试文件前清表。用一个单独的 .env.test 指向测试库，通过 DATABASE_URL 切换：

// vitest.config.ts 里指定测试环境变量文件
import { defineConfig } from 'vitest/config';
export default defineConfig({
  test: {
    environment: 'node',
    include: ['src/**/*.test.ts'],
    setupFiles: ['./test/setup.ts'], // 每个测试文件运行前先执行
  },
});
// test/setup.ts —— 测试专用钩子
import { beforeEach } from 'vitest';
import { prisma } from '../src/prisma.js';

beforeEach(async () => {
  // 关键：清空相关表，保证用例之间互不影响（注意外键顺序）
  await prisma.click.deleteMany();
  await prisma.link.deleteMany();
});
跑测试用独立 DB："test": "dotenv -e .env.test -- vitest run"，或用 Node 20+ 的 node --env-file=.env.test。务必确认 .env.test 的 DATABASE_URL 指向 测试库，否则你会把开发数据 deleteMany 掉。
🌍 实战：集成测试覆盖关键路径，别贪多。对 LinkPulse，「创建短链」和「短码跳转 302」是命脉——这两条挂了产品就废了，必须有集成测试守住。而「列出用户所有链接」这类查询逻辑简单，单元测就够。把有限的集成测试预算花在钱和用户最敏感的路径上。

7覆盖率与「测行为不测实现」

覆盖率（coverage）是「测试运行时执行了多少比例的代码」。npm run test:cov 会生成报告，告诉你哪些行没被任何测试碰过。它有用，但有两个陷阱：

  • 高覆盖率 ≠ 测得好。你可以「执行」一行代码却不断言它的结果——覆盖率 100% 也抓不到 bug。覆盖率只说「跑到了」，不说「验对了」。
  • 盲目追 100% 不划算。从 80% 提到 100% 的成本常常远高于收益，还会逼你给一堆 getter、配置代码写无意义的测试。Fowler 等人长期主张：覆盖率是发现「没测到的关键路径」的工具，不是 KPI。

测行为，不测实现（test behavior, not implementation）是最重要的心法。看这两种写法的区别：

❌ 测实现（脆）

// 断言「内部调用了 nanoid」
it('调用了 nanoid', () => {
  const spy = vi.spyOn(idMod, 'nanoid');
  service.createLink('https://a.com', 'u1');
  expect(spy).toHaveBeenCalled();
});
// 一旦你换掉短码生成库，
// 行为没变，测试却红了 —— 假警报

✅ 测行为（稳）

// 只断言「对外可观察的结果」
it('返回 6 位、URL 安全的短码', async () => {
  const link =
    await service.createLink('https://a.com', 'u1');
  expect(link.shortCode).toMatch(/^[\w-]{6}$/);
});
// 换库、改实现，只要结果对，
// 测试照样绿 —— 这才是安全网

判断标准：问自己「如果重构实现但功能不变，这个测试会不会无辜变红？」会的话，就是测得太细、绑死了实现。好测试盯着「输入 → 可观察的输出/副作用」，对内部怎么实现保持沉默。

🌍 实战：CI 里自动跑测试。测试的真正威力是自动化：0025 会配 GitHub Actions，让每次 push / 开 PR 都自动 npm test，红了就拦住合并。本章只要保证 npm test 在干净环境（fresh clone + npm ci）下能一条命令跑通——这是上 CI 的前提。所以别在测试里依赖「你本机才有的东西」（比如本地某个手动建的数据、写死的绝对路径）。
