Lesson 0017 · Node.js 后端之路

认证基础：注册、登录、密码哈希与 JWT

为什么学这个？ 到目前为止，LinkPulse 的任何短链都是「公共」的——谁都能创建、谁都能调你的接口。可一旦它要收费上线，第一个绕不开的问题就是：「这个请求背后到底是谁？」 本章给 LinkPulse 装上「门」——实现 POST /api/auth/register 与 POST /api/auth/login：用户用 email + 密码注册，密码经过不可逆哈希后落到我们已有的 users.password_hash 字段；登录成功后签发一张 JWT 令牌，让后续请求能证明「我就是刚才那个人」。这是后面「我的短链列表」「按 plan 计费」「权限控制」的地基。

1认证 vs 授权：先分清「你是谁」和「你能干啥」

这两个词在中文里都带「权」，新手极易混淆，但它们是两件先后发生的事：

  • 认证（Authentication，简称 authn）——「你是谁？」。系统确认请求背后的身份。登录就是认证：你出示 email + 密码，系统说「OK，你确实是 alice@x.com」。
  • 授权（Authorization，简称 authz）——「你能干啥？」。在已知你是谁的前提下，判断你有没有权限做这件事。比如「免费用户每月最多建 50 条短链」「你只能删自己的短链」就是授权。

类比： 进公司大楼。门口刷工卡确认「你是张三」= 认证；张三的工卡只能开 3 楼的门、开不了机房 = 授权。先认证、后授权，顺序不能反——不知道你是谁，就无从谈你能干啥。

本章只做认证（注册 + 登录 + 签发令牌）。下一课 0018 才做授权（用中间件校验令牌、按 plan 限流、资源归属判断）。把它们拆成两章，正是因为它们是两个独立的关注点。

作为前端，你早就天天和这两者打交道：调登录接口拿 token（认证的结果）、根据后端返回的角色字段决定按钮显隐（授权的体现）。区别在于——在后端，这两件事都必须由服务端权威地做，前端的任何判断都只是 UI 体验，不能当安全边界。

2密码为什么绝不能明文存——哈希 + 加盐

设想 LinkPulse 数据库被拖库（database breach，被攻击者导出整张表）。如果 password_hash 里存的是明文 "hunter2"，攻击者立刻拿到所有人的密码——而且很多人会在别的网站复用同一个密码，灾难会蔓延到用户的邮箱、银行。所以行业铁律：服务端永远不应该「知道」用户的明文密码，连数据库管理员也不该。

什么是哈希（hash）？

哈希函数是一种单向运算：输入任意数据，输出一段固定长度的「指纹」。它的关键性质是不可逆——从指纹算不回原文。

类比： 把一块牛排煎熟。煎熟容易（哈希），但你没法把熟牛排还原成生牛排（不可逆）。登录时我们不需要「还原」，只要把用户这次输入的密码再煎一次，比较两块「熟牛排」是否一样即可。

为什么还要「加盐」（salt）？

光哈希不够。两个用户若用了相同密码，纯哈希会得到相同的指纹；攻击者还能预先算好海量「常见密码 → 哈希」对照表（彩虹表 / rainbow table），拖库后一查就破。

盐是为每个用户随机生成的一段数据，哈希前混进密码里。这样：① 相同密码的两个用户，指纹也不同；② 彩虹表失效，因为攻击者得为每个盐单独重算，成本与用户数成正比1。OWASP 明确：必须为每个密码加唯一的盐。

为什么要用「慢」的算法？

MD5、SHA-256 这类是通用快速哈希，为大文件校验设计——一秒能算几十亿次，对密码反而是灾难：攻击者拖库后可以暴力穷举。密码必须用专门的、故意很慢的算法，并带可调的「工作因子」（work factor），让单次哈希耗时几十到几百毫秒。对正常登录无感，对暴力破解则是天文级成本1。

🌍 实战：别自己造密码哈希。 OWASP《Password Storage Cheat Sheet》给出的标准选择是：首选 Argon2id（内存硬，抗 GPU/ASIC，2015 密码哈希竞赛冠军），参数基线 m=19456 (19 MiB), t=2, p=1；次选 bcrypt（成熟、库多、够用），工作因子（cost / saltRounds）最少 10，生产常用 121。绝对不要用 MD5/SHA1/SHA256「裸哈希」存密码，更不要自己拼盐写算法——密码学自研几乎必翻车。本课用 bcrypt（npm 上最常见、装起来最省心），理解了原理后切 argon2 只是换个库。

❌ 错误：明文 / 裸 SHA256

// 拖库即裸奔
user.password = "hunter2";
// 或者：快速哈希、无盐、可被彩虹表秒破
user.password_hash =
  sha256("hunter2");

✅ 正确：bcrypt（盐已内嵌进结果）

// $2b$12$N9qo...salt...hash
// 段含：算法 / cost / 盐 / 哈希
user.password_hash =
  await bcrypt.hash("hunter2", 12);

注意上面右边那串：bcrypt 把算法版本、cost、盐、最终哈希全打包进一个字符串，所以你不用在数据库单独存盐——登录比对时 bcrypt 会从这串里把盐读出来重算4。这也是 users 表只有一个 password_hash 字段就够用的原因。

bcrypt 的 72 字节坑： bcrypt 只取输入的前 72 字节，超出部分被静默忽略。注意是字节不是字符——一串含 emoji 的 UTF-8 密码可能不到 72 字符却超 72 字节4。所以 OWASP 建议把密码长度上限设到 72 字节（或对超长密码先做一次 HMAC 预哈希）1。我们在 zod 校验里直接限制长度即可。

3注册流程：zod 校验 → 哈希 → 落库

注册做三件事，每一步都对应一个你已经掌握的工具：

1
校验输入 zod—— email 格式合法、密码长度达标（≥8、≤72 字节）。脏数据挡在门外。
↓
2
查重 Prisma—— 该 email 是否已注册。注意防枚举，见下
↓
3
哈希密码 bcrypt—— await bcrypt.hash(password, 12)，得到 password_hash。
↓
4
写 users 表 Prisma—— 存 email + password_hash + plan（默认 free）。返回时绝不带 password_hash。

先定义 zod schema 和密码工具模块。沿用我们已有的分层（route → controller → service）。

// src/modules/auth/auth.schema.ts
import { z } from "zod";

export const registerSchema = z.object({
  email: z.string().email("邮箱格式不正确"),
  // 下限 8 防弱密码；上限 72 对齐 bcrypt 字节上限
  password: z.string().min(8, "密码至少 8 位").max(72, "密码过长"),
});
export const loginSchema = registerSchema; // 登录字段相同

export type RegisterInput = z.infer<typeof registerSchema>;
export type LoginInput = z.infer<typeof loginSchema>;
// src/lib/password.ts —— 把 bcrypt 封一层，全项目统一用
import bcrypt from "bcrypt";

const SALT_ROUNDS = 12; // OWASP 最少 10，生产常用 12

export function hashPassword(plain: string): Promise<string> {
  return bcrypt.hash(plain, SALT_ROUNDS);
}

export function verifyPassword(
  plain: string,
  hash: string,
): Promise<boolean> {
  return bcrypt.compare(plain, hash); // 内部读出盐与 cost 重算后比对
}
// src/modules/auth/auth.service.ts
import { prisma } from "../../lib/prisma.js";
import { hashPassword } from "../../lib/password.js";
import type { RegisterInput } from "./auth.schema.js";

// 自定义错误，让 controller 能区分「邮箱已占用」
export class EmailTakenError extends Error {}

export async function registerUser(input: RegisterInput) {
  const existing = await prisma.user.findUnique({
    where: { email: input.email },
  });
  if (existing) throw new EmailTakenError();

  const passwordHash = await hashPassword(input.password);

  const user = await prisma.user.create({
    data: {
      email: input.email,
      password_hash: passwordHash, // 存哈希，永不存明文
      plan: "free",
    },
    // 只选安全字段，从源头杜绝 hash 泄露
    select: { id: true, email: true, plan: true, created_at: true },
  });
  return user;
}
为什么用 select 显式挑字段？ Prisma 默认返回整行，包括 password_hash。哪怕你只是疏忽地 res.json(user)，哈希就泄露了。在最底层（service/查询）就把敏感字段排除，比在每个出口处「记得删掉」更可靠——这叫「安全默认值（secure by default）」。

4登录流程：查用户 → 比对哈希（并防用户枚举）

登录是注册的镜像：用同样的算法把这次输入的密码再哈希一遍，和库里的比——但我们从不解密，而是用 bcrypt.compare 让它内部完成「取盐重算再比」。

🌍 实战：登录失败别告诉用户「是邮箱错还是密码错」。 如果邮箱不存在返回「用户不存在」、密码错返回「密码错误」，攻击者就能拿一堆邮箱来「探」哪些已注册——这叫用户枚举（user enumeration），是注册/登录/找回密码接口的经典漏洞。正确做法：两种情况都返回同一句模糊提示，如「邮箱或密码错误」，HTTP 状态统一 401。
进阶：时序攻击（timing attack）。 若邮箱不存在就直接返回，响应会比「邮箱存在但密码错」（要跑一次 bcrypt）快，攻击者能借响应时间差推断邮箱是否注册。严谨做法是：即使没查到用户，也对一个「假哈希」跑一次 bcrypt.compare，让两条路径耗时相近。本课先理解概念，下面代码给出注释位置。
// src/modules/auth/auth.service.ts （续）
import { verifyPassword } from "../../lib/password.js";
import type { LoginInput } from "./auth.schema.js";

export class InvalidCredentialsError extends Error {}

export async function loginUser(input: LoginInput) {
  const user = await prisma.user.findUnique({
    where: { email: input.email },
  });

  // 防枚举：邮箱不存在 与 密码错误，对外是同一种结果
  if (!user) {
    // 进阶（抗时序）：此处也跑一次假比对，凑齐耗时
    // await verifyPassword(input.password, DUMMY_HASH);
    throw new InvalidCredentialsError();
  }

  const ok = await verifyPassword(input.password, user.password_hash);
  if (!ok) throw new InvalidCredentialsError();

  // 校验通过：返回安全字段（绝不含 password_hash）
  return { id: user.id, email: user.email, plan: user.plan };
}

5登录之后怎么「记住」你：Session vs JWT

HTTP 是无状态的——每个请求都是「初次见面」，服务器默认不记得你上一个请求是谁。那登录成功后，怎么让接下来的请求（建短链、看列表）都知道「这是 alice」？两条主流路线：

Session（服务端有状态）

登录后服务端生成一个 session id，把「这个 id 对应 alice」存在服务端（内存/Redis/库）。浏览器拿着 id（通常放 cookie），每次请求带上，服务端拿 id 去查是谁。

✅ 能随时吊销（删服务端记录即登出）
❌ 服务端要存状态，多实例需共享存储（如 Redis）

JWT（无状态令牌）

登录后服务端签发一张 JWT，把「我是 alice」直接写进令牌并用密钥签名。之后服务端只验签名就知道是谁，无需查库。

✅ 无状态、易水平扩展、跨服务通用
❌ 签出去就难主动吊销，只能等它过期

JWT 到底长啥样？

JWT（JSON Web Token，RFC 7519）是三段 Base64URL 用 . 连接的字符串：header.payload.signature2。

  • Header：声明算法，如 {"alg":"HS256","typ":"JWT"}。
  • Payload（载荷）：放声明（claims），如 sub（subject，主体，常放 userId）、exp（过期时间）、自定义字段如 plan。注意：payload 只是 Base64 编码，不是加密，任何人都能解开看！绝不能放密码、敏感信息。2
  • Signature（签名）：用服务端密钥对前两段签名。改一个字符签名就对不上——这保证令牌没被篡改，且确实是我们签的。

类比： JWT 像演唱会的防伪门票。票面写着座位号（claims，公开可见），但带防伪水印（签名）。检票员不用打电话回票务中心查（不查库），看一眼水印真假就放行。缺点也在此：票一旦印出去，散场前你很难「作废某一张」。

🌍 实战：access token + refresh token。 因为 JWT 难吊销，业界通常签两张：① access token，寿命很短（15 分钟），带在每个请求里，被偷了损失也有限；② refresh token，寿命长（数天/周），存在服务端可记录、可吊销，只用来「换发」新的 access token。本课先实现单张 access token 跑通登录，refresh token 留到讲会话续期时再加——先建立心智模型。

6给 LinkPulse 签发 JWT

用 jsonwebtoken 签发。密钥必须放 .env，绝不能硬编码进代码——密钥一旦泄露，攻击者就能伪造任意人的令牌。

🌍 实战：JWT 密钥进 .env，且要足够强。 HS256 用的是对称密钥，太短易被暴力破解。生成一串高熵随机串：openssl rand -base64 48，写进 .env 的 JWT_SECRET，并确保 .env 在 .gitignore 里（你前面已配好 dotenv / --env-file）。不同环境用不同密钥。
// src/lib/jwt.ts
import jwt from "jsonwebtoken";

const SECRET = process.env.JWT_SECRET;
if (!SECRET) throw new Error("缺少环境变量 JWT_SECRET"); // 启动即失败，胜过运行时才炸

// 我们要塞进令牌的业务字段
export interface JwtPayload {
  sub: string;   // 用户 id（RFC 7519 标准声明 subject）
  plan: string;  // 计费档位，授权时会用到
}

export function signAccessToken(payload: JwtPayload): string {
  return jwt.sign(payload, SECRET, {
    expiresIn: "15m",        // 短寿命，降低泄露风险
    issuer: "linkpulse",     // iss：签发方，便于多服务校验
  });
}

export function verifyAccessToken(token: string): JwtPayload {
  // verify 验签 + 验过期；失败抛错，由调用方处理
  return jwt.verify(token, SECRET) as JwtPayload;
}
expiresIn 接受秒数或时间串（"15m"、"1h"、"7d"）；它会自动写入标准的 exp 声明，jwt.verify 校验时若已过期会抛 TokenExpiredError3。verify（验签）这一步本章只在「签发后立即验证跑通」时用；真正用它做请求拦截是下一课 0018 的认证中间件。

串起 controller（衔接你已有的 zod 校验中间件与 4 参数错误处理风格）：

// src/modules/auth/auth.controller.ts
import type { Request, Response, NextFunction } from "express";
import { registerSchema, loginSchema } from "./auth.schema.js";
import {
  registerUser, loginUser,
  EmailTakenError, InvalidCredentialsError,
} from "./auth.service.js";
import { signAccessToken } from "../../lib/jwt.js";

export async function register(req: Request, res: Response, next: NextFunction) {
  try {
    const input = registerSchema.parse(req.body); // 校验 + 类型收窄
    const user = await registerUser(input);
    res.status(201).json({ user }); // 201 Created；不下发 token，让其显式登录
  } catch (err) {
    if (err instanceof EmailTakenError) {
      // 注册接口可以明确提示邮箱占用（这是注册必要的反馈）
      return res.status(409).json({ error: "该邮箱已被注册" });
    }
    next(err); // 其余交给全局错误中间件（含 zod 校验错误）
  }
}

export async function login(req: Request, res: Response, next: NextFunction) {
  try {
    const input = loginSchema.parse(req.body);
    const user = await loginUser(input);
    const token = signAccessToken({ sub: user.id, plan: user.plan });
    res.json({ token, user });
  } catch (err) {
    if (err instanceof InvalidCredentialsError) {
      // 防枚举：邮箱不存在 / 密码错，统一这一句、统一 401
      return res.status(401).json({ error: "邮箱或密码错误" });
    }
    next(err);
  }
}
// src/modules/auth/auth.routes.ts
import { Router } from "express";
import { register, login } from "./auth.controller.js";

export const authRouter = Router();
authRouter.post("/register", register);
authRouter.post("/login", login);

// src/app.ts 里挂载：
// app.use("/api/auth", authRouter);
🌍 实战：token 该放客户端哪里？ 登录返回 token 后，前端有两种存法，各有安全权衡：
① localStorage——读写方便，但任何 XSS（注入的 JS）都能读走它，风险高；
② httpOnly cookie——JS 读不到（防 XSS 偷 token），但需防 CSRF（配 SameSite=Strict/Lax + Secure）。
对要上线收费的 SaaS，更稳的是 httpOnly + Secure + SameSite 的 cookie。本课接口先把 token 放响应体返回（便于你用 curl/Postman 调试理解），cookie 下发与 CSRF 防护放到 0018 一起讲。

登录接口里，当用户提交的邮箱在数据库中查不到时，下列哪种处理最稳妥？

⌨️ 动手：给 LinkPulse 装上注册与登录

  1. 安装依赖：npm i bcrypt jsonwebtoken 与 npm i -D @types/bcrypt @types/jsonwebtoken。
  2. 在 .env 里加 JWT_SECRET，值用 openssl rand -base64 48 生成；确认 .env 已被 .gitignore 忽略。
  3. 新建 src/lib/password.ts、src/lib/jwt.ts，以及 src/modules/auth/ 下的 schema / service / controller / routes，按本课代码实现。
  4. 在 app.ts 挂载 app.use("/api/auth", authRouter)。
  5. 用 curl 跑通：先 POST /api/auth/register 注册一个邮箱，去数据库确认 password_hash 是一串 $2b$12$... 而非明文；再 POST /api/auth/login，确认拿到 token。
  6. 把拿到的 token 粘到 jwt.io 解一下，验证 payload 里能看到你的 sub 和 plan——亲眼确认「payload 不是加密、人人可读」。
  7. （加分）故意用错误密码登录、用不存在的邮箱登录，确认两者返回完全相同的 401 提示。
