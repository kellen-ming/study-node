Lesson 0021 · Node.js 后端之路

实时点击看板：WebSocket 与 SSE

为什么学这个？ LinkPulse 现在能创建短链、跳转、缓存，上一课我们还把「点击落库 + 计数」搬进了后台队列异步处理。但用户花钱买的 SaaS，往往最爱盯着一块东西看——仪表盘上的实时点击数。问题是：HTTP 的本质是「客户端问、服务端答」，服务端没法主动把「刚刚又有人点了你的链接」推给浏览器。本章我们补上后端工程师必备的一块拼图：服务端主动推送。你将搞懂轮询 / SSE / WebSocket 三种方案的取舍，用 ws 给 LinkPulse 建一条实时通道，让 worker 处理完一次点击就广播给所有看板页面，做出一个会跳动的实时点击看板。

1为什么 HTTP 推不动：先认清问题

你做前端时调接口，流程永远是：浏览器发 fetch() → 服务器返回 → 连接关闭。这就是 HTTP 的请求-响应模型（request-response）：必须由客户端先「问」，服务端才能「答」，答完连接就断了。服务端无法在没人问的时候主动找上门说「喂，数据变了」。

对大多数页面这没问题。但「实时点击看板」要的恰恰相反——数据在服务端先变（某人点了短链，worker 算出新计数），我们希望它立刻出现在所有正在看看板的浏览器上。HTTP 原生做不到这件事，于是工程上演化出三条路。

🌍 实战：不要一上来就「实时 = WebSocket」。先问自己两个问题：谁推给谁？（只有服务端→客户端单向，还是双向交互）多频繁？（每分钟一次 vs 每秒几十条）。答错选型，代价是把一个本可以三行 SSE 搞定的需求，做成需要心跳、重连、鉴权、集群广播的 WebSocket 工程。

2三种方案对比：轮询 vs SSE vs WebSocket

① 轮询 Polling：用 HTTP 硬凑实时

既然服务端不能主动推，那就让客户端反复问。每隔几秒发一次请求拉最新数据，这叫短轮询（short polling）。它什么新技术都不用，前端一个 setInterval 就行：

// 前端：每 3 秒问一次后端「现在点击数多少」
setInterval(async () => {
  const res = await fetch('/api/links/abc123/stats');
  const { clickCount } = await res.json();
  document.querySelector('#count').textContent = String(clickCount);
}, 3000);

缺点很明显：延迟（最坏要等满一个间隔才看到变化）和浪费（99% 的请求拉回来发现没变化，纯属空跑，还给数据库和服务器加压）。轮询是「能用但笨」的兜底方案。

② SSE：服务端单向推送的「细水长流」

Server-Sent Events（服务端发送事件，SSE）让浏览器开一条长连接，服务端可以在这条连接上持续单向地往客户端推消息，直到一方关闭3。它本质上还是一个普通的 HTTP 响应，只是这个响应「永远不结束」，服务端不停 write 新数据进去。浏览器端用内置的 EventSource 对象接收，不需要装任何库，而且断线会自动重连3。

③ WebSocket：一条全双工长连接

WebSocket 是浏览器和服务器之间一条双向（full-duplex，全双工）的持久连接：建立之后，两端都能随时主动发消息，无需再走「请求-响应」1。它先用一个普通 HTTP 请求带 Upgrade: websocket 头完成「协议升级握手」，之后这条 TCP 连接就脱离 HTTP，改说 WebSocket 协议（ws:// 或 wss://）。

维度轮询 PollingSSEWebSocket
方向客户端反复问服务端→客户端（单向）双向（全双工）
底层很多次独立 HTTP一条长 HTTP 响应升级后的独立 ws 协议
浏览器 APIfetch + setIntervalEventSource（内置）WebSocket（内置）
自动重连不涉及浏览器内置要自己写
能否客户端主动发本来就在问不能（只能另开 HTTP）能

适合 LinkPulse 看板能用但浪费✅ 单向推送够用✅ 但稍重
🌍 实战：选型口诀——只有服务端往客户端推（通知、日志流、实时计数、进度条）→ 用 SSE，它更简单、自带重连、走标准 HTTP（穿透代理/防火墙更省心）；需要双向高频交互（聊天、协同编辑、游戏、客户端频繁上行）→ 才上 WebSocket。LinkPulse 的点击看板是典型的「服务端单向推」，SSE 本就够用；但本章主线用 WebSocket 实现，因为它是后端必会的硬技能，同时给你 SSE 的等价写法做对比。

3用 ws 给 LinkPulse 建 WebSocket 服务端

ws 是 Node.js 上事实标准的 WebSocket 库，「简单、极快、测试充分」，既能当服务端也能当客户端2。注意：ws 只在 Node 端用；浏览器端直接用内置的 WebSocket 对象，不需要也不能装 ws2。

npm i ws
npm i -D @types/ws

LinkPulse 已经有一个 Express 的 http.Server 在跑。我们不另起一个端口，而是用 noServer: true 把 WebSocket 挂到同一个 HTTP 服务器上，靠 HTTP 的 upgrade 事件来接管握手2。这样一个端口同时服务普通 API 和 ws，部署最省事。

// src/realtime/ws-hub.ts
import { WebSocketServer, WebSocket } from 'ws';
import type { Server } from 'node:http';
import { logger } from '../logger.js';

// 看板要推的事件结构（与前端约定好）
export interface ClickEvent {
  type: 'click';
  shortCode: string;
  clickCount: number;   // worker 算出的最新计数
  at: string;           // ISO 时间戳
}

// noServer：不自己监听端口，复用 Express 的 http server
const wss = new WebSocketServer({ noServer: true });

wss.on('connection', (socket: WebSocket) => {
  logger.info({ clients: wss.clients.size }, 'dashboard client connected');

  // 客户端断开时 ws 会自动从 wss.clients 移除，无需手动清理
  socket.on('close', () => {
    logger.info({ clients: wss.clients.size }, 'dashboard client disconnected');
  });

  socket.on('error', (err) => logger.error({ err }, 'ws socket error'));
});

// 把 wss 挂到已有的 http server 上：监听 HTTP 的 upgrade 握手
export function attachWebSocket(server: Server): void {
  server.on('upgrade', (request, socket, head) => {
    // 只接管 /ws/dashboard 这个路径的升级请求，其余拒绝
    if (request.url !== '/ws/dashboard') {
      socket.destroy();
      return;
    }
    wss.handleUpgrade(request, socket, head, (ws) => {
      wss.emit('connection', ws, request);
    });
  });
}

// 广播：把一个事件发给所有还连着的看板客户端
export function broadcastClick(event: ClickEvent): void {
  const payload = JSON.stringify(event);
  for (const client of wss.clients) {
    // 只发给状态为 OPEN 的连接，避免给正在关闭的 socket 写数据报错
    if (client.readyState === WebSocket.OPEN) {
      client.send(payload);
    }
  }
}
为什么用 noServer 而不是 { port: 8080 }？ 传 port 会让 ws 自己起一个独立 HTTP 服务器占一个新端口；noServer: true 则让你手动在已有服务器的 upgrade 事件里调 handleUpgrade2。生产环境通常一个进程一个对外端口（反向代理后面），复用端口最省心。

把它接进启动入口（你现有的 server.listen 那段）：

// src/server.ts
import { createServer } from 'node:http';
import { app } from './app.js';
import { attachWebSocket } from './realtime/ws-hub.js';

const server = createServer(app);   // 用 app 创建 http server
attachWebSocket(server);            // 把 WebSocket 挂上去
server.listen(3000, () => logger.info('LinkPulse up on :3000'));

wss.clients 是一个 Set，装着当前所有活跃连接；广播就是遍历它、对每个 readyState === WebSocket.OPEN 的连接调 send2。检查 OPEN 很重要：连接可能正处于「正在关闭」状态，往里写会抛错。

4让 worker 处理完点击就广播

上一课我们把「点击落库 + 计数」放进了后台队列的 worker。现在的目标：worker 算出某个短码的最新 clickCount 后，调一下 broadcastClick，把事件推给所有看板。

// src/jobs/click.worker.ts（在你已有的处理逻辑末尾追加）
import { prisma } from '../db.js';
import { broadcastClick } from '../realtime/ws-hub.js';

async function processClickJob(data: { shortCode: string; linkId: number; /* ... */ }) {
  // 1. 落库一条 click 明细（已学）+ 原子自增计数
  const link = await prisma.link.update({
    where: { id: data.linkId },
    data: { click_count: { increment: 1 } },
    select: { short_code: true, click_count: true },
  });

  // 2. 实时广播给所有看板客户端
  broadcastClick({
    type: 'click',
    shortCode: link.short_code,
    clickCount: link.click_count,
    at: new Date().toISOString(),
  });
}
关键前提（多实例时会咬人）：上面 broadcastClick 只能广播到当前进程里 wss.clients 持有的连接。如果 worker 和 Web 服务器是同一个进程，没问题；但生产上 worker 往往是独立进程，它根本没有那些浏览器连接。第 6 节用 Redis pub/sub 解决这个跨进程/跨实例的问题。

5前端看板页面：浏览器内置 WebSocket 接收

浏览器端用内置的 WebSocket 对象连上来。建实例就开始连，连上触发 open，之后能收 message、能 send1。下面是一个最简单、自包含的看板页（放在 public/dashboard.html 由 Express 静态托管即可）：

<!doctype html>
<meta charset="utf-8">
<h1>LinkPulse 实时点击看板</h1>
<ul id="feed"></ul>
<script>
  // wss:// 用于 https 站点；本地开发用 ws://
  const proto = location.protocol === 'https:' ? 'wss' : 'ws';
  const ws = new WebSocket(`${proto}://${location.host}/ws/dashboard`);

  ws.addEventListener('open', () => console.log('看板已连接'));

  ws.addEventListener('message', (e) => {
    const ev = JSON.parse(e.data);            // {type,shortCode,clickCount,at}
    const li = document.createElement('li');
    li.textContent = `${ev.shortCode} → ${ev.clickCount} 次（${ev.at}）`;
    document.getElementById('feed').prepend(li);
  });

  // WebSocket 不自带重连，断了要自己重连（这里简单延时重连）
  ws.addEventListener('close', () => {
    console.warn('连接断开，3 秒后重连');
    setTimeout(() => location.reload(), 3000);
  });
</script>

注意最后那段：WebSocket 协议本身不负责自动重连，网络抖动、服务重启都会让连接断掉，必须自己监听 close 重连（生产里要做指数退避，不是简单 reload）。这正是 SSE 相对省心的地方——它内置自动重连3。下面给出 SSE 的等价实现作对比。

SSE 等价版：更少代码搞定单向推送

SSE 服务端就是一个永不结束的 HTTP 响应：把 Content-Type 设成 text/event-stream，然后不停 res.write()3。报文格式是纯文本，每条消息用两个换行分隔；常见字段有 data:（负载）、event:（事件名，省略则触发 message）、id:、retry:（重连间隔毫秒），以 : 开头的行是注释（常用于心跳保活）3。

SSE 服务端（Express 路由）

// src/realtime/sse.ts
import type { Request, Response } from 'express';

const clients = new Set<Response>();

export function sseHandler(req: Request, res: Response) {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    Connection: 'keep-alive',
  });
  res.write('retry: 3000\n\n');   // 告诉浏览器断后 3s 重连
  clients.add(res);
  req.on('close', () => clients.delete(res));
}

export function ssePush(event: object) {
  const line = `event: click\ndata: ${JSON.stringify(event)}\n\n`;
  for (const res of clients) res.write(line);
}

SSE 客户端（浏览器内置）

// 不用装任何库，断线浏览器自动重连
const es = new EventSource('/sse/dashboard');

es.addEventListener('click', (e) => {
  const ev = JSON.parse(e.data);
  console.log(ev.shortCode, ev.clickCount);
});

es.onerror = (err) =>
  console.error('SSE 出错（会自动重连）', err);

// 需要时手动关闭，停止重连
// es.close();
🌍 实战：SSE 真实会踩的坑——① 必须设 Content-Type: text/event-stream，否则浏览器不当它是事件流；② 经过 Nginx 反向代理时要关掉响应缓冲（proxy_buffering off;），否则消息会被攒着不实时下发；③ 长连接要定期发心跳注释行（如每 15s 一行 : ping\n\n）防止中间代理把「看似空闲」的连接掐掉；④ HTTP/1.1 下每个域名同时只有 6 个连接的浏览器限制对 SSE 影响明显，HTTP/2 下基本无忧。

6两个绕不开的工程问题：鉴权 + 多实例

① WebSocket 也要鉴权

很多人以为「升级成 ws 就管不了权限了」。错。看板数据是用户私有的（谁的链接被点了），不能裸奔。握手阶段就是鉴权时机——握手是一个普通 HTTP 请求，能带 Cookie、Header、查询参数。在 upgrade 事件里校验，失败就 destroy() 拒绝，根本不让它升级成功：

// 在 attachWebSocket 里加鉴权（复用你已有的 JWT 校验，Lesson 0017 起学过）
import { verifyToken } from '../auth/jwt.js';

server.on('upgrade', (request, socket, head) => {
  if (request.url?.split('?')[0] !== '/ws/dashboard') { socket.destroy(); return; }

  try {
    // token 从查询参数取：浏览器原生 WebSocket 不能自定义请求头
    const url = new URL(request.url, `http://${request.headers.host}`);
    const token = url.searchParams.get('token');
    const user = verifyToken(token ?? '');   // 失败抛错
    wss.handleUpgrade(request, socket, head, (ws) => {
      (ws as any).userId = user.id;           // 把身份挂到连接上，后续按用户过滤
      wss.emit('connection', ws, request);
    });
  } catch {
    socket.write('HTTP/1.1 401 Unauthorized\r\n\r\n');
    socket.destroy();                          // 拒绝握手
  }
});
🌍 实战：浏览器原生 WebSocket 构造函数不支持自定义请求头，所以传 JWT 常见两种做法：放在 URL 查询参数（如上，简单但 token 可能进日志，建议用短时效 token），或先连上再发一条「认证消息」。SSE 因为是普通 HTTP，可直接带 Cookie，鉴权更顺。无论哪种，鉴权要在能收到任何数据之前完成。

② 多实例部署：用 Redis pub/sub 跨实例广播

问题来了：线上为了扛量，LinkPulse 会跑多个实例（多进程/多容器），用户 A 的看板连在实例 1，处理点击的 worker 却在实例 2。实例 2 调 broadcastClick 只能推到自己 wss.clients 里的连接——A 在实例 1，收不到。

解法是你早就学过的 Redis 的 pub/sub（发布订阅）：每个实例既向一个 Redis 频道发布点击事件，又订阅这个频道；任何实例收到消息后，都向自己本地的 WebSocket 连接广播。这样事件就能跨实例扩散到所有看板4。

worker（实例2）算出计数→ publish → Redis 频道 clicks→ 所有实例都 subscribe → 实例1 / 实例2 各自广播给本地 ws 客户端 
// src/realtime/redis-fanout.ts
import Redis from 'ioredis';
import { broadcastClick, type ClickEvent } from './ws-hub.js';

const CHANNEL = 'linkpulse:clicks';
// ioredis 要求：订阅连接和发布连接必须分开（订阅模式下不能再发普通命令）
const sub = new Redis(process.env.REDIS_URL!);
const pub = new Redis(process.env.REDIS_URL!);

// worker 那边：不再直接调 broadcastClick，而是 publish 到 Redis
export function publishClick(event: ClickEvent): void {
  pub.publish(CHANNEL, JSON.stringify(event));
}

// 每个 Web 实例启动时调一次：订阅频道，收到就广播给本地连接
export function startRedisFanout(): void {
  sub.subscribe(CHANNEL);
  sub.on('message', (_channel, message) => {
    broadcastClick(JSON.parse(message) as ClickEvent);
  });
}
🌍 实战：如果你选 socket.io 而不是裸 ws，这套跨实例广播官方已经帮你封装好了——装 @socket.io/redis-adapter，io.to('room').emit() 会自动经由 Redis pub/sub 扩散到集群里所有节点4。socket.io 还自带断线自动重连、心跳、对老浏览器的传输降级（fallback 到 HTTP 长轮询）、以及「房间（rooms）」——比如把每个用户的所有看板连接放进 room: user:42，推送时只发给这个房间，天然实现「按用户过滤」。代价是它有自己的协议、包更大，且非标准 WebSocket。权衡：要房间/降级/集群开箱即用 → socket.io；要轻量贴近标准 → ws 自己补这几块。

LinkPulse 的实时点击看板只需要「服务端把最新计数单向推给浏览器」，浏览器不需要主动上行高频消息。从「实现最省心 + 自带断线重连」角度，哪个方案最契合？

⌨️ 动手：给 LinkPulse 接一条实时通道

  1. npm i ws && npm i -D @types/ws，新建 src/realtime/ws-hub.ts，实现 attachWebSocket(server) 与 broadcastClick(event)（用 noServer: true + upgrade 事件，只接管 /ws/dashboard）。
