Lesson 0018 · Node.js 后端之路

授权与 Web 安全加固

为什么学这个？ 上一课你给 LinkPulse 加了认证（Authentication）——登录后签发 JWT，证明「你是谁」。但光知道你是谁还不够：现在 POST /api/links 任何人都能调，DELETE /api/links/:id 也没人拦——用户 A 能删用户 B 的短链。这一课补上授权（Authorization）：「你能不能做这件事」。我们会写 requireAuth 中间件保护接口、做越权防护（IDOR）让用户只能碰自己的短链、用 helmet 一行加固响应头、用 express-rate-limit 给登录接口限流防撞库，最后用 OWASP Top 10 给整个 LinkPulse 体检。做收费 SaaS，安全不是加分项，是底线。

1先分清：认证 vs 授权

这两个词中文都带「权」，但英文很清楚，务必分开理解：

  • 认证 Authentication（AuthN）：确认「你是谁」。上一课的登录 + 验证 JWT 就是这一步。结果是一个「身份」——比如 userId = 42。
  • 授权 Authorization（AuthZ）：确认「你能不能做这件事」。已知你是 42 号用户，那你能删 99 号短链吗？你的 plan 是 free，能用批量导出功能吗？这些都是授权问题。
类比：进公司大楼。认证是刷工牌进门（证明你是员工）；授权是你的工牌能不能开机房门（证明你有权限）。两者独立——能进门不代表能进机房。

OWASP Top 10 2021 把「Broken Access Control（失效的访问控制）」列为第一名风险，官方数据是被测应用里 94% 存在某种访问控制缺陷1。授权做不好，是当今 Web 应用最高频的漏洞来源——所以这一课分量很重。

2鉴权中间件 requireAuth：从 Bearer 取 JWT 到挂上 req.user

授权的第一步，是在每个受保护请求上「认出用户」。前端调受保护接口时会带一个 HTTP 头：

Authorization: Bearer <你的 JWT>

Bearer（持有者）是 OAuth 2.0 定义的一种 token 类型，意思是「谁持有这个 token，谁就被当作合法用户」——所以 token 绝不能泄露。我们写一个中间件（middleware），它的职责很单一：

取 Authorization 头→ 切出 Bearer token→ jwt.verify 验签 + 验过期→ 挂 req.user，next() ／失败 →401 Unauthorized
快速复习（你前端见过中间件思想）： Express 中间件就是 (req, res, next) => {} 的函数，能读改 req/res，然后 next() 放行或直接 res 结束。在后端它是「请求流水线上的一道工序」——鉴权就是其中一道。

先扩展 Express 的类型，让 TS 知道 req.user 存在（在项目里建 src/types/express.d.ts）：

// src/types/express.d.ts —— 用声明合并给 Request 加上 user 字段
import 'express';

declare global {
  namespace Express {
    interface Request {
      user?: { id: number; plan: 'free' | 'pro'; role: 'user' | 'admin' };
    }
  }
}

然后是中间件本体。我们沿用上一课签发 JWT 时放进 payload 的字段（sub 放 userId，外加 plan、role）：

// src/middleware/requireAuth.ts
import type { Request, Response, NextFunction } from 'express';
import jwt from 'jsonwebtoken';
import { logger } from '../lib/logger.js'; // 第 16 课的 pino

interface JwtPayload {
  sub: string;                 // userId，JWT 标准字段 subject
  plan: 'free' | 'pro';
  role: 'user' | 'admin';
}

export function requireAuth(req: Request, res: Response, next: NextFunction) {
  const header = req.headers.authorization;

  // 1) 没头、或格式不是 "Bearer xxx" → 401
  if (!header?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  const token = header.slice('Bearer '.length).trim();

  try {
    // 2) verify 会同时校验「签名是否被篡改」和「是否过期」
    const payload = jwt.verify(token, process.env.JWT_SECRET!) as JwtPayload;

    // 3) 挂到 req.user，后续 controller / 越权校验都用它
    req.user = { id: Number(payload.sub), plan: payload.plan, role: payload.role };
    next();
  } catch (err) {
    // 4) 签名错、过期、格式坏都进这里 —— 注意：对外只说 401，别回显内部错误
    logger.warn({ err: (err as Error).name }, 'jwt verify failed');
    return res.status(401).json({ error: 'Unauthorized' });
  }
}
🌍 实战：错误信息别泄露内部细节。 上面 catch 里我们把真实原因（TokenExpiredError / JsonWebTokenError）只写进服务端日志，对外统一回 401 Unauthorized。如果你把 err.message 直接返回给前端，攻击者就能据此推断你的实现细节（用了什么库、token 哪里坏了），这正是 OWASP「Security Misconfiguration / 安全配置错误」的典型表现1。Express 官方安全最佳实践也强调：生产环境要关掉详细错误堆栈的外泄4。
🌍 实战：JWT 过期时间要短。 JWT 是无状态的——服务端不存它，签出去之后在过期前都有效，你没法「立即吊销」。所以一旦泄露，攻击者能用到过期为止。业界做法是 access token 设短（如 15m），配一个长效 refresh token 做续签。OWASP 在「失效的访问控制」防护建议里也明确写到要让 JWT 短命以缩小被利用的时间窗1。

3保护 LinkPulse 的受保护路由

有了中间件，挂上去就是「保护」。在 LinkPulse 里，创建 / 列表 / 改 / 删短链都必须登录，但跳转 GET /:code 是公开的（任何人点短链都要能跳），所以不能全局挂，要按路由分。

// src/routes/links.route.ts
import { Router } from 'express';
import { requireAuth } from '../middleware/requireAuth.js';
import * as ctrl from '../controllers/links.controller.js';

const router = Router();

// 整个 /api/links 子路由都要登录：挂在 router 级别一次搞定
router.use(requireAuth);

router.post('/', ctrl.createLink);
router.get('/', ctrl.listMyLinks);       // 只列「我的」短链
router.patch('/:id', ctrl.updateLink);
router.delete('/:id', ctrl.deleteLink);

export default router;
// src/app.ts （节选）—— 注意公开跳转不经过 requireAuth
app.use('/api/links', linksRouter);   // 受保护
app.get('/:code', redirectHandler);   // 公开：短链跳转，不需要登录
router.use(requireAuth) 把中间件挂在这个子路由上，下面所有路由都会先过它。比在每个 handler 前重复写一遍更不容易漏——漏挂一个就是一个洞。

4越权防护（IDOR）：用户只能碰自己的短链

这是本章的核心。requireAuth 只解决了「登录了没」，没解决「这资源是不是你的」。看一个真实会出的洞：

// ❌ 危险版：只要登录就能删 —— 但删的是谁的短链？没校验！
export async function deleteLink(req: Request, res: Response) {
  const id = Number(req.params.id);
  await prisma.link.delete({ where: { id } });  // 用户 42 能删掉 99 号短链
  res.status(204).end();
}

用户 A 登录后，把 URL 里的 :id 换成别人的短链 id，就能删/改/看到别人的数据。这就是 IDOR（Insecure Direct Object Reference，不安全的直接对象引用）——OWASP 给的经典例子就是把请求参数从 acct=自己 改成 acct=别人 来访问他人资源1。它属于「Broken Access Control」，OWASP Top 10 第 1 名。

🌍 实战：IDOR 是最高频漏洞之一，务必校验资源归属。 它隐蔽在于——功能「看起来正常工作」，自己测自己的数据永远发现不了，只有换个 id 才暴露。修复原则就一句话：每次操作资源，都要确认 resource.userId === req.user.id。OWASP 把「在应用层强制记录归属（enforce record ownership），而不是信任用户只访问自己的数据」列为头号防护建议1。

正确写法——把「取 + 校验归属 + 操作」放进 service 层，先查再判：

// src/services/links.service.ts
import { prisma } from '../lib/prisma.js';

// 自定义错误类型，让 controller 知道该回 404 还是 403
export class NotFoundError extends Error {}
export class ForbiddenError extends Error {}

export async function deleteOwnedLink(linkId: number, userId: number) {
  const link = await prisma.link.findUnique({ where: { id: linkId } });

  // 1) 资源不存在 → 404
  if (!link) throw new NotFoundError('link not found');

  // 2) 资源存在但不属于你 → 403（关键的越权校验）
  if (link.userId !== userId) throw new ForbiddenError('not your link');

  // 3) 通过校验，才真正删除
  await prisma.link.delete({ where: { id: linkId } });
}
// src/controllers/links.controller.ts
export async function deleteLink(req: Request, res: Response, next: NextFunction) {
  try {
    await deleteOwnedLink(Number(req.params.id), req.user!.id);
    res.status(204).end();
  } catch (err) {
    if (err instanceof NotFoundError)  return res.status(404).json({ error: 'Not Found' });
    if (err instanceof ForbiddenError) return res.status(403).json({ error: 'Forbidden' });
    next(err); // 其它丢给第 16 课的错误处理中间件
  }
}

列表查询同理——别 findMany() 把所有人的短链全捞出来，要带上 where: { userId }，从数据库层就只返回「我的」：

export async function listMyLinks(userId: number) {
  return prisma.link.findMany({
    where: { userId },                 // ← 越权防护：天然只查自己的
    orderBy: { createdAt: 'desc' },
  });
}
403 vs 404 的取舍： 资源存在但不是你的，标准答案是 403 Forbidden（如上）。但更保守的做法是统一返回 404——这样攻击者连「这个 id 到底存不存在」都探测不出来（避免信息泄露）。两种都对，按你的威胁模型选；金融/隐私敏感场景倾向 404。

5RBAC 初步：用 role / plan 做权限分级

IDOR 防的是「同一种角色之间互相越权」。还有一类授权是「不同角色能做的事不同」——这叫 RBAC（Role-Based Access Control，基于角色的访问控制）：先把用户分成角色（如 user / admin），再按角色发权限。LinkPulse 的 users 表已经有 plan 字段（free/pro），认证章节也约定了 role，正好用来做可组合的小中间件：

// src/middleware/authorize.ts
import type { Request, Response, NextFunction } from 'express';

// 工厂函数：返回一个「只放行指定 role」的中间件
export function requireRole(...roles: Array<'user' | 'admin'>) {
  return (req: Request, res: Response, next: NextFunction) => {
    // 走到这里前必须已过 requireAuth，所以 req.user 一定有
    if (!req.user || !roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

// 同理：限制只有付费 plan 才能用的功能
export function requirePlan(...plans: Array<'free' | 'pro'>) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.user || !plans.includes(req.user.plan)) {
      return res.status(403).json({ error: 'Upgrade required' });
    }
    next();
  };
}
// 用法：组合中间件，从左到右依次跑
import { requireAuth } from '../middleware/requireAuth.js';
import { requireRole, requirePlan } from '../middleware/authorize.js';

// 后台统计只给管理员
router.get('/admin/stats', requireAuth, requireRole('admin'), ctrl.adminStats);

// 自定义短码是 pro 专属功能
router.post('/links/custom', requireAuth, requirePlan('pro'), ctrl.createCustomLink);
🌍 实战：授权要「deny by default（默认拒绝）」。 OWASP 的头号防护建议就是——除了公开资源，默认一律拒绝，显式授权才放行1。落到代码上：受保护路由默认都挂 requireAuth，需要更高权限的再叠 requireRole/requirePlan；千万别反过来「默认开放，记得时再去加锁」——那样一定会漏。还要把访问控制集中（统一用这几个中间件），而不是每个 handler 各写各的 if，否则改一处漏十处。

6OWASP Top 10 概览：给 LinkPulse 体检

OWASP（Open Worldwide Application Security Project）是公认的 Web 安全权威组织，它每隔几年发布的 Top 10 是「最关键的十大 Web 应用安全风险」清单1。下面是 2021 版（当前最广泛引用的版本），并对照 LinkPulse 现状2：

#风险（中英）LinkPulse 里长什么样 / 我们怎么防
A01Broken Access Control 失效的访问控制本章重点：IDOR + RBAC + deny by default（第 4、5 节）
A02Cryptographic Failures 加密失误密码用 bcrypt/argon2 hash（上一课）；线上强制 HTTPS；JWT_SECRET 进 .env
A03Injection 注入（SQL/命令注入）Prisma 参数化查询天然防 SQL 注入；输入仍用 zod 校验
A04Insecure Design 不安全的设计限流、归属校验从设计阶段就考虑，而非事后补
A05Security Misconfiguration 安全配置错误helmet 加固响应头（第 7 节）；关掉详细错误外泄
A06Vulnerable & Outdated Components 组件过时npm audit 定期查；锁版本；及时升级依赖
A07Identification & Authentication Failures 认证失效登录限流防撞库（第 8 节）；JWT 短过期

关于 XSS： 跨站脚本（Cross-Site Scripting）在 2021 版被并入 A03 Injection。LinkPulse 是纯 JSON API，本身不直接渲染 HTML，XSS 风险主要在前端展示环节；但我们仍要靠 helmet 的 CSP 等头做纵深防御（下一节）。

7helmet：一行加固安全响应头

helmet 是 Express 生态最常用的安全中间件，它通过设置一组 HTTP 响应头来缓解常见攻击。helmet() 默认会设置约 15 个安全相关响应头，并移除 Express 默认暴露的 X-Powered-By（否则等于告诉别人「我用 Express」）3。

npm i helmet
// src/app.ts —— 越早挂越好，让所有响应都带上安全头
import express from 'express';
import helmet from 'helmet';

const app = express();

app.use(helmet());     // ← 一行，默认设置全套安全响应头

app.use(express.json());
// ... 路由

它设置的几个关键头（理解它们防什么，而不是只记住「加了就行」）：

  • Content-Security-Policy（CSP）：限制页面能加载哪些来源的脚本/样式，是防 XSS 的主力。
  • Strict-Transport-Security（HSTS）：告诉浏览器「以后只用 HTTPS 访问我」，防降级到明文 HTTP3。
  • X-Content-Type-Options: nosniff：禁止浏览器「猜」内容类型，防 MIME 混淆攻击。
  • X-Frame-Options：禁止你的页面被 iframe 嵌套，防点击劫持（clickjacking）。
🌍 实战：helmet 一行加固，但 CSP 要按需调。 app.use(helmet()) 几乎零成本，Express 官方安全最佳实践直接推荐使用它4。坑在于默认 CSP 可能挡掉你前端的内联脚本/外部 CDN。如果 LinkPulse 是纯后端 API（不返回 HTML），可以放心用默认；如果同域托管前端页面，就要为 CSP 显式配置允许的来源，例如 helmet({ contentSecurityPolicy: { directives: { ... } } })，否则页面会白屏。

8express-rate-limit：给登录接口限流防撞库

限流（rate limiting）就是限制「同一来源在一段时间内最多调多少次接口」。它能防撞库 / 暴力破解（brute-force）：攻击者拿着泄露的密码字典对 /api/auth/login 一秒打几百次，限流让他每分钟只能试几次，攻击成本飙升。

npm i express-rate-limit
// src/middleware/rateLimiters.ts
import { rateLimit } from 'express-rate-limit';

// 1) 全局兜底：每个 IP 15 分钟内最多 100 次普通请求
export const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,   // 时间窗：15 分钟（毫秒）
  limit: 100,                 // 窗口内允许的最大请求数（v7 用 limit，旧版叫 max）
  standardHeaders: 'draft-8', // 用标准的 RateLimit-* 响应头
  legacyHeaders: false,       // 关掉老的 X-RateLimit-* 头
  message: { error: 'Too many requests, please try again later.' },
});

// 2) 登录接口单独收紧：每 IP 15 分钟只许 5 次失败尝试
export const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  limit: 5,                       // 比全局严得多 —— 这是防撞库的关键
  standardHeaders: 'draft-8',
  legacyHeaders: false,
  skipSuccessfulRequests: true,   // 登录成功不计数，只数失败尝试
  message: { error: 'Too many login attempts, try again in 15 minutes.' },
});

选项名 windowMs（时间窗毫秒）、limit（窗口内上限）来自官方文档；超限时中间件会自动返回 429 Too Many Requests5。挂上去：

// src/app.ts
import { apiLimiter, loginLimiter } from './middleware/rateLimiters.js';

app.use(helmet());
app.use('/api', apiLimiter);                  // 全局兜底
app.use('/api/auth/login', loginLimiter);     // 登录单独严格限流
app.use(express.json());
// ... 路由
🌍 实战：对登录接口单独限流防撞库。 全站统一限流不够——登录是攻击重灾区，要给它单独、更严的策略（这里 5 次 vs 全局 100 次），并配 skipSuccessfulRequests 只惩罚失败尝试，避免误伤正常用户。注意默认按内存计数，多实例部署会各算各的（每台机器一份计数）——生产环境要换成 Redis store（你已经在用 ioredis），让限流跨实例共享，否则横向扩容后限流形同虚设5。

9CORS 简述：浏览器跨域怎么放行

CORS（Cross-Origin Resource Sharing，跨域资源共享）是浏览器的安全机制：默认情况下，https://app.linkpulse.io 的前端页面用 fetch 调 https://api.linkpulse.io 的后端，会被浏览器拦下（不同源）。后端需要显式返回 Access-Control-Allow-Origin 等响应头，浏览器才放行。

npm i cors
// src/app.ts
import cors from 'cors';

app.use(cors({
  origin: 'https://app.linkpulse.io',  // 只允许你自己的前端域名（别用 '*'）
  credentials: true,                   // 若要带 cookie
}));
常见误区： CORS 不是用来「保护后端」的——它是约束浏览器的。curl、Postman、其它服务端调用根本不受 CORS 限制。所以真正的访问控制还得靠 requireAuth + 归属校验；CORS 只是控制「哪些网页能在浏览器里调你的 API」。生产配置时白名单具体域名，别图省事写 origin: '*'。

  7. 验证越权已堵： 用两个账号各建一条短链，拿 A 的 token 去 DELETE B 的短链 id，确认返回 403 而不是 204。
  8. 验证限流生效： 对 /api/auth/login 连发 6 次错误密码，第 6 次应返回 429。
